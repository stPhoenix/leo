# F51 — MCP client, config & transports

## Purpose

Turn Leo into a Model Context Protocol **host** per [FR-MCP-01](../../context.md#fr-mcp-01): load enabled server entries from `.leo/config.json` `mcpServers` with standard host shape (`{command, args, env}` for stdio, `{url}` for HTTP+SSE) per [FR-MCP-03](../../context.md#fr-mcp-03), open both transports per [FR-MCP-02](../../context.md#fr-mcp-02), connect to every enabled server in parallel at plugin load without blocking `onload` per [FR-MCP-04](../../context.md#fr-mcp-04) and [NFR-PERF-06](../../context.md#nfr-perf-06), discover each server's `tools/list` / `resources/list` / `prompts/list` per [FR-MCP-05](../../context.md#fr-mcp-05), and register tools into [F16](../tool-registry-builtin-read/feature.md)'s `ToolRegistry` under the namespace `mcp.<serverId>.<toolName>` per [FR-MCP-06](../../context.md#fr-mcp-06). Secrets living inside `env` (stdio) or auth headers (SSE) are stored through [F38](../cloud-providers-safestorage/feature.md)'s `SafeStorage` so vault plaintext is never a secret store per [NFR-DATA-04](../../context.md#nfr-data-04). All lifecycle events flow through [F01](../plugin-bootstrap-logging/feature.md)'s `Logger`. This slice delivers the core host — confirmation default (F52), resource picker (F53), prompts-as-skills (F54), settings UI (F55), and reconnect / shutdown (F56) build on it.

## Scope

### In scope

- `MCPClient` module (under `src/mcp/` per [architecture §9](../../../../architecture/architecture.md#9-project-file-layout-proposed)) owning a `Map<serverId, ServerRuntime>` of connected servers where `ServerRuntime = {id, transport, client, tools, resources, prompts, status, disposer}`; exposes `connectAll()`, `disconnectAll()`, `listResources(serverId?)`, `listPrompts(serverId?)`, `callTool(serverId, toolName, args, ctx)` per [architecture §3.2 / §4 / §5.5](../../../../architecture/architecture.md#32-agent-layer).
- `.leo/config.json` loader with Zod schema `McpServerConfig = {id: string, enabled: boolean, transport: 'stdio'|'sse'} & ({command, args?, env?: Record<string,string>} | {url: string, headers?: Record<string,string>})` per [FR-MCP-03](../../context.md#fr-mcp-03); schema parse errors are logged and the offending entry skipped (never throws out of `onload`).
- Transports: `StdioClientTransport` (Node `child_process.spawn` via the Electron renderer per [constraints](../../context.md#constraints)) and `SSEClientTransport` (`fetch` + `EventSource`-compatible SSE) supplied by `@modelcontextprotocol/sdk` per [tech-stack — MCP client (phase 6)](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring); each transport wrapped in a thin adapter so the domain code never imports SDK types directly.
- Parallel, non-blocking startup at `Plugin.onload`: `MCPClient.connectAll()` kicks one `connect(serverId)` promise per enabled entry via `Promise.allSettled`; `onload` awaits the kick-off, not the results, honouring [NFR-PERF-06](../../context.md#nfr-perf-06) and the [architecture §5.1](../../../../architecture/architecture.md#51-plugin-startup) "async, non-blocking" edge.
- Per-server discovery pipeline on successful `initialize`: call `tools/list`, `resources/list`, `prompts/list` in parallel, capture JSON-Schema for each tool, store resources + prompts in the `ServerRuntime` for F53 / F54 consumers, surface tools via registration below per [FR-MCP-05](../../context.md#fr-mcp-05).
- Namespaced tool registration: for every discovered tool, build a `ToolSpec` with `id = "mcp." + serverId + "." + toolName`, `description` from the server, Zod schema compiled from the tool's JSON Schema, `source: "mcp"`, `mcpServerId: serverId`, and `invoke` delegating to `MCPClient.callTool(serverId, toolName, args, ctx)` per [architecture §4](../../../../architecture/architecture.md#4-key-contracts); registered via [F16](../tool-registry-builtin-read/feature.md) `ToolRegistry.register` per [FR-MCP-06](../../context.md#fr-mcp-06). `requiresConfirmation` defaulting is owned by F52 and is out of scope here.
- Secret resolution: before spawning / connecting, the loader substitutes any value shaped `"safestorage:<key>"` in `env` / `headers` with a plaintext read from [F38](../cloud-providers-safestorage/feature.md)'s `SafeStorage.get(key)`; plaintext never touches `data.json` nor `.leo/config.json` per [NFR-DATA-04](../../context.md#nfr-data-04) and [architecture §6](../../../../architecture/architecture.md#6-state-ownership).
- Failure isolation: a failed `initialize`, transport error, or discovery error on one server logs a `warn` via [F01](../plugin-bootstrap-logging/feature.md), flags that `ServerRuntime.status = "failed"`, and never blocks the rest or the plugin itself per [FR-MCP-04](../../context.md#fr-mcp-04).
- Structured log events (`mcp.connect.start`, `mcp.connect.ok`, `mcp.connect.fail`, `mcp.discovery.ok`, `mcp.tool.register`, `mcp.tool.invoke.start/ok/error`) through [F01](../plugin-bootstrap-logging/feature.md) with `{serverId, transport, toolCount, resourceCount, promptCount, durationMs}` fields; never logs `env`, `headers`, or tool argument payloads above `debug` per [code-style — Logging](../../../../standards/code-style.md#logging).
- Vitest coverage per [NFR-TEST-05](../../context.md#nfr-test-05): a bundled reference stdio MCP fixture server (minimal `tools/list` + echo tool) in `tests/integration/mcp.test.ts` covers connect → discover → register → invoke round-trip; SSE path fixture via `msw`; failure-isolation test with one bad + one good server; secret-substitution test with a mocked `SafeStorage`.

### Out of scope

- `requiresConfirmation: true` default on MCP tools and per-thread pre-approval semantics — ships with [F52 mcp-tool-confirmation](../../features-index.md) per [FR-MCP-07](../../context.md#fr-mcp-07).
- Resource picker UI that inserts MCP resource content into the next message — ships with [F53 mcp-resources-picker](../../features-index.md) per [FR-MCP-08](../../context.md#fr-mcp-08); this feature stores the discovered list but does not render it.
- Prompt surfacing in the skill picker as "From MCP" — ships with [F54 mcp-prompts-in-skills](../../features-index.md) per [FR-MCP-09](../../context.md#fr-mcp-09); discovery only here.
- Settings-tab UI for add / edit / remove / enable / connection-status — ships with [F55 mcp-settings-ui](../../features-index.md) per [FR-MCP-10](../../context.md#fr-mcp-10); this feature only reads `.leo/config.json`.
- Reconnect-with-exponential-backoff, tool-error-on-crash, and clean `SIGTERM` → `SIGKILL` shutdown — ship with [F56 mcp-reconnect-shutdown](../../features-index.md) per [FR-MCP-11](../../context.md#fr-mcp-11) / [FR-MCP-12](../../context.md#fr-mcp-12). This feature exposes a `ServerRuntime.disposer()` hook so F56 can plug in.
- `SafeStorage` adapter itself — already shipped by [F38 cloud-providers-safestorage](../cloud-providers-safestorage/feature.md); this feature only consumes `get(key)`.
- `ToolRegistry` module and serial-tool-loop plumbing — already shipped by [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md); this feature only calls `register`.

## Acceptance criteria

1. On `Plugin.onload` with at least one `enabled: true` entry in `.leo/config.json` `mcpServers`, `MCPClient.connectAll()` kicks one connect promise per entry in parallel via `Promise.allSettled`, and `onload` returns before any server's `initialize` resolves — asserted by a Vitest test that holds the fixture's `initialize` open and verifies `onload` has already resolved. ([FR-MCP-04](../../context.md#fr-mcp-04), [NFR-PERF-06](../../context.md#nfr-perf-06))
2. Stdio entries (`{transport: "stdio", command, args?, env?}`) spawn a child via the SDK's `StdioClientTransport`; SSE entries (`{transport: "sse", url, headers?}`) open an SSE connection via `SSEClientTransport`; the `.leo/config.json` shape matches the standard MCP host config and unknown fields are preserved for forward compatibility. ([FR-MCP-02](../../context.md#fr-mcp-02), [FR-MCP-03](../../context.md#fr-mcp-03))
3. After a server's `initialize` resolves, `MCPClient` calls `tools/list`, `resources/list`, `prompts/list` in parallel and stores the results on the `ServerRuntime`; each discovered tool is wrapped into a `ToolSpec` with `id = "mcp." + serverId + "." + toolName`, `source: "mcp"`, `mcpServerId: serverId`, and registered into [F16](../tool-registry-builtin-read/feature.md)'s `ToolRegistry.register`. ([FR-MCP-05](../../context.md#fr-mcp-05), [FR-MCP-06](../../context.md#fr-mcp-06))
4. When an agent turn invokes a registered `mcp.<serverId>.<toolName>` tool, `ToolSpec.invoke` delegates to `MCPClient.callTool`, which issues a `tools/call` JSON-RPC request to the owning transport and returns `{ok:true, data}` on success or `{ok:false, error}` on transport / server error, matching the per-[architecture §5.5](../../../../architecture/architecture.md#55-mcp-tool-call) flow. ([FR-MCP-01](../../context.md#fr-mcp-01), [FR-MCP-06](../../context.md#fr-mcp-06))
5. A failed `initialize`, transport error, or discovery error on one server logs `mcp.connect.fail` at `warn` via [F01](../plugin-bootstrap-logging/feature.md), sets that `ServerRuntime.status = "failed"`, and does not prevent other servers from registering tools nor prevent the plugin from finishing load — covered by a Vitest scenario with one bad + one good server. ([FR-MCP-04](../../context.md#fr-mcp-04))
6. Any value in `env` / `headers` matching the literal `"safestorage:<key>"` is replaced with `await SafeStorage.get(key)` before transport construction; a Vitest FS / `data.json` sniff over the write paths asserts that no plaintext secret is persisted into the vault or plugin data. ([NFR-DATA-04](../../context.md#nfr-data-04))
7. Structured log events (`mcp.connect.start`, `mcp.connect.ok`, `mcp.connect.fail`, `mcp.discovery.ok`, `mcp.tool.register`, `mcp.tool.invoke.start / ok / error`) emit via [F01](../plugin-bootstrap-logging/feature.md)'s `Logger` with fields `{serverId, transport, toolCount, resourceCount, promptCount, durationMs}`; tool arguments, `env`, and `headers` are never logged at `info` or above. ([FR-MCP-05](../../context.md#fr-mcp-05), [FR-MCP-06](../../context.md#fr-mcp-06))
8. Vitest integration suite drives a bundled reference stdio MCP fixture server through connect → discover → register → invoke and asserts the registered `ToolSpec.id` namespace and round-trip `ToolResult`; SSE path covered via `msw`; a failure-isolation test and a secret-substitution test round out coverage. ([NFR-TEST-05](../../context.md#nfr-test-05))

## Dependencies

- [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md) — provides the `Plugin.onload` / `onunload` lifecycle that `MCPClient.connectAll()` is kicked from, the `.leo/` directory bootstrap that hosts `.leo/config.json`, and the `Logger` consumed by every `mcp.*` structured event per [NFR-LOG-04](../../context.md#nfr-log-04).

- Drives requirements [FR-MCP-01](../../context.md#fr-mcp-01), [FR-MCP-02](../../context.md#fr-mcp-02), [FR-MCP-03](../../context.md#fr-mcp-03), [FR-MCP-04](../../context.md#fr-mcp-04), [FR-MCP-05](../../context.md#fr-mcp-05), [FR-MCP-06](../../context.md#fr-mcp-06), [NFR-DATA-04](../../context.md#nfr-data-04), [NFR-PERF-06](../../context.md#nfr-perf-06), [NFR-TEST-05](../../context.md#nfr-test-05).

## Open questions

- `.leo/config.json` vs `data.json` precedence ([SRS open question](../../context.md#open-questions) "`.leo/config.json` vs settings tab") — this feature reads `.leo/config.json` as authoritative for MCP entries; the F55 settings-tab writes back through the same file so the stores do not diverge. Verifier to confirm that path before F55 lands.
- Secret placeholder shape — SRS does not fix a syntax. Proposing `"safestorage:<key>"` as the literal sigil since it is explicit, backwards-compatible with non-secret string values, and easy to lint. Verifier to accept or pick an alternative before F55 persists it.
- Zod compilation of MCP tool JSON Schema — SRS is silent on the converter. Proposing the same helper chosen for F16's Zod → JSON Schema path, run in reverse via a thin transformer; fallback is an `unknown`-typed passthrough with runtime JSON Schema validation. Verifier to pick.
- `child_process` accessibility in the Obsidian renderer — [tech-stack open decision](../../../../standards/tech-stack.md#open-decisions) flags this. F51 must validate the SDK's `StdioClientTransport` actually spawns from the renderer before the fixture test can claim `NFR-TEST-05` coverage; if it cannot, stdio stays behind a feature flag and only SSE ships in phase 6.
- `resources/list` / `prompts/list` behaviour on servers that do not implement them — MCP spec permits absence. Proposing treat `MethodNotFound` as empty list with a `debug` log; verifier to confirm versus a hard `warn`.
