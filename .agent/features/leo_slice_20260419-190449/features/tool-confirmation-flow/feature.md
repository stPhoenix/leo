# F17 — Tool confirmation prompt

## Purpose

Deliver the inline tool-confirmation flow that pauses the serial turn loop whenever [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) surfaces a `ToolSpec` with `requiresConfirmation: true` — the agent emits a [`StreamEvent.tool_confirmation`](../../../../architecture/architecture.md#4-key-contracts) per [FR-AGENT-10](../../context.md#fr-agent-10), the `ChatView` renders an inline prompt in the `InlineConfirmation` region scaffolded by [F04 chat-sidebar-view](../chat-sidebar-view/feature.md) showing the tool name, pretty-printed args, and three actions — Allow once / Allow for thread / Deny — per [FR-CHAT-13](../../context.md#fr-chat-13), the awaiting-confirmation amber palette + per-tool icon are sourced from [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) with a read-vs-write visual distinction per [NFR-USE-04](../../context.md#nfr-use-04), Allow-for-thread persists the tool id into the thread's `metadata.allowedTools` via [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) per [FR-AGENT-11](../../context.md#fr-agent-11), and the whole lifecycle is modelled as an explicit Vitest-unit-covered state machine per [NFR-TEST-01](../../context.md#nfr-test-01). This is the seam every write tool (F19, F20, F24) and every MCP tool (F51+) depends on to reach the user safely.

## Scope

### In scope

- `ConfirmationController` module owning the confirmation lifecycle as an explicit state machine (`idle → awaiting-user → resolving → resolved(allow-once|allow-thread|deny) → idle`) wired into [F10 agent-controller-core](../agent-controller-core/feature.md)'s serial turn loop via the [LangGraph `interrupt()` pattern named in tech-stack](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring), pausing the graph before the tool node and resuming with the user's `Decision` per [FR-AGENT-10](../../context.md#fr-agent-10) and [architecture §5.3](../../../../architecture/architecture.md#53-chat-turn-with-tool-call--confirmation).
- Pre-invoke gate in the turn loop: when `ToolRegistry.lookup(id)` returns `requiresConfirmation: true` AND the tool id is NOT present in the current thread's `metadata.allowedTools`, the agent emits `StreamEvent.tool_confirmation{call, resolve}` and suspends until the `ChatView` resolves with a `Decision` per [FR-AGENT-10](../../context.md#fr-agent-10).
- Inline confirmation dialog rendered into the `InlineConfirmation` region from [F04 chat-sidebar-view](../chat-sidebar-view/feature.md) — never an Obsidian native `Modal` per [FR-UI-08](../../context.md#fr-ui-08) — with: tool icon via [F13 `iconFor(toolId)`](../ui-visual-states-notifications/feature.md), tool name, pretty-printed JSON arguments (indented, whitespace-preserved), and three buttons — `Allow once` / `Allow for thread` / `Deny` per [FR-CHAT-13](../../context.md#fr-chat-13).
- Read-vs-write visual distinction: the dialog's header band and primary-action tint differentiate read tools (neutral accent) from write / destructive tools (amber `awaiting-confirmation` palette from [F13](../ui-visual-states-notifications/feature.md)), driven by the spec's `requiresConfirmation` category plus icon family (read vs write/edit) per [NFR-USE-04](../../context.md#nfr-use-04).
- Assertive screen-reader announcement when the dialog mounts (`aria-live="assertive"` + `role="dialog"` + `aria-modal="true"` per [NFR-USE-08](../../context.md#nfr-use-08) and [NFR-USE-07](../../context.md#nfr-use-07)); focus moves to the primary action, focus is trapped inside the dialog until a `Decision` is emitted.
- Thread-scoped allowlist persistence: `Allow for thread` appends the tool id to `thread.metadata.allowedTools` through `ConversationStore.mutate(...)` per [F14](../conversation-persistence-v1/feature.md) and [FR-AGENT-11](../../context.md#fr-agent-11); future calls of the same tool in the same thread bypass the dialog.
- Deny path: emits `ToolResult{ok:false, error:"user denied <toolId>"}` back to the graph per [architecture §7](../../../../architecture/architecture.md#7-error-handling-strategy); the model sees a tool-error and the turn continues without invoking the tool.
- Keyboard: Esc equals Deny (consistent with `NFR-USE-06`); Tab / Shift-Tab cycle the three buttons in the order Allow once → Allow for thread → Deny.
- Vitest unit coverage of the confirmation state machine per [NFR-TEST-01](../../context.md#nfr-test-01): pause-on-requires-confirmation, bypass-when-thread-allowed, Allow-once resumes without persisting, Allow-for-thread persists via `ConversationStore`, Deny emits tool-error, Esc equals Deny, focus-trap + assertive announcement, no native `Modal` invoked.
- Structured log events `tool.confirmation.request / allow-once / allow-thread / deny` via the [F01 Logger](../plugin-bootstrap-logging/feature.md).

### Out of scope

- Write tool implementations (`create_note`, `append_to_note`) — ship with [F19 tools-write-vault](../../features-index.md); this feature only routes their confirmations.
- `edit_note` under CM6 edit lock, accept/reject inline diff — ships with [F20 tool-edit-note-with-lock](../../features-index.md).
- Plan mode write-tool gating (block all write tools while `mode === 'plan'`) — ships with [F24 plan-mode-permission-gate](../../features-index.md) per [FR-PLAN-05](../../context.md#fr-plan-05) / [NFR-REL-07](../../context.md#nfr-rel-07); plan-mode precedence is a separate gate, not this dialog.
- MCP-specific `requiresConfirmation: true` defaulting for `mcp.<server>.<tool>` ids — ships with [F52 mcp-tool-namespace-confirmation](../../features-index.md) per [FR-MCP-07](../../context.md#fr-mcp-07).

## Acceptance criteria

1. When the turn loop encounters a `tool_call` whose `ToolSpec.requiresConfirmation === true` and whose id is absent from `thread.metadata.allowedTools`, the `AgentRunner` emits `StreamEvent.tool_confirmation{call, resolve}` and the tool is NOT invoked until `resolve(decision)` is called. (FR-AGENT-10, FR-CHAT-13)
2. The `ChatView` renders the confirmation into the `InlineConfirmation` region from [F04](../chat-sidebar-view/feature.md) with the tool icon via [F13 `iconFor`](../ui-visual-states-notifications/feature.md), the tool name, pretty-printed JSON arguments, and exactly three buttons labelled `Allow once`, `Allow for thread`, `Deny`; a Vitest assertion verifies Obsidian's native `Modal` API is never invoked along this path. (FR-CHAT-13, FR-AGENT-10)
3. The dialog visually distinguishes read tools (neutral accent) from write / destructive tools (amber `awaiting-confirmation` palette sourced from [F13](../ui-visual-states-notifications/feature.md)), driven by `ToolSpec.requiresConfirmation` category + icon family — zero hardcoded colours; a Vitest snapshot asserts the `data-visual-state` attribute differs between read and write specs. (NFR-USE-04, FR-CHAT-13)
4. On mount the dialog receives focus, sets `role="dialog" aria-modal="true"` with `aria-live="assertive"`, traps Tab / Shift-Tab inside its three buttons in order Allow once → Allow for thread → Deny, and Esc is equivalent to Deny. (FR-CHAT-13, NFR-USE-04)
5. `Allow once` resolves with `{decision:"allow-once"}` — the tool executes for this call only; `thread.metadata.allowedTools` is NOT mutated; the next call of the same tool in the same thread re-prompts. (FR-AGENT-10, FR-AGENT-11)
6. `Allow for thread` resolves with `{decision:"allow-thread"}` — the tool id is appended to `thread.metadata.allowedTools` through [`ConversationStore.mutate`](../conversation-persistence-v1/feature.md) (deduplicated), the tool executes, and subsequent calls of the same tool in the same thread bypass the dialog and run directly; the allowlist persists across plugin reloads per [F14](../conversation-persistence-v1/feature.md). (FR-AGENT-11, FR-CHAT-13)
7. `Deny` (including Esc) resolves with `{decision:"deny"}` — the tool is NOT invoked, a `ToolResult{ok:false, error:"user denied <toolId>"}` is fed back into the graph so the model can respond, and no mutation to `thread.metadata.allowedTools` occurs. (FR-AGENT-10)
8. Vitest unit suite covers the confirmation state machine end-to-end per [NFR-TEST-01](../../context.md#nfr-test-01): pause-on-requires-confirmation, bypass-when-thread-allowed, Allow-once vs Allow-for-thread persistence difference, Deny → tool-error, Esc = Deny, focus-trap + assertive live-region assertion, and zero native-`Modal` invocations; structured log events `tool.confirmation.request / allow-once / allow-thread / deny` fire with `{toolId, thread, decision}`. (NFR-TEST-01, FR-AGENT-10, FR-AGENT-11, FR-CHAT-13)

## Dependencies

- [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) — provides the `ToolSpec.requiresConfirmation` flag, the `ToolRegistry.lookup` / `invoke` surface, and the serial `tool_call` → invoke → `tool_result` loop this feature pauses and resumes per [FR-AGENT-04](../../context.md#fr-agent-04), [FR-PROV-04](../../context.md#fr-prov-04), [FR-AGENT-06](../../context.md#fr-agent-06).
- [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) — provides the `ConversationStore` with `thread.metadata.allowedTools: string[]` where `Allow for thread` writes and future turns read to bypass the dialog per [FR-CHAT-08](../../context.md#fr-chat-08), [FR-AGENT-11](../../context.md#fr-agent-11).
- [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) — provides the `awaiting-confirmation` visual state + palette, the `iconFor(toolId)` registry, and the "tool confirmation never uses native Modal" routing rule per [FR-UI-05](../../context.md#fr-ui-05), [FR-UI-06](../../context.md#fr-ui-06), [FR-UI-08](../../context.md#fr-ui-08).
- Drives requirements [FR-CHAT-13](../../context.md#fr-chat-13), [FR-AGENT-10](../../context.md#fr-agent-10), [FR-AGENT-11](../../context.md#fr-agent-11), [NFR-USE-04](../../context.md#nfr-use-04), [NFR-TEST-01](../../context.md#nfr-test-01).
- Downstream consumers tracked in [features-index.md](../../features-index.md): F19 (write tools route through this dialog), F20 (`edit_note` confirmation before lock acquisition), F24 (plan-mode gate layered before this dialog), F52 (MCP-specific defaults feeding the same dialog).

## Implementation notes

- [Architecture §1 — Interrupt-driven tool flow](../../../../architecture/architecture.md#1-architectural-principles) — LangGraph `interrupt()` pauses the graph before `requiresConfirmation` tools; no ad-hoc event bus.
- [Architecture §4 Key Contracts — `StreamEvent.tool_confirmation`](../../../../architecture/architecture.md#4-key-contracts) — pins `{call, resolve(d: Decision)}`; this feature consumes that union member.
- [Architecture §5.3 Chat Turn (with tool call + confirmation)](../../../../architecture/architecture.md#53-chat-turn-with-tool-call--confirmation) — canonical sequence: `tool_call` → `lookup` → `tool_confirmation` → Allow/Deny → resume.
- [Architecture §6 State Ownership](../../../../architecture/architecture.md#6-state-ownership) — per-thread allowlist owned by `ConversationStore` in thread JSON; this feature is its sole writer.
- [Architecture §7 Error Handling Strategy](../../../../architecture/architecture.md#7-error-handling-strategy) — "User denies tool" produces `tool-error "user denied <toolId>"`.
- [Architecture §10 Concurrency & Lifecycle Rules](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — teardown on `ChatView` close: focus-trap detached, `resolve(deny)` fired on any pending confirmation.
- [Tech stack — Agent / Tool / Skill / MCP Wiring](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) — LangGraph `interrupt()` pattern with per-thread allowlist persisted with the thread.
- [Code style — React 18](../../../../standards/code-style.md#react-18) — portals for tool-confirmation modals; don't nest inside chat tree.
- [Code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer) — `AbortSignal` threaded through; tool results typed `{ok, data}|{ok, error}`; the Deny branch synthesises the error surface.
- [Code style — Styling (Tailwind + Obsidian)](../../../../standards/code-style.md#styling-tailwind--obsidian) — Obsidian CSS variables for the read-vs-write distinction; zero hex literals.
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — confirmation routed inline, never via Obsidian `Modal`.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — selects the harness called out by AC 8 and [NFR-TEST-01](../../context.md#nfr-test-01).
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — single source of truth for the confirmation state machine; every tool gate funnels through it.

## Open questions

- Pretty-print policy for nested / very large `args` — SRS fixes "pretty-printed" but not a size cap. Proposing JSON `stringify(args, null, 2)` with a soft cap (e.g. 4 KB) after which trailing bytes collapse to `… (truncated, N bytes total)`; verifier to confirm.
- Allow-for-thread scope when the thread's active skill changes mid-thread ([FR-SKILL-06](../../context.md#fr-skill-06)) — does a later skill with a narrower `allowedTools` still honour the allowlist, or does the skill filter take precedence? Proposing: skill filter gates visibility first; allowlist only suppresses the dialog for tools that are visible. Flagged for F22.
