# F55 — MCP server management UI

## Purpose

Light the MCP Servers section of [F03](../settings-tab-scaffold/feature.md)'s settings-tab scaffold with a full add / edit / remove / enable-toggle / per-server connection-status surface per [FR-MCP-10](../../context.md#fr-mcp-10), writing non-secret entries back to `.leo/config.json` `mcpServers` (the authoritative store [F51](../mcp-client-config-transports/feature.md) loads from per [FR-MCP-03](../../context.md#fr-mcp-03)) and routing any `env` / `headers` flagged secret through [F38](../cloud-providers-safestorage/feature.md)'s `SafeStorage` with the `"safestorage:<key>"` placeholder F51 resolves, so vault plaintext is never a secret store per [NFR-DATA-04](../../context.md#nfr-data-04). Each row observes [F51](../mcp-client-config-transports/feature.md)'s `ServerRuntime.status` (connecting / connected / failed / disabled) plus tool / resource / prompt counts and wires Enable/Disable/Retry actions back to the `MCPClient.connect(serverId)` / `disconnect(serverId)` seam so users can recover from configuration mistakes without reloading the plugin. All form fields are native Obsidian `Setting` rows under the existing collapsible section from [FR-UI-10](../../context.md#fr-ui-10) — the settings tab stays the single source of user-facing configuration per [NFR-USE-01](../../context.md#nfr-use-01) — and every lifecycle event flows through [F01](../plugin-bootstrap-logging/feature.md)'s `Logger`.

## Scope

### In scope

- `McpServersSection` React subtree mounted into [F03](../settings-tab-scaffold/feature.md)'s existing MCP Servers collapsible section, listing every entry from `.leo/config.json` `mcpServers` as a row with `{id, transport, enabled, status, toolCount, resourceCount, promptCount}` summary and per-row action buttons `[Edit]` / `[Enable|Disable]` / `[Retry]` / `[Delete]`.
- Add-server flow: a `[+ Add MCP server]` button opens an inline editor form (never a native `Modal` per [FR-UI-08](../../context.md#fr-ui-08)) with fields for `id`, transport (`stdio` | `sse`), `enabled`, plus transport-specific fields — stdio: `command`, `args[]`, `env{}`; sse: `url`, `headers{}` — each validated against the same Zod `McpServerConfig` schema F51 reads so bad entries never reach disk.
- Edit-server flow: same form pre-populated from the selected row, `id` field disabled to preserve registry keys stable for F52's `thread.metadata.allowedTools` entries; Save writes the updated entry and triggers `MCPClient.reload(serverId)` so the runtime picks up the change without `plugin.onunload`.
- Remove-server flow: `[Delete]` opens an inline confirmation prompt (reusing [F13](../ui-visual-states-notifications/feature.md)'s inline dialog surface, not a native Modal) with a stale-approvals warning; on confirm, the entry is removed from `.leo/config.json`, its `ServerRuntime` is disposed via `MCPClient.disconnect(serverId)`, and every `mcp.<serverId>.*` row in `ToolRegistry` is unregistered; per-thread `allowedTools` entries from [F14](../conversation-persistence-v1/feature.md) are left in place (no cross-thread cleanup) — they simply become unmatchable until the user re-adds a server with the same `id`, matching [F52](../mcp-tool-confirmation/feature.md)'s re-match-on-same-id semantic.
- Toggle-enabled flow: `[Enable|Disable]` flips `mcpServers[id].enabled` and either fans out `MCPClient.connect(serverId)` (disabled → enabled) or `MCPClient.disconnect(serverId)` (enabled → disabled); the UI row reflects the transition via the status subscription below.
- Retry flow: `[Retry]`, visible only when `ServerRuntime.status === "failed"`, calls `MCPClient.connect(serverId)` to re-attempt; the action is disabled during in-flight `connecting` status and aligns with the [SRS open question](../../context.md#open-questions) "MCP reconnect disable (FR-MCP-11)" user-re-trigger gap so F56's automated reconnect and this manual retry compose cleanly.
- Live status subscription: `McpServersSection` subscribes to an `MCPClient.onStatusChange(serverId, next)` emitter (a thin observer on top of F51's existing `ServerRuntime.status` field) so the row badge, counts, and action-button availability update in-place without a section re-mount or polling.
- Secret-field handling: `env` (stdio) / `headers` (sse) entries with a `secret: true` flag in the UI route the plaintext value through [F38](../cloud-providers-safestorage/feature.md) `SafeStorage.set(key)` at save time, persist only `"safestorage:<key>"` into `.leo/config.json`, and render a masked `••••••` cell when displayed — plaintext is never shown after the edit form closes.
- Field validation: the same Zod `McpServerConfig` schema from [F51](../mcp-client-config-transports/feature.md) validates the form before save (`id` non-empty + unique + URL-safe, transport enum, stdio `command` non-empty, sse `url` parseable), rendering per-field errors inline; save is disabled on any validation error.
- Persistence split: the loader reads `.leo/config.json` (vault) as authoritative per [architecture §6](../../../../architecture/architecture.md#6-state-ownership) "MCP server configs → mixed" row; the non-secret mirror lives in `.leo/config.json`, only secret placeholders resolve through `SafeStorage`, and save uses `Vault` / `VaultAdapter` atomic write per [code-style Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns).
- Structured log events (`mcp.settings.add`, `mcp.settings.edit`, `mcp.settings.delete`, `mcp.settings.toggle`, `mcp.settings.retry`, `mcp.settings.secret.store`) through [F01](../plugin-bootstrap-logging/feature.md) carrying `{serverId, transport, enabled, field?, durationMs}` only; no `command`, `args`, `env`, `headers`, or `url` values above `debug` per [code-style Logging](../../../../standards/code-style.md#logging).
- Vitest coverage: add / edit / delete round-trip against an in-memory `VaultAdapter` + mocked `MCPClient` + mocked `SafeStorage` asserting `.leo/config.json` shape, secret-placeholder substitution, status-subscription re-render, and negative FS sniff for plaintext secrets.

### Out of scope

- `MCPClient` + transports + parallel startup + tool registration — already shipped by [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md); this feature only calls the `connect(serverId)` / `disconnect(serverId)` / `reload(serverId)` seams F51 exposes.
- Tool-call confirmation dialog, `requiresConfirmation: true` default, per-thread `allowedTools` persistence — shipped by [F52 mcp-tool-confirmation](../mcp-tool-confirmation/feature.md); this feature does not alter confirmation behaviour on add / remove.
- Resource picker and content insertion — shipped by [F53 mcp-resources-picker](../mcp-resources-picker/feature.md); this section only reports the per-server `resourceCount`.
- Skill-picker "From MCP" section — shipped by [F54 mcp-prompts-in-skills](../mcp-prompts-in-skills/feature.md); this section only reports `promptCount`.
- Automated exponential-backoff reconnect and clean `SIGTERM` → `SIGKILL` shutdown — shipped by [F56 mcp-reconnect-shutdown](../../features-index.md); the manual `[Retry]` button here composes with, but does not replace, F56.
- `SafeStorage` adapter itself — already shipped by [F38 cloud-providers-safestorage](../cloud-providers-safestorage/feature.md); this feature only consumes `get(key)` / `set(key, value)`.
- Cross-thread `thread.metadata.allowedTools` cleanup on server delete — left to each thread's natural unmatchable-id behaviour per the [SRS open question](../../context.md#open-questions) "MCP tool per-thread pre-approval semantics".

## Acceptance criteria

1. Opening Obsidian Settings → Leo → MCP Servers renders the section scaffolded by [F03](../settings-tab-scaffold/feature.md) populated with one row per entry in `.leo/config.json` `mcpServers`, each showing `id`, transport, `enabled` toggle, live status badge (`connecting` / `connected` / `failed` / `disabled`), and `{toolCount, resourceCount, promptCount}` counts sourced from F51's `ServerRuntime`. ([FR-MCP-10](../../context.md#fr-mcp-10), [FR-UI-10](../../context.md#fr-ui-10))
2. Clicking `[+ Add MCP server]` opens an inline form (never a native `Modal`); filling valid `stdio` fields (`id`, `command`, `args[]`, optional `env{}`) or `sse` fields (`id`, `url`, optional `headers{}`) and pressing Save writes the entry to `.leo/config.json` `mcpServers` in the shape F51 consumes, and triggers `MCPClient.connect(serverId)` so the new row appears in `connecting` → `connected` without a plugin reload. ([FR-MCP-10](../../context.md#fr-mcp-10), [FR-MCP-03](../../context.md#fr-mcp-03), [FR-UI-08](../../context.md#fr-ui-08))
3. Editing an existing row opens the same form pre-populated with `id` disabled; saving writes the new fields to `.leo/config.json` and calls `MCPClient.reload(serverId)`, which atomically disconnects the prior `ServerRuntime` and reconnects with the updated config so subsequent `tools/list` reflects the edit. ([FR-MCP-10](../../context.md#fr-mcp-10))
4. Deleting a row shows an inline confirmation, then removes the entry from `.leo/config.json`, calls `MCPClient.disconnect(serverId)`, unregisters every `mcp.<serverId>.*` ToolSpec from F16's `ToolRegistry`, and the row disappears from the section; per-thread `allowedTools` entries from F14 are NOT mutated (unmatchable until re-add with same `id`). ([FR-MCP-10](../../context.md#fr-mcp-10))
5. Toggling a row's `[Enable|Disable]` flips `mcpServers[id].enabled` and fans out `MCPClient.connect(serverId)` or `MCPClient.disconnect(serverId)`; a `[Retry]` button, visible only when `ServerRuntime.status === "failed"`, re-invokes `connect(serverId)`, and both paths update the row status badge in-place via the `MCPClient.onStatusChange` subscription. ([FR-MCP-10](../../context.md#fr-mcp-10))
6. Any `env` / `headers` entry flagged secret in the UI is persisted by calling `SafeStorage.set(key, plaintext)` and writing only `"safestorage:<key>"` into `.leo/config.json`; the cell renders `••••••` on subsequent view; a Vitest FS sniff over `.leo/config.json` after a full add → save → view cycle asserts that no plaintext secret string appears in the vault file. ([NFR-DATA-04](../../context.md#nfr-data-04), [FR-MCP-03](../../context.md#fr-mcp-03))
7. Structured log events `mcp.settings.add`, `mcp.settings.edit`, `mcp.settings.delete`, `mcp.settings.toggle`, `mcp.settings.retry`, `mcp.settings.secret.store` emit via [F01](../plugin-bootstrap-logging/feature.md)'s `Logger` with fields `{serverId, transport, enabled, field?, durationMs}`; `command`, `args`, `env`, `headers`, and `url` values are never logged at `info` or above. ([NFR-LOG-04](../../context.md#nfr-log-04), [NFR-DATA-04](../../context.md#nfr-data-04))
8. Vitest suite drives add → edit → delete → toggle → retry against an in-memory `VaultAdapter` + mocked `MCPClient` + mocked `SafeStorage`, asserting `.leo/config.json` byte-shape matches F51's Zod schema, `SafeStorage.set` is called for every secret field, the section re-renders on `onStatusChange`, and no plaintext secret is written to disk. ([NFR-TEST-05](../../context.md#nfr-test-05), [NFR-TEST-01](../../context.md#nfr-test-01))

## Open questions

- Status-subscription emitter shape — F51's published seam is `ServerRuntime.status` as a field; this feature proposes a thin `MCPClient.onStatusChange(serverId, cb)` emitter layered on top (single-listener-per-serverId acceptable). Verifier to confirm F51 exposes or will expose this seam before this feature lands.
- Retry vs F56 reconnect interleave — when F56 ships automatic exponential backoff, the manual `[Retry]` button here must cancel any pending backoff timer before calling `connect(serverId)` to avoid double-connects. Verifier to check F56's contract exposes a `cancelReconnect(serverId)` seam or equivalent.
- Secret-field opt-in UX — SRS does not fix how a user flags a value as secret. Proposing a per-row checkbox "Store as secret" on each `env` / `headers` entry so the user intentionally opts in; alternative is an auto-heuristic on field name (`*_KEY`, `*_TOKEN`). Verifier to pick before F38 persists it.
- `id` uniqueness scope — should the form reject an `id` that matches a previously-deleted-but-still-referenced id (a thread with `allowedTools` still pointing at it)? Proposing allow, since F52's re-match-on-same-id behaviour is the intended recovery path for users who mistyped and fixed the config.
- Edit-form `id` lock override — the `id` field is disabled on edit to preserve registry keys stable; if the user really wants to rename, they must delete + re-add. Verifier to confirm this constraint is acceptable, or we add a two-step "rename" flow that atomically re-keys `thread.metadata.allowedTools` entries across all threads in [F14](../conversation-persistence-v1/feature.md).
- Connection-status latency — `MCPClient.connect(serverId)` can take several seconds on stdio spawn; the row badge stays in `connecting` with a spinner but the Save button has already closed the form. Verifier to confirm no modal blocker is needed during `connecting`.
