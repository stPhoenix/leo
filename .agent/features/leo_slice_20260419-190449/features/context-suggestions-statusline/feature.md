# F48 — `/context` suggestions + token status line

## Purpose

Deliver the two remaining downstream surfaces on top of [F47](../context-command-grid/feature.md)'s `ContextData`: (1) a pure `generateContextSuggestions(data)` engine with the six thresholds from [context.md §12.1](../../../../srs/context.md#121-thresholds) (`NEAR_CAPACITY_PERCENT = 80`, `LARGE_TOOL_RESULT_PERCENT = 15`, `LARGE_TOOL_RESULT_TOKENS = 10_000`, `READ_BLOAT_PERCENT = 5`, `MEMORY_HIGH_PERCENT = 5`, `MEMORY_HIGH_TOKENS = 5_000`), executing the five ordered checks from [§12.2](../../../../srs/context.md#122-suggestion-checks-in-order) (near-capacity, large-tool-results with per-tool advice for Bash / Read / Grep / WebFetch / generic, read-result bloat, memory bloat, autocompact-disabled), and sorting warnings-first then by `savingsTokens` descending per [§12.3](../../../../srs/context.md#123-sorting); and (2) a persistent token status-line widget in the chat view reporting `{total_input_tokens, total_output_tokens, context_window_size, current_usage, used_percentage, remaining_percentage}` updated on last-assistant-message-id change, debounced 500ms, non-blocking, as drawn in [context.md §14](../../../../srs/context.md#14-status-line-integration). Satisfies [FR-CTX-05](../../context.md#fr-ctx-05) and [FR-CTX-06](../../context.md#fr-ctx-06).

## Scope

### In scope

- Pure `generateContextSuggestions(data: ContextData): ContextSuggestion[]` module in the domain/core layer (no Obsidian / React / fetch imports) consuming the `ContextData` produced by [F46](../context-analyzer-pipeline/feature.md) and surfaced into the UI layer by [F47](../context-command-grid/feature.md); returns a typed `ContextSuggestion[]` with `{severity: 'info' | 'warning', title, detail, savingsTokens?}` per [context.md §7 ContextSuggestion](../../../../srs/context.md#7-data-model--types).
- Threshold constants exported as a single frozen `CONTEXT_SUGGESTION_THRESHOLDS` object with the six values from [context.md §12.1](../../../../srs/context.md#121-thresholds) and [§16](../../../../srs/context.md#16-constants-reference): `NEAR_CAPACITY_PERCENT = 80`, `LARGE_TOOL_RESULT_PERCENT = 15`, `LARGE_TOOL_RESULT_TOKENS = 10_000`, `READ_BLOAT_PERCENT = 5`, `MEMORY_HIGH_PERCENT = 5`, `MEMORY_HIGH_TOKENS = 5_000`; constants owned here (not restated elsewhere) and re-exported for any future re-use.
- Five ordered suggestion checks executed exactly as written in [context.md §12.2](../../../../srs/context.md#122-suggestion-checks-in-order): (1) near-capacity at `percentage >= 80` — `severity: 'warning'` with autocompact-aware detail text per [§12.2 #1](../../../../srs/context.md#1-near-capacity-percentage--80); (2) large-tool-results at `> 15% AND > 10k tokens` with the per-tool table (Bash = warning / 50% savings, Read = info / 30%, Grep = info / 30%, WebFetch = info / 40%, generic `>= 20%` = info / 20%) per [§12.2 #2](../../../../srs/context.md#2-large-tool-results--15-and--10k-tokens); (3) read-result bloat at `>= 5% AND >= 10k tokens` gated on not-already-flagged per [§12.2 #3](../../../../srs/context.md#3-read-result-bloat--5-and--10k-tokens); (4) memory bloat at `>= 5% AND >= 5k tokens` exposing the top 3 largest memory files per [§12.2 #4](../../../../srs/context.md#4-memory-bloat--5-and--5k-tokens); (5) autocompact-disabled at `50% <= usage < 80%` per [§12.2 #5](../../../../srs/context.md#5-autocompact-disabled-50--usage--80).
- Sorting pass per [context.md §12.3](../../../../srs/context.md#123-sorting): warnings before info, then descending `savingsTokens` (entries without `savingsTokens` treated as `0` for the tiebreak); stable sort preserves check-order for equal-key ties.
- `savingsTokens` estimation per the [§12.2 #2 per-tool table](../../../../srs/context.md#2-large-tool-results--15-and--10k-tokens): Bash/Read/Grep/WebFetch/generic apply the listed multipliers (50% / 30% / 30% / 40% / 20%) to the offending category's token count; near-capacity reports the autocompact-buffer-sized savings estimate; memory-bloat reports the sum of the top 3 listed files above threshold; read-bloat reports 30% of the read-result tokens.
- Suggestion rendering inside the [F47](../context-command-grid/feature.md) detail column bottom slot per [context.md §12.4](../../../../srs/context.md#124-suggestion-rendering): status glyph by severity (warning uses `setIcon('alert-triangle')`, info uses `setIcon('info')`), bold title, optional `→ save ~<formatTokens(savings)>` tail when `savingsTokens` is set, indented dimmed detail line; formatting via [F47](../context-command-grid/feature.md)'s shared `formatTokens()` helper; no inline color literals (Obsidian CSS variables only).
- Persistent status-line widget mounted as an Obsidian `Plugin.addStatusBarItem()` entry scoped to the plugin lifecycle, showing a compact rendering of `{used_percentage}% · <formatTokens(current_usage)> / <formatTokens(context_window_size)>` with a tooltip carrying the full `{total_input_tokens, total_output_tokens, remaining_percentage}` triple per [context.md §14 Context Window Section](../../../../srs/context.md#14-status-line-integration); the widget is disposed on `plugin.onunload()` via `Plugin.register(...)`.
- Status-line data builder `buildStatusLineContext(apiUsage, contextWindowSize)` returning the exact six-field record from [context.md §14](../../../../srs/context.md#14-status-line-integration): `total_input_tokens = usage.input_tokens + usage.cache_creation_input_tokens + usage.cache_read_input_tokens`, `total_output_tokens = usage.output_tokens`, `context_window_size`, `current_usage = total_input_tokens`, `used_percentage = clamp(round((current_usage / context_window_size) * 100), 0, 100)`, `remaining_percentage = 100 - used_percentage`. Pure function, no side effects.
- Update trigger wired to the chat thread's "last assistant message id changed" event from [F07 chat-streaming-stop](../chat-streaming-stop/feature.md)/[F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md): subscribe on mount, debounce by 500ms (trailing edge) per [context.md §14 Status Line Update Triggers](../../../../srs/context.md#14-status-line-integration), recompute the record, and write it into the status-bar widget; the debounce timer is tracked on the owning `Component` and flushed on teardown so no writes leak after `onunload`.
- Non-blocking background execution: the update callback runs via `queueMicrotask` / `setTimeout(0)` wrapped in a best-effort `try/catch`; any throw routes through [F01 Logger](../plugin-bootstrap-logging/feature.md) as a structured `context.statusline.error` event and keeps the last-known-good widget text in place. The widget never awaits a network call.
- Integration with the [F43 autocompact](../compaction-autocompact/feature.md) flag: the near-capacity suggestion branch reads `ContextData.isAutoCompactEnabled` to pick between the two detail strings from [context.md §12.2 #1](../../../../srs/context.md#1-near-capacity-percentage--80) ("Use /compact now to control what gets kept" vs "Use /compact or enable autocompact"); no other interaction with the autocompact state.
- Vitest coverage per [NFR-TEST-08](../../context.md#nfr-test-08): threshold fixtures at each boundary (`79/80/81%`, `14/15/16%` with `9_999/10_000/10_001` tokens, `4/5/6%` with `4_999/5_000/5_001` tokens, `49/50/80%` autocompact-disabled window); per-tool advice fixtures for Bash / Read / Grep / WebFetch / generic with the expected severity + savings multiplier; sorting fixture asserting warnings-first + `savingsTokens`-descending + stable tiebreak; read-bloat not-already-flagged fixture; top-3 memory-file fixture; status-line record fixture covering API-usage presence, API-usage absence (widget shows `—%` sentinel without throwing), and 500ms debounce fixture using fake timers.

### Out of scope

- `ContextData` shape, category ordering, grid allocation, partial-square fullness, command registration, and the right-column detail renderer — all owned by [F47](../context-command-grid/feature.md); this feature only inserts its suggestion block into F47's layout and emits to the status-bar entry point.
- `analyzeContextUsage()`, post-compact-boundary filter, microcompact pass, the seven parallel counting ops, and API-usage extraction — owned by [F46](../context-analyzer-pipeline/feature.md); this feature consumes `ContextData` as a read-only input.
- Token-warning state and thresholds from [context.md §13](../../../../srs/context.md#13-token-warnings) ("{percentLeft}% until auto-compact" banner / "Context low · Run /compact" inline notice) — these are inline conversation-UI warnings (not the status line or the `/context` suggestions) and are carried by [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md)'s notification channel; this feature does not render them. The overlap between FR-CTX-06 and §13 is resolved by this split; see Open questions.
- Per-assistant-message token footer (input / output / total per turn) — owned by [F12 token-usage-indicator](../token-usage-indicator/feature.md); this feature only reads the same API-usage payload for the status-line record.
- Cost-in-$ display for cloud providers — owned by [F38 cloud-providers-safestorage](../cloud-providers-safestorage/feature.md); phase 1 Leo is LM Studio only.
- Non-interactive Markdown `/context` output per [context.md §11](../../../../srs/context.md#11-non-interactive-output-markdown) — Leo is Obsidian-only (no REPL); dropped in v1 and already marked out-of-scope by [F47](../context-command-grid/feature.md).
- Visual wireframes, ARIA roles, keyboard reachability, focus management, color-token mapping, icon selection, reduced-motion gating — delivered by the `ui.md` companion.

## Acceptance criteria

1. `generateContextSuggestions(data)` emits a near-capacity warning when `data.percentage >= 80` and omits it when `data.percentage < 80`; the detail string is "Use /compact now to control what gets kept" when `data.isAutoCompactEnabled === true` and "Use /compact or enable autocompact" when `false`, per [context.md §12.2 #1](../../../../srs/context.md#1-near-capacity-percentage--80). Verified by three Vitest fixtures at `79 / 80 / 81%` plus the autocompact on/off branches. ([FR-CTX-05](../../context.md#fr-ctx-05), [NFR-TEST-08](../../context.md#nfr-test-08))
2. Large-tool-results checks fire at `tool-result-tokens > 15% AND > 10k tokens` and emit the per-tool advice from [context.md §12.2 #2](../../../../srs/context.md#2-large-tool-results--15-and--10k-tokens): Bash → `severity: 'warning'`, savings 50%; Read → `info`, 30%; Grep → `info`, 30%; WebFetch → `info`, 40%; generic at `>= 20%` → `info`, 20%. A Vitest table drives all five with boundary values (`9_999 / 10_000 / 10_001` tokens, `14 / 15 / 16%`) and asserts severity, title, detail text, and `savingsTokens` equal `round(tokens * multiplier)`. ([FR-CTX-05](../../context.md#fr-ctx-05), [NFR-TEST-08](../../context.md#nfr-test-08))
3. Read-result-bloat fires at `read-tokens >= 5% AND >= 10k` but ONLY when the large-tool-results check did not already flag the Read tool; the `info` suggestion references earlier reads / offset-limit parameters per [context.md §12.2 #3](../../../../srs/context.md#3-read-result-bloat--5-and--10k-tokens). Verified by two Vitest fixtures (one where large-tool-results already flagged Read → read-bloat suppressed; one where it did not → read-bloat emitted). ([FR-CTX-05](../../context.md#fr-ctx-05), [NFR-TEST-08](../../context.md#nfr-test-08))
4. Memory-bloat fires at `memory-tokens >= 5% AND >= 5k`, lists the top 3 largest memory files in the detail text, and points at `/memory`, per [context.md §12.2 #4](../../../../srs/context.md#4-memory-bloat--5-and--5k-tokens). Verified by a Vitest fixture with five memory files and asserts the detail names exactly the three largest in descending order. ([FR-CTX-05](../../context.md#fr-ctx-05), [NFR-TEST-08](../../context.md#nfr-test-08))
5. Autocompact-disabled `info` suggestion fires only when `data.isAutoCompactEnabled === false` AND `50 <= data.percentage < 80`, per [context.md §12.2 #5](../../../../srs/context.md#5-autocompact-disabled-50--usage--80). Verified by Vitest fixtures at `49 / 50 / 79 / 80%` with autocompact on/off; near-capacity fires at 80% independently and is not double-counted. ([FR-CTX-05](../../context.md#fr-ctx-05))
6. The returned `ContextSuggestion[]` is sorted warnings-first, then by `savingsTokens` descending, per [context.md §12.3](../../../../srs/context.md#123-sorting); entries without `savingsTokens` sort as if `0`; the sort is stable so equal-key ties preserve §12.2 check order. Verified by a Vitest fixture mixing two warnings and three info entries with varied savings. ([FR-CTX-05](../../context.md#fr-ctx-05), [NFR-TEST-08](../../context.md#nfr-test-08))
7. `generateContextSuggestions` is a pure function in the domain/core layer: no Obsidian `App` / `Plugin` imports, no `fetch`, no React, no logging side effects. A Vitest import-graph assertion pins the module to domain/core. ([Architecture §3.3](../../../../architecture/architecture.md#33-domain--core-pure), [FR-CTX-05](../../context.md#fr-ctx-05))
8. `buildStatusLineContext(apiUsage, contextWindowSize)` returns exactly the six fields from [context.md §14](../../../../srs/context.md#14-status-line-integration): `total_input_tokens = input_tokens + cache_creation_input_tokens + cache_read_input_tokens`, `total_output_tokens`, `context_window_size`, `current_usage = total_input_tokens`, `used_percentage = clamp(round(current_usage / context_window_size * 100), 0, 100)`, `remaining_percentage = 100 - used_percentage`. Verified by Vitest fixtures covering zero / typical / near-full / over-window (clamped to 100) cases and a null-usage fixture that returns `null` without throwing. ([FR-CTX-06](../../context.md#fr-ctx-06), [NFR-TEST-08](../../context.md#nfr-test-08))
9. The status-bar widget is created via `Plugin.addStatusBarItem()` on `plugin.onload()`, registered for auto-disposal via `Plugin.register(...)`, and subscribes to the last-assistant-message-id event from [F07](../chat-streaming-stop/feature.md) / [F14](../conversation-persistence-v1/feature.md) with a 500ms trailing-edge debounce per [context.md §14 Status Line Update Triggers](../../../../srs/context.md#14-status-line-integration). Verified by a Vitest fake-timer fixture that fires five events in 100ms and asserts exactly one widget write 500ms after the last event. ([FR-CTX-06](../../context.md#fr-ctx-06), [Architecture §10](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules))
10. Widget updates are non-blocking: the update callback never awaits a network call and a throw inside it is caught, logged as `context.statusline.error` via [F01](../plugin-bootstrap-logging/feature.md), and leaves the last-known-good widget text in place. The debounce timer is tracked on the owning `Component` and flushed during `plugin.onunload()` so no writes happen after teardown. Verified by a Vitest fixture injecting a throw and asserting widget text stability + log event + no post-unload writes. ([FR-CTX-06](../../context.md#fr-ctx-06), [Architecture §10](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules))

## Implementation notes

- [Architecture §3.1 UI Layer](../../../../architecture/architecture.md#31-ui-layer) — the suggestion render block lives as a React child of [F47](../context-command-grid/feature.md)'s right column; the status-bar widget is an Obsidian `statusBarItem` DOM element managed imperatively inside the plugin shell. No domain logic in either surface.
- [Architecture §3.2 Agent Layer](../../../../architecture/architecture.md#32-agent-layer) — agent layer owns the subscription from last-assistant-message-id → debounce → `buildStatusLineContext` → widget write; the debounce timer is tracked on the owning `Component` per [Architecture §10](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules).
- [Architecture §3.3 Domain / Core](../../../../architecture/architecture.md#33-domain--core-pure) — `generateContextSuggestions`, `buildStatusLineContext`, and `CONTEXT_SUGGESTION_THRESHOLDS` live in domain/core, fully pure and side-effect-free.
- [Architecture §4 Key Contracts](../../../../architecture/architecture.md#4-key-contracts) — consumes `ContextData`, `ContextCategory`, `ContextSuggestion`, `MessageBreakdown`, `MemoryFile`, `ApiUsage` authored by F46.
- [Architecture §10 Concurrency & Lifecycle Rules](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — debounce timer + event subscription + status-bar DOM node all registered for auto-disposal; no post-unload writes allowed; errors logged, never thrown into the event loop.
- [Tech stack — Platform APIs](../../../../standards/tech-stack.md#platform-apis), [UI Layer](../../../../standards/tech-stack.md#ui-layer) — `Plugin.addStatusBarItem()`, `Plugin.register()`, `Component`, `setIcon('alert-triangle' | 'info')`, React 18 for the suggestion block.
- [Code style — React 18](../../../../standards/code-style.md#react-18), [Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns), [Styling](../../../../standards/code-style.md#styling) — govern the suggestion-block component (presentational, pure render over props) and the widget's DOM-node treatment; Obsidian CSS variables only, zero color literals.
- [Code style — Async & Concurrency](../../../../standards/code-style.md#async--concurrency) — trailing-edge debounce implemented with a `setTimeout` handle cleared on re-entry and on teardown; no `setInterval`.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — threshold-boundary fixtures, per-tool advice table, sort stability, status-line record purity, fake-timer debounce test, teardown-safety test; matches [NFR-TEST-08](../../context.md#nfr-test-08).
- [Code style — Logging](../../../../standards/code-style.md#logging) — `context.statusline.error` structured event for widget-update failures; no content payload ever logged.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — single responsibility: the suggestion engine is a pure function over `ContextData`; the status-line widget is a thin subscription-plus-write shell.
- [context.md §12](../../../../srs/context.md#12-context-suggestions), [§14](../../../../srs/context.md#14-status-line-integration), [§16](../../../../srs/context.md#16-constants-reference) — authoritative source for every threshold, per-tool multiplier, sort rule, status-line field, and update-trigger rule referenced above. Not restated here.

## Open questions

- **FR-CTX-06 vs context.md §13 boundary (token warnings)**: FR-CTX-06 lists "Token warning state AND status-line integration per context.md §13–§14" together, but §13 describes an inline conversation-UI banner ("X% until auto-compact" / "Context low · Run /compact") whose thresholds are derived from `effectiveContextSize - 20_000` — a *different* surface from the persistent status-bar widget. Proposing this feature owns §14 (the status-bar widget) plus the `/context` suggestions from §12, and the §13 inline banner is routed through [F13](../ui-visual-states-notifications/feature.md)'s notification channel as a follow-up (no new feature row needed). Flagging for verifier.
- **Status-line widget location (FR-CTX-06)**: [context.md §14](../../../../srs/context.md#14-status-line-integration) describes a "persistent footer" with a configured hook command; Leo is an Obsidian plugin with a native `Plugin.addStatusBarItem()` API, not a CLI status line. Proposing the Obsidian status bar as the host and the hook-command machinery is dropped (not applicable to Obsidian). Alternative: embed the widget inside the chat-view header instead of the global status bar; flagging for verifier.
- **`savingsTokens` calculation for the near-capacity warning (§12.2 #1)**: the SRS table shows a "~12k" savings in the example but doesn't specify a formula for the near-capacity case. Proposing `savingsTokens = data.autoCompactThreshold ? (data.totalTokens - data.autoCompactThreshold) : 0` so the estimate equals the tokens currently above the autocompact cut point; verifier to confirm.
- **Per-tool attribution for large-tool-results (§12.2 #2)**: the check needs per-tool-name token totals, which live in `ContextData.messageBreakdown.toolCallsByType` and `toolResultsByType`. Proposing the suggestion engine reads from `messageBreakdown` only and the check is skipped (no emission) when `messageBreakdown` is absent; verifier to pin.
- **Debounce semantics (§14)**: SRS says "Debounced by 500ms" without specifying leading-vs-trailing edge or max-wait. Proposing trailing-edge with no max-wait so rapid sequences collapse into exactly one write; flagging for verifier to confirm vs. a leading-edge or hybrid approach.
- **Status-bar widget when `apiUsage` is null (§14)**: [context.md §14](../../../../srs/context.md#14-status-line-integration) says the record reads from "the last API response" — if the session has no assistant turns yet, there is no usage. Proposing the widget renders `—%` with a tooltip "No usage yet" and `buildStatusLineContext` returns `null` in that case; verifier to confirm vs. showing `0%` from `current_usage=0`.
- **Suggestion-block placement within F47's right column**: [context.md §10](../../../../srs/context.md#10-interactive-ui-layout) puts suggestions at the bottom of the right column after the detail sections. Proposing F48 owns only the suggestion React subtree and hands it to F47 as a slot child; exact DOM ordering is confirmed by the F47 ui.md companion.
