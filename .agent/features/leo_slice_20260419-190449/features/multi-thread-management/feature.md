#  F37 — Multi-thread management

## Purpose

Deliver thread CRUD (create / switch / rename / delete) on top of the single-thread persistence basis from [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md), lifting `.leo/conversations/` from one hard-coded `default.json` to one JSON file per thread each carrying its own conversation + per-thread `metadata` (allow-list, active `skillId`). A ThreadsStore enumerates the directory, a thread-list entry in the [F04 chat-sidebar-view](../chat-sidebar-view/feature.md) HeaderBar region lets the user switch between threads, the HeaderBar's current-thread title supports inline rename, a delete action emits a confirmation [Notice](../../../../standards/tech-stack.md#platform-apis) with Undo, and a new "Leo: New thread" [command palette entry](../../context.md#fr-ui-04) creates a fresh thread wired to the default skill. The on-disk layout is the exact forward-compatible JSON already declared by [FR-CHAT-08](../../context.md#fr-chat-08), so this slice extends that layout without a migration.

## Scope

### In scope

- `ThreadsStore` module enumerating `.leo/conversations/*.json` through the [Architecture §3.4 VaultAdapter](../../../../architecture/architecture.md#34-adapters) seam and exposing `list() / create() / switch(id) / rename(id, name) / delete(id)` on top of the [F14 `ConversationStore`](../conversation-persistence-v1/feature.md) per-thread `load / save / mutate` surface per [FR-CHAT-08](../../context.md#fr-chat-08); one JSON file per thread keeps the Phase-2 layout declared in [Architecture §9 Project File Layout](../../../../architecture/architecture.md#9-project-file-layout-proposed) byte-compatible.
- Thread-list sidebar entry mounted in [F04](../chat-sidebar-view/feature.md)'s `HeaderBar` region: dropdown / popover listing every thread sorted by `updatedAt` desc, each row showing the thread title and a last-activity hint; selecting a row invokes `ThreadsStore.switch(id)` which swaps the active-thread id on `ConversationStore` and rehydrates [F10 agent-controller-core](../agent-controller-core/feature.md) with the new thread's messages before the [F05 chat-message-list-markdown](../chat-message-list-markdown/feature.md) next render.
- HeaderBar current-thread title region: displays `thread.metadata.title` (default `"New thread"` on create); click or keyboard-focus + Enter switches it to an inline `<input>` that commits on Enter / blur via `ThreadsStore.rename(id, name)` (persists through [F14](../conversation-persistence-v1/feature.md) into `metadata.title`), cancels on Esc; empty / whitespace-only input falls back to the previous title.
- Delete action (accessible from the thread-list row overflow menu and a "Leo: Delete current thread" command palette entry) that invokes `ThreadsStore.delete(id)`: the on-disk file is moved to `.leo/conversations/.trash/<id>.json`, a confirmation [Notice](../../../../standards/tech-stack.md#platform-apis) is shown with an Undo action that restores the file within a [configurable window](../../../../standards/code-style.md#async--concurrency); after the window expires the trashed file is removed; deleting the last thread auto-creates a fresh one so `ConversationStore` always has an active thread.
- Command palette entry `"Leo: New thread"` per [FR-UI-04](../../context.md#fr-ui-04) via [`Plugin.addCommand`](../../../../standards/tech-stack.md#platform-apis): generates a new thread id (UUID v4 / ULID), writes an empty thread document with `schemaVersion: 1`, `metadata.title = "New thread"`, `metadata.skillId = "general"` (resolved through [F22 skills-picker-active-skill](../skills-picker-active-skill/feature.md) on next turn), `metadata.allowedTools = []`, switches the active thread to it, and focuses the [F06 chat-composer-input](../chat-composer-input/feature.md).
- Active-thread id persisted in plugin data (`data.json`) through Obsidian's `loadData` / `saveData` per [Architecture §6 State Ownership](../../../../architecture/architecture.md#6-state-ownership) so the same thread is restored on plugin reload; if the stored id no longer exists on disk, falls back to the most-recently-updated thread or auto-creates a `"New thread"`.
- Structured log events `thread.create / thread.switch / thread.rename / thread.delete / thread.delete.undo` via the [F01 Logger](../plugin-bootstrap-logging/feature.md) per [NFR-LOG-04](../../context.md#nfr-log-04) — ids + counts only, never thread title or message content above `debug`. Vitest coverage for list enumeration, create / rename / switch / delete round-trips, Undo restore, last-thread auto-create, missing-stored-id fallback, and per-thread metadata (allow-list, `skillId`) isolation across switches.

### Out of scope

- Single-thread persistence basis (`ConversationStore` `load / save / mutate`, atomic writes, debounce, schema) — ships with [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md); this feature consumes it read/write without reimplementing.
- Compaction snapshots, autocompact history, microcompaction PTL recovery writes into per-thread JSON — ships with [F42+](../../features-index.md); this feature touches no compaction fields.
- Per-thread active-skill selection UI (picker + `metadata.skillId` write path on switch) — ships with [F22 skills-picker-active-skill](../skills-picker-active-skill/feature.md); this feature only ensures `skillId` round-trips per thread on switch / create / delete.
- Visual states / loading spinners / success-highlight motion on thread switch — ships with [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md); this feature reuses F13's primitives without defining new ones.
- Thread export / import, cross-vault sync, archive folder browsing beyond the single `.trash/` Undo bucket — not in any SRS requirement; deferred.

## Acceptance criteria

1. `ThreadsStore.list()` enumerates every `.leo/conversations/*.json` (excluding `.trash/`) through the [F14](../conversation-persistence-v1/feature.md) `VaultAdapter` seam, returning `{ id, title, updatedAt, messageCount }` rows sorted by `updatedAt` desc; the HeaderBar thread-list entry renders exactly those rows with the title and last-activity hint visible. (FR-CHAT-08)
2. `"Leo: New thread"` command-palette entry creates a new thread with a fresh id, writes `.leo/conversations/<id>.json` with `schemaVersion: 1`, `metadata.title = "New thread"`, `metadata.skillId = "general"`, `metadata.allowedTools = []`, empty `messages`; switches the active thread to it; focuses the composer; emits `thread.create { id }` via the [F01 Logger](../plugin-bootstrap-logging/feature.md). (FR-CHAT-08, FR-UI-04)
3. Selecting a thread-list row invokes `ThreadsStore.switch(id)` which updates the active-thread id on `ConversationStore`, hydrates [F10 agent-controller-core](../agent-controller-core/feature.md) with the new thread's messages + `metadata` (allow-list, `skillId`) before the next `ChatView` render; the previously-active thread's in-memory state is flushed through [F14's](../conversation-persistence-v1/feature.md) debounced save before the swap. (feature-specific)
4. Clicking or keyboard-focusing the HeaderBar current-thread title switches it to an inline text input; Enter / blur commits via `ThreadsStore.rename(id, name)` which calls `ConversationStore.mutate(id, t => { t.metadata.title = name })` so the rename persists through [F14](../conversation-persistence-v1/feature.md); Esc cancels; empty / whitespace-only commits fall back to the previous title. (FR-CHAT-08)
5. `ThreadsStore.delete(id)` moves `.leo/conversations/<id>.json` to `.leo/conversations/.trash/<id>.json`, shows an Obsidian `Notice` with `Undo` that calls `ThreadsStore.restore(id)` to move the file back and re-switch to it; after the Undo window expires the trashed file is removed; deleting the only remaining thread triggers an automatic `ThreadsStore.create()` so `ConversationStore` always has exactly one active thread. (feature-specific)
6. The active-thread id is persisted through Obsidian's `loadData` / `saveData` and restored on `Plugin.onload`; if the stored id no longer exists on disk, the store falls back to the most-recently-updated thread or, if none remain, auto-creates a `"New thread"` — no uncaught error escapes `onload`. (feature-specific)
7. Per-thread `metadata.allowedTools` (from [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md)) and `metadata.skillId` (from [F22 skills-picker-active-skill](../skills-picker-active-skill/feature.md)) are isolated across `switch(id)` — a Vitest scenario that sets a per-thread allow-list + skill on thread A, switches to thread B, mutates B, switches back to A, and asserts A's `metadata` is byte-identical to its pre-switch snapshot. (FR-CHAT-08)
8. Structured log events `thread.create / thread.switch / thread.rename / thread.delete / thread.delete.undo` are emitted with `{ id, count? }` payloads only (no title, no message content above `debug`) per [NFR-LOG-04](../../context.md#nfr-log-04); Vitest asserts the event shapes. (feature-specific)

## Dependencies

- [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) — supplies the per-thread JSON layout, `schemaVersion: 1`, `metadata` block (`allowedTools`, `skillId`), debounced atomic writes, and the `ConversationStore.load / save / mutate` seam this feature extends from one active thread to N threads per [FR-CHAT-08](../../context.md#fr-chat-08).
- [F22 skills-picker-active-skill](../skills-picker-active-skill/feature.md) — owns per-thread `metadata.skillId` writes and the next-turn `systemPrompt` binding; this feature only ensures `skillId` round-trips through `create / switch / delete` without touching the picker's write path.
- Also consumes [F04 chat-sidebar-view](../chat-sidebar-view/feature.md) for the HeaderBar mount region + palette scaffolding, [F10 agent-controller-core](../agent-controller-core/feature.md) for the in-memory thread hydration seam, [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) for the `Notice` + Undo primitive, and [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) for the per-thread `allowedTools` round-trip invariant.
- Drives requirement [FR-CHAT-08](../../context.md#fr-chat-08) (multi-thread extension of single-thread persistence) and the `"Leo: New thread"` entry from [FR-UI-04](../../context.md#fr-ui-04); covers the Phase-5 "Multiple conversation threads" deliverable noted in [features-index.md](../../features-index.md).
- Downstream consumers tracked in [features-index.md](../../features-index.md): F42+ (compaction snapshots extending per-thread `metadata`), F50 (attachments stored per thread).

## Implementation notes

- [Architecture §3.2 Agent Layer — ConversationStore](../../../../architecture/architecture.md#32-agent-layer) — places the thread store alongside `ConversationStore`; this feature adds the N-thread surface on top.
- [Architecture §5.1 Plugin Startup](../../../../architecture/architecture.md#51-plugin-startup) — fixes the `load conversations (async)` step this feature extends to "load active-thread id + hydrate its JSON".
- [Architecture §6 State Ownership](../../../../architecture/architecture.md#6-state-ownership) — pins `Open threads` and `Active thread id` as store-owned; this feature realises the switch seam.
- [Architecture §9 Project File Layout](../../../../architecture/architecture.md#9-project-file-layout-proposed) — confirms `.leo/conversations/<id>.json` layout is already multi-thread-shaped.
- [Architecture §11 Mapping SRS FR → Modules](../../../../architecture/architecture.md#11-mapping-srs-fr--modules) — routes `FR-CHAT-*` through `AgentRunner` + persistence; this feature binds the multi-thread part.
- [Tech stack — Storage Layout](../../../../standards/tech-stack.md#storage-layout) — one JSON per thread; this feature consumes that shape as-is.
- [Tech stack — Platform APIs](../../../../standards/tech-stack.md#platform-apis) — `addCommand` for "Leo: New thread" / "Leo: Delete current thread"; `Notice` for Undo; `loadData` / `saveData` for active-thread id.
- [Code style — TypeScript](../../../../standards/code-style.md#typescript) — strict mode, no `any`, named exports on the `ThreadsStore` surface.
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — FS access via `VaultAdapter`; listeners registered with auto-cleanup.
- [Code style — Async & Concurrency](../../../../standards/code-style.md#async--concurrency) — shared `debounce` on Undo-window timer; no synchronous FS.
- [Code style — Error Handling](../../../../standards/code-style.md#error-handling) — trash-move failures surface a `Notice` and log; `Result`-typed returns.
- [Code style — Logging](../../../../standards/code-style.md#logging) — structured `thread.*` events; no title / content above `debug`.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — `VaultAdapter` faked at the seam for enumeration + trash + restore round-trips.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — Single Responsibility: `ThreadsStore` owns CRUD, `ConversationStore` owns per-thread IO, `AgentRunner` owns in-memory state.

## Open questions

- Undo window for `thread.delete` — SRS is silent; proposing 10s aligned with Obsidian's default `Notice` duration; verifier to confirm against [NFR-USE-07](../../context.md#nfr-use-07) motion-reduced behaviour for the Notice.
- Thread id scheme — proposing ULID for lexicographic `updatedAt`-adjacent ordering without exposing creation time in the filename; verifier to confirm no collision against [F14's](../conversation-persistence-v1/feature.md) `"default"` constant used before this feature ships (migrating the legacy `default.json` on first multi-thread load).
- Trash retention beyond the Undo window — proposing immediate hard delete after the window expires (no long-term trash folder); if users need a recovery workflow, it belongs to a later feature not currently sliced.
