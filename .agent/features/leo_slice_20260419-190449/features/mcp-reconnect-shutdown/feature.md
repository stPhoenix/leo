# F56 — MCP reconnect & clean shutdown

## Purpose

Wire the two resilience edges left open by [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md) onto the `ServerRuntime.disposer()` hook it exposed: (a) an exponential-backoff reconnect loop triggered by transport `close` / `error` events that retries up to 5 times before giving up and surfacing the failure per [FR-MCP-11](../../context.md#fr-mcp-11), (b) tool-error surfacing when an in-flight `tools/call` observes a mid-call server crash per [NFR-REL-05](../../context.md#nfr-rel-05), and (c) a deterministic `SIGTERM` → `SIGKILL`-after-2s shutdown sweep over every tracked stdio child in `MCPClient._procs` during `Plugin.onunload` per [FR-MCP-12](../../context.md#fr-mcp-12). No new user surface is introduced; the running agent turn continues past a crashed MCP call and the next turn (or the user's `[Retry]` from F55) sees either the reconnected server or a failed status badge.

## Scope

### In scope

- Reconnect scheduler attached to every `ServerRuntime` produced by [F51](../mcp-client-config-transports/feature.md): on transport `close` or `error` (not on `onunload`), move `status` to `reconnecting`, schedule the next attempt after `delayMs = min(baseMs * 2^attempt, capMs)` with `attempt ∈ [0..4]` (5 total attempts), defaults `baseMs = 500`, `capMs = 16000`, jitter ±20% to avoid thundering-herd when many servers fail at once.
- Give-up branch: after the 5th failure, `status` moves to `failed`, a single `mcp.reconnect.gaveUp` `warn` log fires via [F01](../plugin-bootstrap-logging/feature.md), and the server is left dormant until a manual `MCPClient.connect(serverId)` (invoked by F55's `[Retry]` button or the next `Plugin.onload`); no auto-escalation.
- In-flight tool-call crash surfacing: when a transport `close` / `error` fires while `MCPClient.callTool(serverId, toolName, args, ctx)` has an unresolved `tools/call` promise, the adapter layer rejects that promise into a typed `ToolResult{ok:false, error:"mcp server <serverId> crashed during tool call"}` rather than throwing, so [F10 agent-controller-core](../agent-controller-core/feature.md)'s turn loop records a tool error and the agent continues per [NFR-REL-05](../../context.md#nfr-rel-05).
- Successful-reconnect side-effects: on the first successful `initialize` after a crash, re-run F51's discovery pipeline (`tools/list`, `resources/list`, `prompts/list`), update the `ServerRuntime.tools / resources / prompts` snapshots, and re-register any newly-appearing `mcp.<serverId>.<toolName>` specs (tool-id stays stable across reconnects so F52's `thread.metadata.allowedTools` keeps re-matching without user action).
- Abort safety: the reconnect loop runs under a per-server `AbortController` owned by the `ServerRuntime`; `disposer()` aborts it, so `onunload` / `MCPClient.disconnect(serverId)` / F55 delete never leaves a pending reconnect timer.
- Shutdown sweep in `Plugin.onunload`: iterate `MCPClient._procs` (the `Map<serverId, ChildProcess>` seam introduced in [architecture §10](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules)), send `SIGTERM` to each, start a 2000 ms timer, call SDK `client.close()` to flush the transport, and on timer expiry (or `exit` event with non-zero code already observed) send `SIGKILL`; return a `Promise.all` so `onunload` waits at most ~2 s before resolving.
- SSE-transport shutdown parity: SSE entries have no OS child to kill but still run through the same `disposer()` path — abort the `EventSource` / `fetch` controller, cancel any in-flight reconnect, and resolve immediately with no `SIGTERM` branch.
- Structured log events (`mcp.disconnect.observed`, `mcp.reconnect.scheduled`, `mcp.reconnect.attempt`, `mcp.reconnect.ok`, `mcp.reconnect.fail`, `mcp.reconnect.gaveUp`, `mcp.toolcall.crashed`, `mcp.shutdown.sigterm`, `mcp.shutdown.sigkill`, `mcp.shutdown.clean`) via [F01](../plugin-bootstrap-logging/feature.md)'s `Logger` with `{serverId, transport, attempt?, delayMs?, durationMs?, exitCode?, signal?}` fields; never logs `env`, `args`, `headers`, or `tools/call` payloads above `debug`.
- Vitest coverage per [NFR-TEST-05](../../context.md#nfr-test-05): a fake-timer suite in `tests/integration/mcp-reconnect.test.ts` drives the F51 stdio fixture through crash → 5 backoff retries → give-up, crash-mid-call → `ToolResult{ok:false}`, and `onunload` → `SIGTERM` + 2 s → `SIGKILL` with a stubbed `child_process.kill` spy asserting the signal sequence; SSE reconnect path covered via `msw` dropping the SSE stream and re-serving on attempt 3; `Promise.all` completion bound inside 2 s asserted by a fake-timer check.

### Out of scope

- Reconnect UX beyond the status badge already painted by [F55 mcp-settings-ui](../mcp-settings-ui/feature.md) — no Notice toast, no in-chat banner on server crash; agent-level tool-error surfacing is the only user-visible hop per [NFR-REL-05](../../context.md#nfr-rel-05).
- Manual `[Retry]` button + `cancelReconnect(serverId)` seam — ships with [F55 mcp-settings-ui](../mcp-settings-ui/feature.md); this feature only exposes the `MCPClient.connect(serverId)` idempotent re-entry.
- Queuing pending `tools/call` invocations during a reconnecting window — this feature rejects with `ToolResult{ok:false, error:"mcp server not connected"}` when `callTool` is invoked on a non-connected `ServerRuntime`; queueing semantics are a future enhancement flagged under Open questions.
- Replaying a crashed `tools/call` once the server reconnects — idempotence is not guaranteed by MCP tools; the agent sees a tool error and decides whether to retry in a subsequent turn.
- The config loader, transport adapters, discovery pipeline, and initial `ToolRegistry.register` — already shipped by [F51](../mcp-client-config-transports/feature.md); this feature only binds onto its `ServerRuntime.disposer()` hook and reuses the discovery pipeline on reconnect.
- The `ToolRegistry` module, the serial-tool-loop, and the `AbortController` plumbing — already shipped by [F16](../tool-registry-builtin-read/feature.md); this feature only threads the reject path through the existing `ToolSpec.invoke` contract.
- `Plugin.onload` / `onunload` registration itself — owned by [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md); this feature plugs its shutdown sweep into the existing `onunload` chain.

## Acceptance criteria

1. When an already-connected stdio or SSE `ServerRuntime` observes a transport `close` or `error` event (outside `onunload`), `MCPClient` sets `status = "reconnecting"`, emits `mcp.disconnect.observed` at `warn` via [F01](../plugin-bootstrap-logging/feature.md), and schedules up to 5 reconnect attempts with delays `500, 1000, 2000, 4000, 8000` ms (±20% jitter, cap 16 000 ms); a fake-timer Vitest asserts the exact `attempt / delayMs` log sequence. ([FR-MCP-11](../../context.md#fr-mcp-11))
2. On the first successful `initialize` during the reconnect loop, `MCPClient` re-runs F51's discovery pipeline (`tools/list` / `resources/list` / `prompts/list`), refreshes `ServerRuntime.tools / resources / prompts`, re-registers any newly-appearing `mcp.<serverId>.<toolName>` specs into [F16](../tool-registry-builtin-read/feature.md)'s `ToolRegistry`, sets `status = "connected"`, and logs `mcp.reconnect.ok {serverId, transport, attempt, durationMs}`. ([FR-MCP-11](../../context.md#fr-mcp-11), [FR-MCP-05](../../context.md#fr-mcp-05))
3. After the 5th attempt fails, `MCPClient` sets `status = "failed"`, emits one `mcp.reconnect.gaveUp {serverId, transport, attempts: 5}` at `warn`, stops scheduling, and leaves the server dormant until a manual `MCPClient.connect(serverId)` call (e.g., F55 `[Retry]` or next `Plugin.onload`); no further timers exist. ([FR-MCP-11](../../context.md#fr-mcp-11))
4. When `MCPClient.callTool(serverId, toolName, args, ctx)` has a pending `tools/call` RPC and the owning transport fires `close` / `error`, the adapter resolves that call into `{ok:false, error: "mcp server <serverId> crashed during tool call"}` rather than throwing, emits `mcp.toolcall.crashed` at `warn`, and [F10](../agent-controller-core/feature.md)'s turn continues with the synthesised tool-error message. ([NFR-REL-05](../../context.md#nfr-rel-05))
5. `MCPClient.callTool` invoked while `ServerRuntime.status ∈ {"reconnecting", "failed", "disconnected"}` resolves to `{ok:false, error: "mcp server <serverId> not connected"}` without touching the transport; agent turn continues. ([NFR-REL-05](../../context.md#nfr-rel-05))
6. On `Plugin.onunload`, `MCPClient.disconnectAll()` iterates every `ServerRuntime` and calls `disposer()`; for stdio children tracked in `MCPClient._procs`, the adapter sends `SIGTERM`, starts a 2000 ms timer, and on expiry (or earlier clean `exit`) resolves; children still alive after 2000 ms receive `SIGKILL`. A `Promise.all` over all disposers resolves within ~2 s asserted by fake-timer Vitest. ([FR-MCP-12](../../context.md#fr-mcp-12))
7. SSE entries route through the same `disposer()` path, aborting their `EventSource` / `fetch` controller and any pending reconnect timer, and never invoke the `SIGTERM` / `SIGKILL` branch; Vitest asserts `child_process.kill` is never called on SSE servers. ([FR-MCP-12](../../context.md#fr-mcp-12))
8. Shutdown and reconnect log events (`mcp.disconnect.observed`, `mcp.reconnect.scheduled / attempt / ok / fail / gaveUp`, `mcp.toolcall.crashed`, `mcp.shutdown.sigterm / sigkill / clean`) emit via [F01](../plugin-bootstrap-logging/feature.md) with `{serverId, transport, attempt?, delayMs?, durationMs?, exitCode?, signal?}` fields; no `env`, `args`, `headers`, or `tools/call` payloads appear at `info` or above. ([FR-MCP-11](../../context.md#fr-mcp-11), [FR-MCP-12](../../context.md#fr-mcp-12), [NFR-REL-05](../../context.md#nfr-rel-05))
9. Vitest integration suite in `tests/integration/mcp-reconnect.test.ts` covers (i) crash → 5 backoff retries → give-up with a stubbed fixture that rejects every `initialize`, (ii) crash-mid-call → `ToolResult{ok:false}` → agent turn continues, (iii) successful reconnect on attempt 3 → discovery refresh → `ToolRegistry.register`, (iv) `onunload` → `SIGTERM` → 2 s → `SIGKILL` with `child_process.kill` spy asserting the signal sequence and the ~2 s `Promise.all` upper bound, (v) SSE-drop reconnect via `msw`. ([NFR-TEST-05](../../context.md#nfr-test-05))

## Dependencies

- [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md) — owns the `MCPClient` module, the `ServerRuntime = {id, transport, client, tools, resources, prompts, status, disposer}` shape, the `Map<serverId, ServerRuntime>`, the `_procs: Map<serverId, ChildProcess>` tracker, the config loader, the transport adapters, the discovery pipeline, and the initial `ToolRegistry.register` call. This feature binds into its `disposer()` hook and reuses its discovery pipeline on successful reconnect; no new `MCPClient` methods are added beyond a private scheduler and the `_procs`-driven shutdown sweep.
- [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md) — provides the `Plugin.onload` / `onunload` lifecycle hooks this feature's shutdown sweep plugs into and the `Logger` consumed by every `mcp.reconnect.*` / `mcp.shutdown.*` / `mcp.toolcall.crashed` structured event per [NFR-LOG-04](../../context.md#nfr-log-04).
- [F10 agent-controller-core](../agent-controller-core/feature.md) — owns the turn loop that interprets `ToolResult{ok:false, error}` as a tool-error message; this feature relies on that contract to let the agent continue past a mid-call MCP crash per [NFR-REL-05](../../context.md#nfr-rel-05).
- [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) — owns `ToolRegistry.register` / `unregister` that F51's discovery pipeline calls on every successful `initialize`; this feature reuses that path on reconnect without modification.
- Drives requirements [FR-MCP-11](../../context.md#fr-mcp-11), [FR-MCP-12](../../context.md#fr-mcp-12), [NFR-REL-05](../../context.md#nfr-rel-05). Transitively relies on [NFR-TEST-05](../../context.md#nfr-test-05) for test harness.
- Downstream consumers tracked in [features-index.md](../../features-index.md): [F55 mcp-settings-ui](../mcp-settings-ui/feature.md) consumes `ServerRuntime.status = "failed"` for its `[Retry]` button and may expose a `cancelReconnect(serverId)` seam in a follow-up; [F52 mcp-tool-confirmation](../mcp-tool-confirmation/feature.md) benefits from the stable `mcp.<serverId>.<toolName>` id across reconnects so `allowedTools` re-matches without user action; [F53 mcp-resources-picker](../mcp-resources-picker/feature.md) / [F54 mcp-prompts-in-skills](../mcp-prompts-in-skills/feature.md) pick up the refreshed resources / prompts snapshots after reconnect.

## Implementation notes

- [Architecture §1 Local-first + Fail-safe editor ops](../../../../architecture/architecture.md#1-architectural-principles) — MCP reconnect must never cascade into the editor-lock or agent-runner paths; failure stays at the adapter edge and surfaces as a typed `ToolResult`.
- [Architecture §3.2 Agent Layer — MCPClient "Reconnect with backoff"](../../../../architecture/architecture.md#32-agent-layer) — explicitly names the behaviour this feature realises; the row is otherwise shipped by F51.
- [Architecture §3.4 Adapters](../../../../architecture/architecture.md#34-adapters) — transport crash / reconnect / kill paths are isolated here so domain code never imports `child_process` or SDK types.
- [Architecture §4 Key Contracts — ToolSpec `invoke → ToolResult`](../../../../architecture/architecture.md#4-key-contracts) — the mid-call-crash reject path conforms to the `{ok:false, error}` shape; no exception escapes `invoke`.
- [Architecture §5.5 MCP Tool Call](../../../../architecture/architecture.md#55-mcp-tool-call) — canonical `tools/call` flow; the crash-mid-call branch adds a new `MC → AR` error edge that resolves the pending promise instead of raising.
- [Architecture §7 Error Handling Strategy](../../../../architecture/architecture.md#7-error-handling-strategy) — "tool error surfaces to agent, agent continues" is the contract NFR-REL-05 traces to.
- [Architecture §8 Extension Points — MCP server](../../../../architecture/architecture.md#8-extension-points) — F55's re-add / retry flow reuses the same `MCPClient.connect(serverId)` seam this feature exits to after `gaveUp`.
- [Architecture §9 Project File Layout — `src/mcp/`](../../../../architecture/architecture.md#9-project-file-layout-proposed) — the reconnect scheduler and shutdown sweep land inside the same `MCPClient.ts` / `transports.ts` files F51 pinned; no new file is added unless factoring helps tests.
- [Architecture §10 Concurrency & Lifecycle Rules — MCP stdio children](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — explicitly pins "`plugin.onunload` → `SIGTERM` each, 2s grace → `SIGKILL`" and the `MCPClient._procs` tracker AC6 traces to.
- [Tech stack — MCP client (phase 6)](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) — the `@modelcontextprotocol/sdk` `StdioClientTransport` / `SSEClientTransport` surfaces this feature reaches into for `close` / `error` events and `client.close()` during shutdown.
- [Tech stack — Platform APIs — `child_process`](../../../../standards/tech-stack.md#platform-apis) — the Electron-renderer `child_process.kill(signal)` surface the SIGTERM → SIGKILL sweep uses; SSE path stays out of this surface.
- [Tech stack — Open Decisions — stdio in Obsidian renderer](../../../../standards/tech-stack.md#open-decisions) — if the F51 resolution is "SSE-only in phase 6", this feature's stdio-specific branches fall behind the same feature flag while the reconnect loop still ships for SSE.
- [Code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer) — `ToolResult` is typed `{ok:true, data} | {ok:false, error}`; the crash-mid-call path must return this discriminated union, never throw, and never leak SDK error types.
- [Code style — Async & Concurrency](../../../../standards/code-style.md#async--concurrency) — backoff uses `setTimeout` bound to a per-server `AbortController`; `disposer()` aborts it so no timer outlives `onunload`; shutdown sweep uses `Promise.all` over per-server disposers with an internal 2 s `Promise.race`.
- [Code style — Error Handling](../../../../standards/code-style.md#error-handling) — adapter layer catches transport errors and surfaces typed `ToolResult` / status transitions; never silently swallow. Give-up emits exactly one `warn`.
- [Code style — Logging](../../../../standards/code-style.md#logging) — fixes the `event: "mcp.reconnect.*" / "mcp.shutdown.*"` structured field shape and the "no argument / env / header payload above `debug`" rule AC 8 traces to.
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — `Plugin.onunload` awaits `MCPClient.disconnectAll()`; no `setTimeout` or listener survives unload; `Plugin.register` is used to bind the disposer so Obsidian's lifecycle guarantees teardown.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — fake timers drive the backoff schedule; `child_process.kill` is spied via Vitest module mock; SSE drop / reinstate is covered via `msw`; crash-mid-call uses the same bundled F51 stdio fixture with a `process.kill(pid, 'SIGKILL')` injection point.
- [Best practices — Make It Observable / Fail Fast](../../../../standards/best-practices.md#core-principles) — justifies per-attempt structured logs, the single-shot `gaveUp` warn, and the typed `ToolResult` error surface over silent swallow.

## Open questions

- Give-up re-arm policy — SRS does not fix whether a later manual `MCPClient.connect(serverId)` resets the attempt counter back to 0. Proposing yes (every fresh entry into the reconnect loop resets `attempt = 0`) since that matches the user's mental model of `[Retry]` and keeps F55 trivial; verifier to confirm versus a `--no-auto` mode.
- Queueing in-flight `callTool` during reconnect — the spec is silent. Proposing fail-fast (`{ok:false, error:"not connected"}`) per AC 5 because MCP tools are not guaranteed idempotent; queueing semantics are deferred. Verifier to accept or promote a queue.
- Backoff base / cap constants — SRS pins "exponential backoff" + 5 attempts but no numbers. Proposing `baseMs = 500`, `capMs = 16 000`, `jitter ±20%` per the same shape as [FR-PROV-06 (provider retry)](../../context.md#fr-prov-06) for consistency across adapters; verifier to pick concrete values.
- `disconnect` vs `close` on reconnect-loop entry — MCP SDK surfaces both events. Proposing treat both as "transport dead" and enter the loop on either; dedupe via a `reconnecting`-state guard so the loop is never double-entered. Verifier to confirm.
- SSE reconnect and `Last-Event-ID` — SSE supports resume via `Last-Event-ID`. Proposing do not implement resume in phase 6 (re-open the stream cold and rely on discovery refresh) since MCP-over-SSE does not guarantee server-side event persistence; verifier to confirm versus a best-effort resume.
- `cancelReconnect(serverId)` seam for F55 — [F55](../mcp-settings-ui/feature.md)'s ASCII wireframes reference this seam as an Open question. Proposing expose it here as `MCPClient.cancelReconnect(serverId)` that aborts the per-server `AbortController` and sets `status = "disconnected"`; verifier to confirm the API shape before F55 lands.
- SIGKILL escalation on already-exited children — if `exit` fires before the 2 s timer, `SIGKILL` would be a no-op on most platforms but may log spuriously. Proposing clear the timer on `exit` and emit `mcp.shutdown.clean` instead of `mcp.shutdown.sigkill`; verifier to confirm.
