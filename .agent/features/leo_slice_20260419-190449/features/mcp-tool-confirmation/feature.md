# F52 — MCP tool confirmation default

## Purpose

Lock in the safety default that every MCP-exposed tool registered by [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md) under the `mcp.<serverId>.<toolName>` namespace starts life with `requiresConfirmation: true` so that the very first invocation per thread funnels through the existing [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) inline dialog per [FR-MCP-07](../../context.md#fr-mcp-07), with the "Allow for thread" branch persisting the namespaced tool id into `thread.metadata.allowedTools` via [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) exactly like built-in tools per [FR-AGENT-11](../../context.md#fr-agent-11). No new UI, no new controller, no new persistence surface — only the registration-time default flip and the pre-approval lookup seam so MCP tools inherit the same Allow once / Allow for thread / Deny flow users already know from [F17](../tool-confirmation-flow/feature.md).

## Scope

### In scope

- Registration-time defaulting: when [F51](../mcp-client-config-transports/feature.md)'s `MCPClient` wraps a discovered MCP tool into a `ToolSpec` and calls `ToolRegistry.register(spec)` per [FR-MCP-06](../../context.md#fr-mcp-06), the spec builder SHALL set `spec.requiresConfirmation = true` unconditionally, regardless of any JSON-Schema hint on the server side, per [FR-MCP-07](../../context.md#fr-mcp-07) and [architecture §4](../../../../architecture/architecture.md#4-key-contracts).
- Pre-approval lookup: the existing pre-invoke gate from [F17](../tool-confirmation-flow/feature.md) (skip-dialog when `thread.metadata.allowedTools` contains the tool id) SHALL apply unchanged to `mcp.*` ids — an MCP tool id added to the allowlist via a prior "Allow for thread" resolution bypasses the dialog on all later calls in the same thread per [FR-AGENT-11](../../context.md#fr-agent-11).
- No UI additions: this feature reuses [F17](../tool-confirmation-flow/feature.md)'s `InlineConfirmation` dialog, [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md)'s `awaiting-confirmation` palette, and the MCP generic icon + server-name label rule already pinned by [FR-UI-05](../../context.md#fr-ui-05); no `Modal`, no new region, no new palette.
- Persistence shape unchanged: Allow-for-thread writes the full `mcp.<serverId>.<toolName>` string into the same `thread.metadata.allowedTools: string[]` array already declared by [F14](../conversation-persistence-v1/feature.md) and written only by [F17](../tool-confirmation-flow/feature.md); this feature neither introduces a new metadata field nor a per-server sub-structure.
- Server-rename resilience: because the allowlist stores the fully namespaced id (not a server-local tool name), removing and re-adding the same `serverId` with the same `toolName` naturally re-matches the allowlist entry; a different `serverId` for the same tool name does NOT match and re-prompts per [FR-AGENT-11](../../context.md#fr-agent-11).
- Structured log events `mcp.tool.confirmation.default` (emitted once per MCP tool registration at `debug` via [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md)) and reuse of [F17](../tool-confirmation-flow/feature.md)'s existing `tool.confirmation.request / allow-once / allow-thread / deny` events with `{toolId, thread, decision}` — no new runtime event types.
- Vitest coverage per [NFR-TEST-01](../../context.md#nfr-test-01) and [NFR-TEST-05](../../context.md#nfr-test-05): assert every registered MCP `ToolSpec` has `requiresConfirmation === true`, assert the F17 dialog fires for a first MCP call, assert Allow-for-thread persists the `mcp.<server>.<tool>` id into `thread.metadata.allowedTools` and bypasses the dialog on the next call, assert a different `serverId` re-prompts.

### Out of scope

- Built-in tool confirmation defaults — owned by [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) per [FR-AGENT-10](../../context.md#fr-agent-10) default rule (true for write/destructive, false for read); this feature is MCP-scoped.
- The confirmation dialog itself, the state machine, focus-trap, read-vs-write palette, Esc = Deny, keyboard ordering — all ship with [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md).
- Thread-metadata persistence surface and `thread.metadata.allowedTools` schema — owned by [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md); this feature only reuses the existing `string[]`.
- Plan-mode write-tool gating (block write tools while `mode === 'plan'`) — owned by [F24 plan-mode-permissions](../plan-mode-permissions/feature.md) per [FR-PLAN-05](../../context.md#fr-plan-05); plan mode is a separate gate layered before this dialog.
- Per-server bulk "trust this server" UI — not in [FR-MCP-07](../../context.md#fr-mcp-07); every MCP tool is individually gated until its namespaced id is allow-listed.
- Export / import of allowlist across threads or vaults — out of scope per the SRS open question "[MCP tool per-thread pre-approval semantics](../../context.md#open-questions)"; this feature deliberately keeps allowlist entries thread-scoped and does not attempt to migrate them.
- `tools/list` change-notification re-registration (server-side tool lifecycle) — owned by [F56 mcp-reconnect-shutdown](../mcp-client-config-transports/feature.md)'s downstream slice; new registrations that arrive later still flip the default via this feature's registration-time rule.

## Acceptance criteria

1. Every `ToolSpec` produced by [F51](../mcp-client-config-transports/feature.md)'s MCP registration pipeline has `requiresConfirmation === true` at the moment `ToolRegistry.register(spec)` is called, regardless of the discovered tool's JSON Schema or server-supplied hints — asserted by a Vitest test that registers a fixture MCP server exposing multiple tools and inspects `ToolRegistry.lookup(id).requiresConfirmation` for every `mcp.*` id. ([FR-MCP-07](../../context.md#fr-mcp-07), [FR-MCP-06](../../context.md#fr-mcp-06))
2. On the first turn that invokes a newly registered `mcp.<serverId>.<toolName>` tool whose id is NOT in `thread.metadata.allowedTools`, [F17](../tool-confirmation-flow/feature.md)'s pre-invoke gate pauses the turn and [F10 agent-controller-core](../agent-controller-core/feature.md) emits `StreamEvent.tool_confirmation{call, resolve}` — the tool is NOT invoked until the user resolves. ([FR-MCP-07](../../context.md#fr-mcp-07), [FR-AGENT-10](../../context.md#fr-agent-10), [FR-CHAT-13](../../context.md#fr-chat-13))
3. Choosing `Allow for thread` on the dialog appends the full namespaced id `mcp.<serverId>.<toolName>` (deduplicated) to `thread.metadata.allowedTools` through [F14](../conversation-persistence-v1/feature.md)'s `ConversationStore.mutate`, the tool executes, and every subsequent call of the same namespaced id within the same thread bypasses the dialog and invokes directly; the allowlist survives plugin reloads per [F14](../conversation-persistence-v1/feature.md). ([FR-AGENT-11](../../context.md#fr-agent-11), [FR-MCP-07](../../context.md#fr-mcp-07))
4. Choosing `Allow once` executes the tool for this call only without mutating `thread.metadata.allowedTools`; the next call of the same MCP tool in the same thread re-prompts. ([FR-MCP-07](../../context.md#fr-mcp-07), [FR-AGENT-10](../../context.md#fr-agent-10))
5. Choosing `Deny` (or Esc) feeds a `ToolResult{ok:false, error:"user denied <toolId>"}` back to the graph via [F17](../tool-confirmation-flow/feature.md)'s existing Deny branch with no mutation to `thread.metadata.allowedTools`. ([FR-AGENT-10](../../context.md#fr-agent-10))
6. A fresh thread always re-prompts on the first call of any `mcp.*` tool (no cross-thread leakage); likewise, a different `serverId` for the same `toolName` (e.g., `mcp.a.read` versus `mcp.b.read`) re-prompts because the allowlist key is the full namespaced id — asserted by Vitest. ([FR-AGENT-11](../../context.md#fr-agent-11), [FR-MCP-06](../../context.md#fr-mcp-06))
7. The reused [F17](../tool-confirmation-flow/feature.md) dialog renders the MCP tool using the generic MCP icon + server-name label per [FR-UI-05](../../context.md#fr-ui-05), uses the `awaiting-confirmation` amber palette from [F13](../ui-visual-states-notifications/feature.md) per [FR-UI-06](../../context.md#fr-ui-06), never uses an Obsidian native `Modal` per [FR-UI-08](../../context.md#fr-ui-08), and a Vitest snapshot verifies the icon family + server-name label resolves from the `mcp.<serverId>.<toolName>` id. ([FR-UI-05](../../context.md#fr-ui-05), [FR-UI-06](../../context.md#fr-ui-06), [FR-UI-08](../../context.md#fr-ui-08))
8. Vitest coverage per [NFR-TEST-01](../../context.md#nfr-test-01) and [NFR-TEST-05](../../context.md#nfr-test-05) covers: registration-time default flip across multiple fixture MCP tools, first-call dialog fire, Allow-for-thread persistence + bypass, Allow-once non-persistence + re-prompt, Deny tool-error, cross-thread isolation, cross-server re-prompt, and a `mcp.tool.confirmation.default` debug log event emission at registration time with `{toolId, serverId}` fields via [F01](../plugin-bootstrap-logging/feature.md). ([NFR-TEST-01](../../context.md#nfr-test-01), [NFR-TEST-05](../../context.md#nfr-test-05))

## Dependencies

- [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md) — provides the MCP registration pipeline (`MCPClient` → `ToolSpec` builder → `ToolRegistry.register`) that this feature hooks at the spec-builder step to stamp `requiresConfirmation = true` per [FR-MCP-06](../../context.md#fr-mcp-06) / [FR-MCP-07](../../context.md#fr-mcp-07).
- [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) — provides the inline confirmation dialog, state machine, Allow once / Allow for thread / Deny actions, thread-allowlist bypass logic, Esc=Deny, and the structured `tool.confirmation.*` log events that this feature reuses byte-identically for MCP tools per [FR-CHAT-13](../../context.md#fr-chat-13) / [FR-AGENT-10](../../context.md#fr-agent-10).
- [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) — owns `thread.metadata.allowedTools: string[]` that the "Allow for thread" branch writes; this feature stores `mcp.*` ids in the same array without schema changes per [FR-AGENT-11](../../context.md#fr-agent-11) / [FR-CHAT-08](../../context.md#fr-chat-08).
- [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) — provides `ToolRegistry.lookup(id).requiresConfirmation` which the [F17](../tool-confirmation-flow/feature.md) pre-invoke gate reads; no changes needed here since MCP `ToolSpec`s flow through the same registry.
- [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) — provides the `awaiting-confirmation` amber palette + generic MCP icon that the reused dialog renders per [FR-UI-05](../../context.md#fr-ui-05) / [FR-UI-06](../../context.md#fr-ui-06).
- [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md) — provides the `Logger` for the `mcp.tool.confirmation.default` debug event fired once per MCP tool registration.
- Drives requirements [FR-MCP-07](../../context.md#fr-mcp-07), [FR-AGENT-11](../../context.md#fr-agent-11), [NFR-TEST-01](../../context.md#nfr-test-01), [NFR-TEST-05](../../context.md#nfr-test-05).
- Downstream consumers tracked in [features-index.md](../../features-index.md): F55 (settings UI may later display the per-thread allowlist; out of scope here), F56 (reconnect / re-registration re-applies this default for fresh `ToolSpec`s).

## Implementation notes

- [Architecture §1 Local-first + Registry pattern](../../../../architecture/architecture.md#1-architectural-principles) — MCP servers are opt-in network egress; the confirmation gate keeps every first call user-supervised.
- [Architecture §4 Key Contracts — ToolSpec `requiresConfirmation`](../../../../architecture/architecture.md#4-key-contracts) — this feature sets the field at registration; it does not extend the contract.
- [Architecture §5.3 Chat Turn (with tool call + confirmation)](../../../../architecture/architecture.md#53-chat-turn-with-tool-call--confirmation) — the canonical sequence `tool_call` → `lookup` → `tool_confirmation` → resume is reused unchanged for `mcp.*` ids.
- [Architecture §5.5 MCP Tool Call](../../../../architecture/architecture.md#55-mcp-tool-call) — confirmation branch is explicitly called out here; this feature wires the default and the lookup, the rest lives in F17 and F51.
- [Architecture §6 State Ownership](../../../../architecture/architecture.md#6-state-ownership) — per-thread allowlist stays owned by `ConversationStore` in thread JSON; MCP tool ids live in the same array.
- [Architecture §7 Error Handling Strategy](../../../../architecture/architecture.md#7-error-handling-strategy) — Deny path re-uses the shared `tool-error "user denied <toolId>"` shape.
- [Architecture §10 Concurrency & Lifecycle Rules](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — on `ChatView` close any pending MCP confirmation resolves with Deny via F17's teardown; no new lifecycle hook added here.
- [Tech stack — Agent / Tool / Skill / MCP Wiring](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) — LangGraph `interrupt()` plus per-thread allowlist persisted with the thread; this feature only adds the MCP default.
- [Code style — Zod & Tool Schemas](../../../../standards/code-style.md#zod--tool-schemas) — the MCP `ToolSpec`'s Zod schema is already compiled by F51; this feature does not change the schema surface.
- [Code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer) — Deny path remains a typed `{ok:false, error}` `ToolResult`; no exception path introduced.
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — confirmation routed inline, never via Obsidian `Modal`; rule inherited from F17.
- [Code style — Logging](../../../../standards/code-style.md#logging) — the `mcp.tool.confirmation.default` debug event follows the existing `mcp.*` structured field shape from F51 and never logs tool arguments.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — fixture MCP server from F51 is reused to drive the registration-time default assertion and the first-call dialog fire.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — single source of truth: default flip lives at registration, allowlist check lives in F17's gate, persistence lives in F14; no duplicate state.

## Open questions

- Pre-approval survival across `MCPClient` server re-registration within the same thread — if the user removes an MCP server and later re-adds it with the same `serverId` and the same `toolName`, the allowlist match is natural because the key is the full namespaced id; if the `serverId` changes (user-chosen), the allowlist entry becomes stale but harmless (it simply never matches). SRS [open question](../../context.md#open-questions) "MCP tool per-thread pre-approval semantics" flags export/import across threads; proposing we treat stale entries as inert and only prune on thread deletion, matching [FR-AGENT-11](../../context.md#fr-agent-11).
- `Allow for thread` UI copy when the tool is MCP — F17 renders the tool name; should the dialog label the MCP tool as `<serverName>: <toolName>` or `mcp.<serverId>.<toolName>`? Proposing the friendlier `<serverName>: <toolName>` surface per [FR-UI-05](../../context.md#fr-ui-05)'s "MCP tools get a generic MCP icon + server-name label" rule, with the fully namespaced id shown in the args-preview header. Verifier to confirm before F55's settings UI mirrors the same label.
- Behaviour if an MCP server re-registers the same tool mid-thread with a different JSON Schema — F51 re-issues `ToolRegistry.register` with the same id; the allowlist entry still matches because it is id-keyed, but the user's prior approval now applies to a potentially different argument surface. Proposing we invalidate the allowlist entry on schema-hash change and re-prompt; verifier to accept or defer to F56.
