# F20 — `edit_note` tool with live edit lock & accept/reject

## Purpose

Deliver the concrete active-note editing tool `edit_note(path, line_start, line_end, new_content)` that ties together [F18 edit-lock-transactions](../edit-lock-transactions/feature.md) (the `EditorBridge.withLock` / `EditorTransaction` / readonly-decoration / 3s highlight substrate) and [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) (the inline Allow once / Allow for thread / Deny prompt) so that when the target `path` resolves to the currently active editor leaf the edit flows through [`EditorBridge.withLock(range, fn)`](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) as a single grouped [`EditorTransaction`](../../../../standards/tech-stack.md#platform-apis), satisfying [FR-AGENT-05](../../context.md#fr-agent-05)'s "Active-note modifying tools SHALL apply via EditorBridge (under edit lock)" invariant, and that after the transaction commits the `ChatView` renders an inline diff affordance with Accept / Reject buttons whose Reject branch drives `Editor.undo()` so the edit reverts atomically in one hop per [FR-EDIT-09](../../context.md#fr-edit-09). When `path` does not match the active leaf the tool transparently falls back to the [`VaultAdapter`](../../../../architecture/architecture.md#34-adapters) read–splice–write path used by [F19 tools-write-vault](../tools-write-vault/feature.md) so the non-active-note branch of [FR-AGENT-05](../../context.md#fr-agent-05) still resolves as [FR-AGENT-06](../../context.md#fr-agent-06), and the 3s post-edit highlight from F18 runs automatically on the active-note path per [FR-EDIT-08](../../context.md#fr-edit-08). Linked here for traceability: [FR-AGENT-05](../../context.md#fr-agent-05) and [FR-EDIT-09](../../context.md#fr-edit-09).

## Scope

### In scope

- `ToolSpec` registration of `edit_note` on the singleton [`ToolRegistry`](../tool-registry-builtin-read/feature.md) at `Plugin.onload` with `source: "builtin"`, a one-line description, and a Zod `{path: string, line_start: number, line_end: number, new_content: string}` schema per [code style — Zod & Tool Schemas](../../../../standards/code-style.md#zod--tool-schemas) and [architecture §4](../../../../architecture/architecture.md#4-key-contracts).
- `requiresConfirmation: true` so every invocation routes through the [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) inline dialog — write-tool amber palette — unless the tool id is already present in `thread.metadata.allowedTools` per [FR-AGENT-10](../../context.md#fr-agent-10) / [FR-AGENT-11](../../context.md#fr-agent-11).
- Active-note routing: if `path` resolves to the `TFile` behind the currently active Markdown leaf (resolved through `app.workspace.getActiveViewOfType(MarkdownView)?.file`), the tool applies the edit through [`EditorBridge.withLock(range, fn)`](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — i.e. through [F18 edit-lock-transactions](../edit-lock-transactions/feature.md)'s single grouped [`EditorTransaction`](../../../../standards/tech-stack.md#platform-apis) wrapping `Editor.replaceRange()` per [FR-AGENT-05](../../context.md#fr-agent-05) and the [§5.3 sequence "AR→EB: acquire lock, apply edit"](../../../../architecture/architecture.md#53-chat-turn-with-tool-call--confirmation).
- Non-active-note fallback: when `path` does NOT match the active leaf, the tool falls back to `VaultAdapter.read` → splice `new_content` into `[line_start, line_end]` in memory → `VaultAdapter.modify(path, next)` (the [F19 tools-write-vault](../tools-write-vault/feature.md) surface) so the non-active-note branch of [FR-AGENT-05](../../context.md#fr-agent-05) still resolves via [FR-AGENT-06](../../context.md#fr-agent-06); same path-traversal guard and typed [`ToolResult`](../../../../standards/code-style.md#langgraph--agent-layer) shape as F19.
- Inline diff accept/reject UI rendered into the `InlineConfirmation` region scaffolded by [F04 chat-sidebar-view](../chat-sidebar-view/feature.md) — never an Obsidian native `Modal` per [FR-UI-08](../../context.md#fr-ui-08) — showing the `{path, line_start, line_end}` header, before/after snippet, and two buttons `[Accept]` `[Reject]`; Accept is a no-op (the edit has already landed under F18's highlight); Reject drives `Editor.undo()` once on the active editor so the grouped transaction reverts in a single step per [FR-EDIT-09](../../context.md#fr-edit-09)'s "Reject reverts atomically".
- Undo-reverts-atomically invariant: because [F18](../edit-lock-transactions/feature.md) already groups the mutation into one [`EditorTransaction`](../../../../standards/tech-stack.md#platform-apis) per [FR-EDIT-05](../../context.md#fr-edit-05), a single `Editor.undo()` (invoked by either the Reject button or the user's native Ctrl/Cmd-Z) restores pre-edit content in one hop per [FR-EDIT-09](../../context.md#fr-edit-09).
- Post-edit 3s highlight: the active-note path inherits F18's [`highlights.ts`](../../../../architecture/architecture.md#9-project-file-layout-proposed) decoration — modified region flashes for 3 seconds after commit — per [FR-EDIT-08](../../context.md#fr-edit-08); no additional timer is introduced by this feature.
- Lock-release-on-every-exit: the `withLock(range, fn)` wrapper's `try { apply } finally { release }` invariant from F18 covers accept / reject / cancel (via `AbortSignal`) / throw paths per [FR-EDIT-07](../../context.md#fr-edit-07) and [NFR-REL-04](../../context.md#nfr-rel-04); this feature adds a Reject branch that calls `Editor.undo()` AFTER the lock has already been released, never during.
- Structured log events `tool.invoke.start / tool.invoke.ok / tool.invoke.error` and `edit_note.accept / edit_note.reject` via the [F01 Logger](../plugin-bootstrap-logging/feature.md) carrying `{toolId, thread, path, routedVia: "editor"|"vault", durationMs}` per [code style — Logging](../../../../standards/code-style.md#logging) (never the content payload).
- Vitest unit coverage per [code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) and [NFR-TEST-01](../../context.md#nfr-test-01): registration + `requiresConfirmation:true`; active-note routing hits `EditorBridge.withLock`; non-active-note routing hits `VaultAdapter.modify`; Accept commits without further action; Reject calls `Editor.undo()` exactly once; `Editor.undo()` alone reverts the grouped transaction in one hop; traversal guard rejection; no exception escapes `invoke`.

### Out of scope

- The CM6 edit-lock itself (readonly decoration, `EditorTransaction` grouping, 3s highlight, blocked-keystroke `Notice`, `withLock` try/finally) — ships with [F18 edit-lock-transactions](../edit-lock-transactions/feature.md) per [FR-EDIT-05](../../context.md#fr-edit-05) / [FR-EDIT-06](../../context.md#fr-edit-06) / [FR-EDIT-07](../../context.md#fr-edit-07) / [FR-EDIT-08](../../context.md#fr-edit-08) / [NFR-REL-02](../../context.md#nfr-rel-02) / [NFR-REL-04](../../context.md#nfr-rel-04); F20 consumes the seam.
- The non-active-note write tools `create_note` and `append_to_note` (Vault API + path guard + typed `ToolResult`) — ship with [F19 tools-write-vault](../tools-write-vault/feature.md) per [FR-AGENT-04](../../context.md#fr-agent-04) / [FR-AGENT-06](../../context.md#fr-agent-06); F20 reuses F19's VaultAdapter surface for its non-active-note fallback only.
- The inline tool-confirmation dialog (tool name, pretty-printed args, Allow once / Allow for thread / Deny, focus-trap, Esc=Deny, thread allowlist persistence, read-vs-write amber palette) — ships with [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) per [FR-CHAT-13](../../context.md#fr-chat-13); F20 only declares `requiresConfirmation: true` and relies on F17 to gate invocation.
- Context compaction of tool-use / tool-result pairs produced by this tool — ships with [F42+ compaction features](../../features-index.md) per [FR-COMPACT-01](../../context.md#fr-compact-01).
- Plan-mode blocking of `edit_note` (the mode-gate that forbids write tools while `mode === 'plan'`) — ships with [F24 plan-mode-permission-gate](../../features-index.md) per [FR-PLAN-05](../../context.md#fr-plan-05) / [NFR-REL-07](../../context.md#nfr-rel-07).

## Acceptance criteria

1. At `Plugin.onload` `edit_note` is registered on the singleton [`ToolRegistry`](../tool-registry-builtin-read/feature.md) with `source:"builtin"`, `requiresConfirmation:true`, a one-line description, and a Zod `{path, line_start, line_end, new_content}` schema; a Vitest assertion verifies the id appears in `ToolRegistry.listFor(thread)` and that `lookup("edit_note")` returns the expected spec shape. ([FR-AGENT-05](../../context.md#fr-agent-05), [FR-AGENT-04](../../context.md#fr-agent-04))
2. When `path` resolves to the `TFile` behind the currently active Markdown leaf, `invoke` applies the edit through [`EditorBridge.withLock(range, fn)`](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — i.e. the [F18 edit-lock-transactions](../edit-lock-transactions/feature.md) seam — and a Vitest assertion counts zero calls to `VaultAdapter.modify` along this path; the mutation is applied via `Editor.replaceRange()` inside a single grouped [`EditorTransaction`](../../../../standards/tech-stack.md#platform-apis) per [FR-EDIT-05](../../context.md#fr-edit-05). ([FR-AGENT-05](../../context.md#fr-agent-05))
3. When `path` does NOT match the active leaf, `invoke` falls back to the [`VaultAdapter`](../../../../architecture/architecture.md#34-adapters) read–splice–write path (`read → splice(line_start, line_end, new_content) → modify`) — i.e. the [F19 tools-write-vault](../tools-write-vault/feature.md) surface — and a Vitest assertion counts zero calls to `EditorBridge.withLock` along this path; path-traversal guard (same policy as F19's `create_note`) rejects `..` / leading `/` / out-of-vault paths with `{ok:false, error:"unsafe path"}` before any I/O. ([FR-AGENT-05](../../context.md#fr-agent-05))
4. On the active-note path, immediately after the grouped transaction commits the modified region displays the [F18](../edit-lock-transactions/feature.md) 3s highlight decoration (auto-clears on timer, cancelled on view detach) per [FR-EDIT-08](../../context.md#fr-edit-08); no additional highlight timer is introduced by this feature. ([FR-AGENT-05](../../context.md#fr-agent-05))
5. The `ChatView` renders an inline diff accept/reject prompt into the `InlineConfirmation` region from [F04](../chat-sidebar-view/feature.md) showing `{path, line_start, line_end}` and `[Accept]` `[Reject]` buttons — never an Obsidian native `Modal` per [FR-UI-08](../../context.md#fr-ui-08). ([FR-EDIT-09](../../context.md#fr-edit-09))
6. Clicking Reject invokes `Editor.undo()` exactly once on the active editor and the buffer returns to pre-edit content in one hop (atomic revert), because the mutation was a single grouped [`EditorTransaction`](../../../../standards/tech-stack.md#platform-apis) per [F18](../edit-lock-transactions/feature.md) / [FR-EDIT-05](../../context.md#fr-edit-05); a Vitest assertion counts `Editor.undo` calls and asserts content equality with the pre-edit snapshot. ([FR-EDIT-09](../../context.md#fr-edit-09))
7. A user-initiated native undo (Ctrl/Cmd-Z) on the active editor after the edit lands produces the same atomic pre-edit state in one hop — the undo-reverts-atomically invariant of [FR-EDIT-09](../../context.md#fr-edit-09) is satisfied equally by the Reject button and by native undo because both route through the single grouped [`EditorTransaction`](../../../../standards/tech-stack.md#platform-apis). ([FR-EDIT-09](../../context.md#fr-edit-09))
8. On any exit path — Accept, Reject, `AbortController` cancel mid-apply, or a throw inside the transaction — the edit lock is released in `finally` per [F18](../edit-lock-transactions/feature.md)'s `withLock` invariant per [FR-EDIT-07](../../context.md#fr-edit-07) / [NFR-REL-04](../../context.md#nfr-rel-04); a Vitest assertion verifies acquire/release symmetry (one acquire ↔ one release) on all four paths and that the Reject branch only calls `Editor.undo()` AFTER the lock has been released, never while held. ([FR-AGENT-05](../../context.md#fr-agent-05), [FR-EDIT-09](../../context.md#fr-edit-09))
9. Vitest unit suite covers: registration + `requiresConfirmation:true`; active-note path routes through `EditorBridge.withLock`; non-active-note path routes through `VaultAdapter.modify`; traversal guard rejection; Accept is a no-op after commit; Reject calls `Editor.undo()` exactly once; native undo reverts atomically; no exception escapes `invoke` (all failures surface as typed `{ok:false, error}` per [code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer)); structured `edit_note.accept` / `edit_note.reject` log events fire via the [F01 Logger](../plugin-bootstrap-logging/feature.md) with `{toolId, thread, path, routedVia, durationMs}`. ([FR-AGENT-05](../../context.md#fr-agent-05), [FR-EDIT-09](../../context.md#fr-edit-09), [NFR-TEST-01](../../context.md#nfr-test-01))

## Open questions

- Accept/reject UI semantics for the non-active-note fallback branch — the inline diff UI assumes an open editor per the context.md [accept/reject gap note](../../context.md#open-questions). Proposing: on the fallback branch the inline prompt still shows `[Accept]` `[Reject]`, but Reject restores the pre-edit bytes via `VaultAdapter.modify(path, pre)` using a snapshot captured before the write; verifier to confirm the snapshot-based revert satisfies "reverts atomically" on the non-active-note branch.
- Line-range semantics when `line_end` exceeds file length or `line_start > line_end` — the Zod schema will accept any non-negative integers; clamping vs error is unspecified. Proposing: surface `{ok:false, error:"invalid range"}` without mutation; verifier to confirm.
