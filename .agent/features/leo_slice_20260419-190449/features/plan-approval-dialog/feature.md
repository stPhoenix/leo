# F25 — Plan approval dialog

## Purpose

Deliver the user-facing plan-approval flow that turns [F24 plan-mode-permissions](../plan-mode-permissions/feature.md)'s raw `ExitPlanMode` tool into a reviewable, editable, rejectable approval checkpoint. When `ExitPlanMode` is invoked, the agent pauses and the [F04 ChatView](../chat-sidebar-view/feature.md) renders an inline `InlineDialog` region (never a native Obsidian `Modal` per [FR-UI-08](../../context.md#fr-ui-08)) that presents `{plan}` as Obsidian-rendered markdown via `MarkdownRenderer.render`, offers `Approve` / `Edit` / `Reject` buttons, and — on `Edit` — swaps the rendered markdown for an editable textarea whose contents, on confirm, set `planWasEdited = true` and rewrite the plan file to disk through [F23 `PlanStore.writePlan`](../plan-files-todos-store/feature.md) before the flag is flipped back to `"normal"`, wiring the exact approve/edit/reject semantics from plan.md §5.6 per [FR-PLAN-07](../../context.md#fr-plan-07). The dialog is focus-trapped with `role="dialog" aria-modal="true"`, `Esc = Reject`, and renders the plan body through Obsidian's `MarkdownRenderer` so themes and embedded links work identically to chat bubbles per [FR-UI-09](../../context.md#fr-ui-09) and [NFR-USE-07](../../context.md#nfr-use-07); `Approve` returns the plan.md §5.8 Case 1 result ("Approved Plan" header, "(edited by user)" suffix iff `planWasEdited`), subagent context collapses to Case 2, and empty/missing plan falls back to Case 3. This slice closes the loop opened by [F24 plan-mode-permissions](../plan-mode-permissions/feature.md).

## Scope

### In scope

- `PlanApprovalDialog` React component mounted into [F04 ChatView](../chat-sidebar-view/feature.md)'s `InlineDialog` region (same slot as [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) uses for its inline confirmation, never an Obsidian `Modal` per [FR-UI-08](../../context.md#fr-ui-08)); takes `{plan: string, isSubagent: boolean, onResolve: (outcome) => void}` props and renders three states: view (rendered markdown + Approve / Edit / Reject), edit (editable `<textarea>` seeded with current plan + Confirm / Cancel), and resolving (buttons disabled while [F23](../plan-files-todos-store/feature.md) write settles).
- `ExitPlanMode` tool integration on [F24 `PlanModeController`](../plan-mode-permissions/feature.md) / [F16 ToolRegistry](../tool-registry-builtin-read/feature.md) — the tool's `call()` pauses the [F10 AgentRunner](../agent-controller-core/feature.md) turn loop via LangGraph `interrupt()` (same mechanism [F17](../tool-confirmation-flow/feature.md) uses) on non-subagent non-empty-plan path, surfaces a `StreamEvent.plan_approval` event carrying `{plan, isSubagent:false}` that [F04 ChatView](../chat-sidebar-view/feature.md) routes to the `PlanApprovalDialog`, waits for the dialog's resolution, and resumes the graph with the outcome per [architecture §5.3](../../../../architecture/architecture.md#53-chat-turn-with-tool-call--confirmation).
- Markdown rendering via `MarkdownRenderer.render(app, plan, containerEl, "", component)` per [FR-CHAT-06](../../context.md#fr-chat-06) and [FR-UI-09](../../context.md#fr-ui-09) — reuses the exact rendering path [F05 chat-message-list-markdown](../chat-message-list-markdown/feature.md) uses for assistant bubbles, so code blocks / links / embeds resolve identically; the owning `Component` is passed so `MarkdownRenderer` can clean up on unmount.
- Edit mode: clicking `Edit` swaps the rendered region for a pre-filled `<textarea>` seeded with the current plan string; `Confirm` commits the edited text into the pending outcome + sets `planWasEdited = true`; `Cancel` reverts to view state without side effects; while in edit state, `Esc` first cancels edit mode (falls back to view), then a second `Esc` rejects (same precedence ladder as [F06 composer / F07 streaming stop / F17 confirmation](../chat-composer-input/feature.md)).
- Disk sync on Edit outcome: the tool's `call()` (after dialog resolves with `planWasEdited = true`) writes the edited plan through [F23 `PlanStore.writePlan(plan)`](../plan-files-todos-store/feature.md) BEFORE [F24](../plan-mode-permissions/feature.md) flips `mode` back to `"normal"`; the same `PlanStore.writePlan` path already exists on F23's allowlist entry for `plan-file-write` referenced in [F24 AC 2](../plan-mode-permissions/feature.md) — this feature routes the edited text through it verbatim per [FR-PLAN-07](../../context.md#fr-plan-07) plan.md §5.7 step 2; `Approve` without edit skips the write (plan already on disk from earlier `plan-file-write` calls in plan mode).
- Focus trap via the same `useFocusTrap` hook [F17](../tool-confirmation-flow/feature.md) uses: on mount, focus moves to `Approve` (or the textarea in edit mode), Tab cycles `Approve` → `Edit` → `Reject` (view state) or `Confirm` → `Cancel` (edit state), Shift+Tab reverses, focus cannot escape the dialog; `role="dialog"`, `aria-modal="true"`, `aria-live="assertive"` on mount and a one-shot SR announcement `"Plan approval required"` per [FR-UI-09](../../context.md#fr-ui-09) and [NFR-USE-07](../../context.md#nfr-use-07) / [NFR-USE-08](../../context.md#nfr-use-08).
- `Esc` precedence: in view state `Esc` resolves with `Reject` (matching plan.md §5.6's reject-keeps-plan-mode semantics — [F24](../plan-mode-permissions/feature.md) stays at `mode === "plan"`); in edit state `Esc` first returns to view, a second `Esc` rejects; this binds before [F07 streaming-stop](../chat-streaming-stop/feature.md) / [F06 composer-blur](../chat-composer-input/feature.md) because the dialog is focus-trapped and captures `keydown` at its own listener per [FR-UI-09](../../context.md#fr-ui-09).
- Tool-result message variants per plan.md §5.8 synthesised in `ExitPlanMode.call()` and returned as a typed `ToolResult` back into the [F10 AgentRunner](../agent-controller-core/feature.md) graph: main-agent non-empty plan → `"## Approved Plan:\n<plan>"` (or `"## Approved Plan (edited by user):\n<plan>"` when `planWasEdited`); subagent context (`ctx.thread.agentId != null`) → Case 2 verbatim (`"User has approved the plan. There is nothing else needed from you now. Please respond with \"ok\""`) with NO dialog shown (short-circuit in `call()` — subagent context never renders the approval UI per plan.md §5.8 Case 2); empty / missing plan → Case 3 verbatim (`"User has approved exiting plan mode. You can now proceed."`) per [FR-PLAN-07](../../context.md#fr-plan-07).
- Reject path: `onResolve({outcome: "reject"})` synthesises a typed `PlanApprovalRejected` tool-error fed back into the graph per [architecture §7](../../../../architecture/architecture.md#7-error-handling-strategy); [F24 `PlanModeController`](../plan-mode-permissions/feature.md) does NOT flip the flag (`mode` stays `"plan"` per plan.md §5.6), no `PlanStore.writePlan` call occurs, no `plan.mode.exit` log event fires, and no mode-transition attachment is queued.
- Teardown on [F04 ChatView](../chat-sidebar-view/feature.md).`onClose`: any pending dialog forcibly resolves with `Reject` (same escape-hatch [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) uses for its dialog); the `MarkdownRenderer` `Component` is detached via `useEffect` cleanup; event listeners registered via `Plugin.registerDomEvent` auto-dispose per [architecture §10](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules).
- Structured log events through the [F01 Logger](../plugin-bootstrap-logging/feature.md) per [NFR-LOG-04](../../context.md#nfr-log-04): `plan.approval.request` (`{threadId, isSubagent, planLength}` — plan body never above `debug`), `plan.approval.approve` (`{threadId, planWasEdited:false}`), `plan.approval.edit` (`{threadId, planWasEdited:true}`), `plan.approval.reject` (`{threadId}`); plan content and edited-plan content never logged above `debug` per [code-style — Logging](../../../../standards/code-style.md#logging).
- Vitest coverage per [NFR-TEST-01](../../context.md#nfr-test-01): dialog mounts into the `InlineDialog` region (never as an Obsidian `Modal` — asserted by mock spying on `Modal.prototype.open`); Approve resolves with `planWasEdited:false` + Case 1 header; Edit → textarea → Confirm resolves with `planWasEdited:true` + Case 1 "(edited by user)" header + `PlanStore.writePlan` called with edited text BEFORE [F24](../plan-mode-permissions/feature.md) flag flip; Reject resolves with `PlanApprovalRejected` + [F24](../plan-mode-permissions/feature.md) flag stays `"plan"` + no `writePlan` call; subagent context short-circuits to Case 2 without dialog mount; empty plan short-circuits to Case 3 without dialog mount; focus trap cycles correctly and `Esc` rejects in view state while first `Esc` exits edit mode in edit state.

### Out of scope

- Plan mode enforcement (`mode: "normal" | "plan"` flag, permission-system gate, subagent prohibition, mode-transition attachments, stale-todo reminder) per [FR-PLAN-03](../../context.md#fr-plan-03) / [FR-PLAN-04](../../context.md#fr-plan-04) / [FR-PLAN-05](../../context.md#fr-plan-05) / [FR-PLAN-08](../../context.md#fr-plan-08) / [NFR-REL-07](../../context.md#nfr-rel-07) → shipped by [F24 plan-mode-permissions](../plan-mode-permissions/feature.md); this feature consumes F24's `ExitPlanMode` tool and only owns the dialog + disk-sync-on-edit behaviour.
- Plan file storage details (slug generation, path-traversal guard, `.leo/plans/<slug>.md` layout, `TodoWrite` / `TodoStore`) per [FR-PLAN-01](../../context.md#fr-plan-01) / [FR-PLAN-02](../../context.md#fr-plan-02) / [FR-PLAN-06](../../context.md#fr-plan-06) / [NFR-REL-08](../../context.md#nfr-rel-08) → shipped by [F23 plan-files-todos-store](../plan-files-todos-store/feature.md); this feature calls `PlanStore.writePlan(plan)` only.
- Plan / todo session resume (todo rehydration from transcript; plan-content recovery fallback chain `file_snapshot` → `ExitPlanMode` tool_use → user attachments) per [FR-PLAN-09](../../context.md#fr-plan-09) → ships with [F26 plan-session-resume](../../features-index.md); the dialog never reads back a persisted approval state.

## Acceptance criteria

1. When [F24](../plan-mode-permissions/feature.md)'s `ExitPlanMode` tool is invoked on the main agent (`ctx.thread.agentId == null`) with a non-empty `{plan}` input, [F10 AgentRunner](../agent-controller-core/feature.md) pauses via LangGraph `interrupt()` and emits a `StreamEvent.plan_approval` event carrying `{plan, isSubagent:false, threadId}`; [F04 ChatView](../chat-sidebar-view/feature.md) mounts `PlanApprovalDialog` into its `InlineDialog` region (never an Obsidian `Modal` — asserted in Vitest by spying on `Modal.prototype.open`) and emits `plan.approval.request` via the [F01 Logger](../plugin-bootstrap-logging/feature.md). ([FR-PLAN-07](../../context.md#fr-plan-07), [FR-UI-08](../../context.md#fr-ui-08), [FR-UI-09](../../context.md#fr-ui-09))
2. The dialog renders `{plan}` as Obsidian markdown via `MarkdownRenderer.render(app, plan, containerEl, "", component)` — reusing the same rendering path [F05 chat-message-list-markdown](../chat-message-list-markdown/feature.md) uses — and exposes `Approve` / `Edit` / `Reject` buttons in that Tab order; `Approve` and `Edit` both lead to `ExitPlanMode` succeeding, `Reject` keeps [F24](../plan-mode-permissions/feature.md) in `mode === "plan"` per plan.md §5.6. ([FR-PLAN-07](../../context.md#fr-plan-07), [FR-UI-09](../../context.md#fr-ui-09), [FR-CHAT-06](../../context.md#fr-chat-06))
3. Clicking `Edit` swaps the rendered markdown for a `<textarea>` pre-filled with the current plan; on `Confirm` the tool's `call()` sets `planWasEdited = true` and writes the edited text through [F23 `PlanStore.writePlan(plan)`](../plan-files-todos-store/feature.md) BEFORE [F24 `PlanModeController`](../plan-mode-permissions/feature.md) flips `mode` back to `"normal"`; `Cancel` returns to view state without side effects; `plan.approval.edit` fires with `{threadId, planWasEdited:true}`. ([FR-PLAN-07](../../context.md#fr-plan-07), [FR-UI-09](../../context.md#fr-ui-09))
4. The dialog has `role="dialog"`, `aria-modal="true"`, a one-shot `aria-live="assertive"` SR announcement on mount, and a focus trap via `useFocusTrap` that moves focus to `Approve` (or the textarea in edit mode), cycles `Approve` → `Edit` → `Reject` on Tab (and `Confirm` → `Cancel` in edit mode), and prevents focus from escaping; Shift+Tab reverses the cycle. ([FR-UI-09](../../context.md#fr-ui-09), [NFR-USE-07](../../context.md#nfr-use-07), [NFR-USE-08](../../context.md#nfr-use-08))
5. In view state, `Esc` resolves with `Reject` (synthesising a typed `PlanApprovalRejected` tool-error fed back into the graph per [architecture §7](../../../../architecture/architecture.md#7-error-handling-strategy), keeping [F24](../plan-mode-permissions/feature.md) at `mode === "plan"`, triggering no `PlanStore.writePlan` call, and emitting `plan.approval.reject`); in edit state, `Esc` first cancels edit mode (falls back to view), and a second `Esc` rejects; this binds before [F07 streaming-stop](../chat-streaming-stop/feature.md) and [F06 composer-blur](../chat-composer-input/feature.md) because the dialog is focus-trapped. ([FR-UI-09](../../context.md#fr-ui-09), [NFR-USE-06](../../context.md#nfr-use-06))
6. `ExitPlanMode.call()` synthesises the tool-result payload per plan.md §5.8: main-agent non-empty plan → `"## Approved Plan:\n<plan>"` (or `"## Approved Plan (edited by user):\n<plan>"` when `planWasEdited`); subagent context (`ctx.thread.agentId != null`) short-circuits WITHOUT mounting the dialog and returns Case 2 verbatim (`"User has approved the plan. There is nothing else needed from you now. Please respond with \"ok\""`); empty / missing plan short-circuits WITHOUT mounting the dialog and returns Case 3 verbatim (`"User has approved exiting plan mode. You can now proceed."`). ([FR-PLAN-07](../../context.md#fr-plan-07))
7. On [F04 ChatView](../chat-sidebar-view/feature.md).`onClose` any pending dialog forcibly resolves with `Reject` (same pattern [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) uses for its dialog); the `MarkdownRenderer` `Component` is detached via `useEffect` cleanup; event listeners registered via `Plugin.registerDomEvent` auto-dispose per [architecture §10](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules). ([FR-UI-09](../../context.md#fr-ui-09))
8. Vitest suite per [NFR-TEST-01](../../context.md#nfr-test-01) covers: dialog mounts into `InlineDialog` (never `Modal.prototype.open`); Approve → Case 1 header + no `writePlan` call; Edit → Confirm → Case 1 "(edited by user)" header + `PlanStore.writePlan` called with edited text BEFORE [F24](../plan-mode-permissions/feature.md) flag flip; Reject → `PlanApprovalRejected` + [F24](../plan-mode-permissions/feature.md) flag stays `"plan"` + no write; subagent context short-circuits to Case 2 without dialog mount; empty plan short-circuits to Case 3 without dialog mount; focus trap cycle; Esc-rejects-in-view-state / Esc-exits-edit-mode-first. ([FR-PLAN-07](../../context.md#fr-plan-07), [FR-UI-09](../../context.md#fr-ui-09))
