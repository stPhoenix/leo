# F46 — ContextAnalyzer data pipeline

## Purpose

Deliver the pure data-collection engine behind `/context`: take the raw in-memory conversation, apply the post-compact-boundary filter (`getMessagesAfterCompact`), run the [F42 microcompact pass](../compaction-microcompact/feature.md) in analysis mode so the counts match what the API actually sees, then drive `analyzeContextUsage()` which fans out seven counting operations through `Promise.all` and layers error-isolated skill counting on top — producing a `ContextData` object consumed downstream by [F47 grid/command](../../features-index.md) and [F48 suggestions/status line](../../features-index.md). This feature owns the pipeline shape and the parallel-fan-out contract only; every count, threshold, data-model field, category definition, grid, suggestion, markdown output, and status-line widget is downstream. Satisfies [FR-CTX-02](../../context.md#fr-ctx-02) and [NFR-PERF-08](../../context.md#nfr-perf-08), sourced verbatim-by-link from [context.md §3 Data Pipeline](../../../../srs/context.md#3-data-pipeline) and [context.md §6 Context Analysis — Data Collection](../../../../srs/context.md#6-context-analysis--data-collection).

## Scope

### In scope

- Pure `analyzeContextUsage(inputs)` orchestrator module whose inputs match [context.md §6.1](../../../../srs/context.md#61-inputs) exactly (`messages`, `model`, `getToolPermissionContext`, `tools`, `agentDefinitions`, optional `terminalWidth`, `toolUseContext`, `mainThreadAgentDefinition`, `originalMessages`) and whose output is a typed `ContextData` value that Phase 1–4 downstream features (grid, suggestions, markdown, status line) consume.
- Pipeline driver that composes the four-step transformation from [context.md §3](../../../../srs/context.md#3-data-pipeline) in the fixed order: `getMessagesAfterCompact(rawMessages)` → optional `projectView` collapse (flag-gated, off by default in v1) → [F42](../compaction-microcompact/feature.md) `microcompactMessages` in analysis mode → `analyzeContextUsage(postMicrocompactMessages, …)`; original untransformed messages flow through as `originalMessages` for the API-usage tier of token totals.
- Post-compact-boundary filter sourced from the `SystemCompactBoundaryMessage` emitted by [F43 autocompact](../compaction-autocompact/feature.md) and the `SystemMicrocompactBoundaryMessage` emitted by [F42](../compaction-microcompact/feature.md); filter keeps everything after the *last* boundary marker, so `/context` shows only what the model currently sees.
- Seven parallel counting operations wired through `Promise.all` exactly as listed in [context.md §6.2](../../../../srs/context.md#62-seven-parallel-counting-operations): `countSystemTokens`, `countMemoryFileTokens`, `countBuiltInToolTokens`, `countMcpToolTokens`, `countCustomAgentTokens`, `countSlashCommandTokens`, `approximateMessageTokens`. Each is a typed async callable supplied by adapters/domain modules; this feature owns the orchestration shape, not the per-op counting rules.
- Error-isolated skill counting: `countSkillTokens(...)` runs in a separate `try { await } catch` block *after* the `Promise.all` batch resolves; any throw inside `countSkillTokens` must produce a zero-token skill category plus a structured `context.skill_count_failed` log through [F01 Logger](../plugin-bootstrap-logging/feature.md) without rejecting the top-level `analyzeContextUsage` promise (matches [context.md §6.2](../../../../srs/context.md#62-seven-parallel-counting-operations) "Skill counting runs separately (error-isolated) after the parallel batch").
- Final-total selector from [context.md §6.7](../../../../srs/context.md#67-final-token-total): prefer API-usage tier (`input_tokens + cache_creation + cache_read` from the *last* API response in `originalMessages`); fall back to the sum of the seven-op estimates when no API usage is available. Selection is pure and the chosen branch is recorded on `ContextData.tokenTotalSource: 'api' | 'estimated'`.
- `AbortSignal` plumbing: the optional `signal?: AbortSignal` on the input object is threaded into every counting op so a `/context` command abort or Leaf-teardown aborts the in-flight run; aborted runs reject with `DOMException('aborted')` and never mutate caller state.
- Vitest coverage: (a) pipeline-step ordering (filter → microcompact → analyze); (b) boundary filter picks the *last* boundary when both a compact and a microcompact boundary are present; (c) `Promise.all` really runs concurrently (spy-based timing assertion; seven spies see overlapping windows); (d) a throwing `countSkillTokens` yields a zero-token skill entry plus a log event and the orchestrator still resolves; (e) `originalMessages` path feeds API-usage-tier selection; (f) abort mid-pipeline rejects and cancels downstream ops.

### Out of scope

- The per-op counting logic itself — `countSystemTokens` / `countBuiltInToolTokens` / `countMcpToolTokens` / `countCustomAgentTokens` / `countSlashCommandTokens` / `countSkillTokens` / `countMemoryFileTokens` / `approximateMessageTokens` bodies are implemented by downstream features (see Open questions); this feature owns their orchestration and the `Promise.all`-plus-error-isolated-skill pattern only.
- `ContextData` field semantics beyond "typed result of the pipeline"; shape detail, category ordering, deferred-category rules, grid sizing, partial-square fullness, suggestion thresholds, markdown rendering, and status-line wiring are all carried by [F47](../../features-index.md) and [F48](../../features-index.md).
- The 3-tier token estimator internals — delivered by [F41](../token-estimator-3tier/feature.md) and consumed by per-op counters.
- Microcompact clearing rules, boundary-marker emission, and compactable-tool allowlist — all live in [F42](../compaction-microcompact/feature.md).
- Autocompact threshold, summarization call, post-compact assembly, and compact boundary emission — all live in [F43](../compaction-autocompact/feature.md).
- `projectView()` context-collapse transform — the slot is reserved in the pipeline but the feature itself is deferred per [context.md §3](../../../../srs/context.md#3-data-pipeline) ("If feature-flagged"); v1 passes through.
- Session memory compaction (Layer 4) — explicitly out of scope per [context.md Out of scope](../../context.md#out-of-scope).

## Acceptance criteria

1. `analyzeContextUsage` accepts exactly the [context.md §6.1](../../../../srs/context.md#61-inputs) input fields and returns a typed `ContextData` object; a Vitest type-level test asserts the input/output surface matches the SRS table. (FR-CTX-02, [context.md §6.1](../../../../srs/context.md#61-inputs))
2. Pipeline order is `rawMessages → getMessagesAfterCompact → (optional) projectView → microcompactMessages → analyzeContextUsage` exactly as drawn in [context.md §3](../../../../srs/context.md#3-data-pipeline); a Vitest fixture replaces each step with a tagging spy and asserts the call order and that `analyzeContextUsage` sees the post-microcompact messages. (FR-CTX-02, [context.md §3](../../../../srs/context.md#3-data-pipeline))
3. The post-compact-boundary filter retains every message *after* the most-recent boundary marker (either [F43](../compaction-autocompact/feature.md)'s `SystemCompactBoundaryMessage` or [F42](../compaction-microcompact/feature.md)'s `SystemMicrocompactBoundaryMessage`, whichever is later); messages before the last boundary are dropped. Verified by a Vitest fixture mixing both boundary types. (FR-CTX-02, [context.md §3](../../../../srs/context.md#3-data-pipeline))
4. The seven counting operations listed in [context.md §6.2](../../../../srs/context.md#62-seven-parallel-counting-operations) run through a single `Promise.all`; a Vitest timing test wraps each counter with a resolver the test controls and asserts all seven see an overlapping active window (i.e. no sequential awaits among them). (NFR-PERF-08, FR-CTX-02, [context.md §6.2](../../../../srs/context.md#62-seven-parallel-counting-operations))
5. Skill counting (`countSkillTokens`) runs *after* the `Promise.all` batch in a separate `try/catch`; when it throws, `ContextData` still resolves with a zero-token skill entry and a `context.skill_count_failed` structured event is logged via [F01 Logger](../plugin-bootstrap-logging/feature.md). Verified by a Vitest fixture that injects a throw. (NFR-PERF-08, FR-CTX-02, [context.md §6.2](../../../../srs/context.md#62-seven-parallel-counting-operations))
6. When any of the seven parallel ops rejects, `analyzeContextUsage` rejects with the first error and does not leak unhandled rejections from the siblings; the `/context` command path surfaces the error and does not partially render. Verified by a Vitest `unhandledRejection` watcher. (FR-CTX-02, [context.md §6.2](../../../../srs/context.md#62-seven-parallel-counting-operations))
7. Final token total prefers the API-usage tier `input_tokens + cache_creation_input_tokens + cache_read_input_tokens` from the last API response in `originalMessages`; when no API usage is present, it falls back to the sum of the seven-op estimates. The chosen branch is recorded on `ContextData.tokenTotalSource`. Verified by two Vitest fixtures (one with `originalMessages` carrying `usage`, one without). (FR-CTX-02, [context.md §6.7](../../../../srs/context.md#67-final-token-total))
8. When `signal?.aborted` fires during any pipeline step, the returned promise rejects with `DOMException('aborted')` and no post-abort writes or logs happen; the seven parallel ops each observe the abort via the threaded signal. Verified by a Vitest fixture aborting between microcompact and `Promise.all`. (FR-CTX-02, [Architecture §10](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules))
9. The orchestrator is a pure function in the domain/core layer: no Obsidian `App`/`Plugin` imports, no direct `fetch`, no React; inputs are injected via the `inputs` argument and adapters. A Vitest import-graph assertion pins the module to the domain layer. (FR-CTX-02, [Architecture §3.3](../../../../architecture/architecture.md#33-domain--core-pure))

## Open questions

- **Per-op counter ownership**: the seven counters (`countSystemTokens`, `countMemoryFileTokens`, `countBuiltInToolTokens`, `countMcpToolTokens`, `countCustomAgentTokens`, `countSlashCommandTokens`, `approximateMessageTokens`) plus `countSkillTokens` are enumerated in [context.md §6](../../../../srs/context.md#6-context-analysis--data-collection) but no feature row owns their bodies. Proposing they ship inside F46's scope as small pure helpers driven by registries from F10/F16/F21/F22/F51; verifier to decide whether to split them into sub-features before F47 lands.
- **Boundary-filter choice when both markers coexist**: AC3 proposes "most-recent boundary wins" regardless of type. Is that correct when a microcompact boundary is *after* an autocompact boundary (common case mid-session)? Counter-proposal: always key off the autocompact boundary because microcompact is idempotent. Pick one and pin the rule in the fixture.
- **`projectView` pipeline slot in v1**: the slot is reserved per [context.md §3](../../../../srs/context.md#3-data-pipeline) but Leo has no feature flag yet. Ship as identity in v1 with a TODO, or elide the slot entirely and reintroduce when the flag lands? Proposing identity-pass-through to keep downstream wiring stable.
- **`originalMessages` source**: [context.md §6.1](../../../../srs/context.md#61-inputs) calls this "pre-transform messages (for API usage extraction)". In Leo, is that the raw conversation before *any* transform, or the conversation before the four-step pipeline specifically (i.e. after persistence load but before filtering)? Proposing the latter so `usage` extraction still reflects the last live turn; verifier to confirm.
- **Error semantics for `countSkillTokens` abort vs throw**: AC5 handles throws with a zero-token entry. Should an `AbortError` during skill counting be treated the same, or propagated to reject the top-level promise? Proposing propagate aborts, swallow non-abort throws; align with AC8.
- **Concurrency-failure semantics (AC6)**: current choice is "first rejection wins". Alternative is `Promise.allSettled` plus a partial `ContextData` with per-category error flags; that would let `/context` render partial results instead of failing. [context.md §6.2](../../../../srs/context.md#62-seven-parallel-counting-operations) says `Promise.all` explicitly, so sticking with it — flagging for verifier in case the UX tradeoff matters.
- **`ContextData` shape ownership**: this feature produces it but [F47](../../features-index.md) / [F48](../../features-index.md) consume every field. Should the interface live in a shared `domain/context/types.ts` owned by F46 with downstream imports, or in an F47-owned module with F46 importing it? Proposing F46-owned to match dependency direction.
