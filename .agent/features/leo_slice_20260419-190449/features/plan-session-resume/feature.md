# F26 ŌĆö Plan/todo session resume

## Purpose

Close the plan-mode loop by rehydrating ephemeral plan/todo state on plugin reload so a user who restarts mid-plan keeps their todos and plan file intact per [FR-PLAN-09](../../context.md#fr-plan-09). On `Plugin.onload`, after [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) hydrates the transcript, this slice scans the loaded thread backwards for the most recent `TodoWrite` tool_use and replays its `newTodos` into the [F23 plan-files-todos-store](../plan-files-todos-store/feature.md) `TodoStore`; in parallel it runs the plan-content recovery fallback chain from plan.md ┬¦8 ŌĆö `file_snapshot` system message ŌåÆ `ExitPlanMode` tool_use `input.plan` ŌåÆ user `planContent` / `plan_file_reference` attachment ŌĆö and writes the first non-empty hit back to `<vault>/<plansDirectory>/<slug>.md` through `PlanStore.writePlan`. Recovery is best-effort, silent on empty transcripts, and emits structured log events for every step so resume success/failure is diagnosable without leaking plan or todo content above `debug`.

## Scope

### In scope

- `PlanSessionResume` module (agent layer per [architecture ┬¦3.2](../../../../architecture/architecture.md#32-agent-layer)) owning `resume(thread)` invoked at `Plugin.onload` after [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) finishes hydration per [architecture ┬¦5.1 Plugin Startup](../../../../architecture/architecture.md#51-plugin-startup), before `ChatView` first renders, per [FR-PLAN-09](../../context.md#fr-plan-09).
- Todo rehydration: scan the loaded thread `messages` array backwards for the most recent assistant message containing a `tool_use` where `name === "TodoWrite"`, extract `input.newTodos`, validate through the shared [F23 plan-files-todos-store](../plan-files-todos-store/feature.md) `Todo` Zod schema, and replay into `TodoStore.write(key, newTodos)` keyed by `agentId ?? sessionId`; empty transcript or no `TodoWrite` hit is a no-op per [FR-PLAN-09](../../context.md#fr-plan-09).
- Plan-content recovery fallback chain per plan.md ┬¦8 (recorded in [context.md](../../context.md#fr-plan-09)) applied in strict order ŌĆö stop on the first non-empty hit, never merge sources ŌĆö and run after todo rehydration:
  1. `file_snapshot` system messages ŌĆö scan backwards for a `role: "system"` message whose payload carries a `file_snapshot` entry for the current plan file (`<plansDirectory>/<slug>.md`); take its `content`.
  2. `ExitPlanMode` tool_use ŌĆö scan backwards for an assistant `tool_use` where `name === "ExitPlanMode"`; take `input.plan`.
  3. User attachments ŌĆö scan backwards for a user message carrying `planContent` or a `plan_file_reference` attachment resolvable through the [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) persisted payload; take the resolved text.
- Write the first non-empty recovered plan body back to disk via [F23 plan-files-todos-store](../plan-files-todos-store/feature.md)'s `PlanStore.writePlan(content)` (which handles slug caching, path-traversal guard, and `plansDirectory` fallback) per [FR-PLAN-09](../../context.md#fr-plan-09), [NFR-REL-08](../../context.md#nfr-rel-08); byte-for-byte round-trip is asserted.
- Structured log events (`plan.resume.start`, `plan.resume.todos.rehydrated`, `plan.resume.todos.none`, `plan.resume.plan.snapshot-hit`, `plan.resume.plan.tooluse-hit`, `plan.resume.plan.attachment-hit`, `plan.resume.plan.none`, `plan.resume.plan.write`, `plan.resume.skipped`) through the [F01 Logger](../plugin-bootstrap-logging/feature.md) per [NFR-LOG-04](../../context.md#nfr-log-04); no plan or todo content logged above `debug` per [code-style ŌĆö Logging](../../../../standards/code-style.md#logging).
- Idempotency: `resume(thread)` is safe to call multiple times ŌĆö the second call with the same transcript re-writes the same todos and same plan body (both are pure functions of the transcript) and emits `plan.resume.skipped` when the recovered body equals the current `PlanStore.readPlan()` so no redundant write fires.
- Vitest coverage for: latest-wins todo scan order; chain-stop-at-first-non-empty (snapshot, tool_use, attachment branches isolated + mixed); empty transcript no-op; non-empty transcript with no `TodoWrite` and no recovery source no-op; validate-and-reject-invalid `newTodos` path (Zod fail ŌåÆ drop silently with `plan.resume.todos.rehydrated` log recording count=0); re-entry idempotency; write-through to `PlanStore` asserted.

### Out of scope

- Plan approval dialog (Approve / Edit / Reject flow, result-message variants, focus-trap, `planWasEdited` disk sync) per [FR-PLAN-07](../../context.md#fr-plan-07), [FR-UI-09](../../context.md#fr-ui-09) ŌåÆ ships with [F25 plan-approval-dialog](../plan-approval-dialog/feature.md).
- Plan-mode permission gate, `mode` flag, `EnterPlanMode` / `ExitPlanMode` tools, mode-transition attachments, subagent prohibition per [FR-PLAN-03](../../context.md#fr-plan-03), [FR-PLAN-04](../../context.md#fr-plan-04), [FR-PLAN-05](../../context.md#fr-plan-05), [FR-PLAN-08](../../context.md#fr-plan-08), [NFR-REL-07](../../context.md#nfr-rel-07) ŌåÆ ships with [F24 plan-mode-permissions](../plan-mode-permissions/feature.md).
- The `TodoWrite` tool itself (schema, verbatim prompt, registry binding, `PlanStore` / `TodoStore` modules) per [FR-PLAN-01](../../context.md#fr-plan-01), [FR-PLAN-02](../../context.md#fr-plan-02), [FR-PLAN-06](../../context.md#fr-plan-06), [NFR-REL-08](../../context.md#nfr-rel-08), [NFR-TEST-07](../../context.md#nfr-test-07) ŌåÆ shipped by [F23 plan-files-todos-store](../plan-files-todos-store/feature.md); this feature only consumes that surface.
- Compaction boundary handling ŌĆö if a compaction run strips the `TodoWrite` or `ExitPlanMode` tool_use from the transcript before this feature runs, recovery silently finds nothing. Compaction policies (`FR-COMPACT-*`) are outside this slice.
- Multi-thread resume ŌĆö Phase 2 single active thread only; multi-thread resume is tracked with [F37 multi-thread-management](../../features-index.md).
- Subagent / sessionId partitioning ŌĆö treated as vacuous until subagent execution lands; this slice keys `TodoStore` the same way [F23 plan-files-todos-store](../plan-files-todos-store/feature.md) does (`agentId ?? sessionId`).

## Acceptance criteria

1. On `Plugin.onload`, after [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) completes `load()` and before `ChatView` first renders, `PlanSessionResume.resume(thread)` runs and emits `plan.resume.start` via the [F01 Logger](../plugin-bootstrap-logging/feature.md); an empty transcript is a no-op with `plan.resume.skipped`. ([FR-PLAN-09](../../context.md#fr-plan-09))
2. Todo rehydration scans `thread.messages` from last to first for an assistant `tool_use` where `name === "TodoWrite"`, parses `input.newTodos` through the shared [F23 plan-files-todos-store](../plan-files-todos-store/feature.md) `Todo` Zod schema, and replays into `TodoStore.write(agentId ?? sessionId, newTodos)`; no `TodoWrite` or Zod failure is a silent no-op logged as `plan.resume.todos.none`. ([FR-PLAN-09](../../context.md#fr-plan-09))
3. Plan-content recovery tries the three sources in strict order ŌĆö (1) `file_snapshot` system messages, (2) `ExitPlanMode` tool_use `input.plan`, (3) user `planContent` / `plan_file_reference` attachments ŌĆö and stops at the first non-empty hit; each hit/miss emits its branch-specific log event (`plan.resume.plan.snapshot-hit` / `plan.resume.plan.tooluse-hit` / `plan.resume.plan.attachment-hit` / `plan.resume.plan.none`). ([FR-PLAN-09](../../context.md#fr-plan-09))
4. When the chain yields a non-empty body, `PlanSessionResume` writes it to disk via `PlanStore.writePlan(content)` so the file at `<vault>/<plansDirectory>/<slug>.md` round-trips byte-for-byte; `plan.resume.plan.write` records the source tier. ([FR-PLAN-09](../../context.md#fr-plan-09), [NFR-REL-08](../../context.md#nfr-rel-08))
5. Recovery never merges tiers and never mutates the transcript ŌĆö sources lower in the chain are ignored once a non-empty hit is taken; the in-memory `ConversationStore` thread object is read-only during resume. ([FR-PLAN-09](../../context.md#fr-plan-09))
6. `resume(thread)` is idempotent: a second call with the same transcript replays the same todos (state converges) and, if the recovered body equals `PlanStore.readPlan()`, emits `plan.resume.skipped` and performs no redundant write. ([FR-PLAN-09](../../context.md#fr-plan-09))
7. Vitest suite covers: latest-wins todo scan on multi-`TodoWrite` transcripts; each recovery tier in isolation; strict stop-at-first-non-empty on mixed transcripts; empty transcript / no-TodoWrite / no-recovery no-ops; Zod-invalid `newTodos` silently rejected; write-through to a faked `PlanStore` asserted; re-entry idempotency with `plan.resume.skipped`; no log above `debug` carries plan or todo content. ([FR-PLAN-09](../../context.md#fr-plan-09), [NFR-TEST-07](../../context.md#nfr-test-07))

## Open questions

- **`file_snapshot` payload shape** ŌĆö plan.md ┬¦8 names `file_snapshot` system messages but Leo does not yet emit them anywhere upstream (no FR attaches a producer). This feature assumes a `{ kind: "file_snapshot", path: string, content: string }` payload persisted verbatim through [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md)'s unknown-field passthrough; verifier to confirm the intended producer (likely a later compaction slice) and spelling.
- **`plan_file_reference` attachment resolution** ŌĆö plan.md ┬¦8 lists `plan_file_reference` as a user-attachment route but the SRS does not pin whether the attachment carries the body inline or a vault path to resolve. This feature reads inline `planContent` first and, failing that, resolves `plan_file_reference.path` through the `VaultAdapter`; verifier to confirm.
- **Scope of "scan transcript backwards"** ŌĆö resume walks the entire hydrated transcript; if autocompaction has already collapsed older messages into a summary block (F43+), tool_use payloads are gone and recovery silently fails. Acceptable per best-effort framing; verifier to confirm no additional persistence hook is required.
