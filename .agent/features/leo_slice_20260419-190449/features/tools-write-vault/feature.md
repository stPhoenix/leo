# F19 — Vault-API write tools (`create_note`, `append_to_note`)

## Purpose

Deliver the two concrete Vault-API-backed write tools — `create_note(path, content)` and `append_to_note(path, content)` — that ride on the [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) `ToolRegistry` seam and the [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) inline confirmation prompt so the agent can add new notes and append to existing non-active notes without ever touching the active editor. The underlying requirement IDs [FR-AGENT-04](../../context.md#fr-agent-04) (built-in write tools belong in the `ToolRegistry`) and [FR-AGENT-06](../../context.md#fr-agent-06) (non-active-note tools SHALL apply via the Vault API) were declared as `covers` on [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md)'s registry slice; F19 is still an independent vertical slice that delivers the concrete `create_note` / `append_to_note` implementations against the [Obsidian `Vault` API](../../../../standards/tech-stack.md#platform-apis--interop) (`app.vault.create` + `app.vault.append`), each registered with `requiresConfirmation: true`, a Zod-parsed argument schema, a path-traversal guard, and a typed [`ToolResult`](../../../../standards/code-style.md#langgraph--agent-layer) returning `{ok:true, data}` on success or `{ok:false, error}` on failure. Linked here for traceability: [FR-AGENT-04](../../context.md#fr-agent-04) and [FR-AGENT-06](../../context.md#fr-agent-06).

## Scope

### In scope

- Two `ToolSpec` registrations on the singleton `ToolRegistry` from [F16](../tool-registry-builtin-read/feature.md) at `Plugin.onload`: `create_note` (Zod `{path: string, content: string}`) and `append_to_note` (Zod `{path: string, content: string}`), each with a one-line human/LLM-readable `description` per [code style — Zod & Tool Schemas](../../../../standards/code-style.md#zod--tool-schemas) and `source: "builtin"` per [architecture §4](../../../../architecture/architecture.md#4-key-contracts).
- Both tools declare `requiresConfirmation: true` per [FR-AGENT-04](../../context.md#fr-agent-04)'s default for write / destructive tools, so every invocation routes through the [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) inline dialog (Allow once / Allow for thread / Deny) unless the tool id is already in `thread.metadata.allowedTools`.
- Vault-API calls via the [`VaultAdapter`](../../../../architecture/architecture.md#34-adapters) surface (never `app.vault.adapter` directly per [code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns)): `create_note` calls `app.vault.create(path, content)` per the [Obsidian `Vault` API](../../../../standards/tech-stack.md#platform-apis--interop) and [FR-AGENT-06](../../context.md#fr-agent-06); `append_to_note` calls `app.vault.append` (Obsidian's `process`-backed append) on the `TFile` resolved from `path`.
- Path-traversal guard applied before any Vault call: reject any `path` containing `..`, starting with `/`, or resolving outside the vault root (same policy as [F16](../tool-registry-builtin-read/feature.md)'s `read_note` guard per [NFR-REL-08](../../context.md#nfr-rel-08)'s plans-directory guard) with `{ok:false, error:"unsafe path"}` before touching the vault.
- Atomic apply: each invocation performs a single Vault API call — no multi-step mutation is attempted, so either the write succeeds (file created / bytes appended) or the vault is unchanged; the Vault API's own write semantics supply the atomicity — this feature does not layer a second transaction on top per [code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer).
- Typed [`ToolResult`](../../../../standards/code-style.md#langgraph--agent-layer): `create_note` returns `{ok:true, data:{path, bytesWritten}}` on success, `{ok:false, error:"file exists"}` when `path` already exists, `{ok:false, error:"unsafe path"}` on guard rejection; `append_to_note` returns `{ok:true, data:{path, bytesAppended}}` on success, `{ok:false, error:"not found"}` when the target `TFile` does not exist, `{ok:false, error:"unsafe path"}` on guard rejection. No exception escapes `invoke` per [code style — Error Handling](../../../../standards/code-style.md#error-handling).
- Structured log events `tool.invoke.start / tool.invoke.ok / tool.invoke.error` via the [F01 Logger](../plugin-bootstrap-logging/feature.md) carrying `{toolId, thread, durationMs, bytes}` (not the content payload) per [code style — Logging](../../../../standards/code-style.md#logging).
- Vitest unit coverage per [code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) and [NFR-TEST-01](../../context.md#nfr-test-01): both tools registered at load; both carry `requiresConfirmation: true`; `create_note` happy / already-exists / traversal-blocked paths; `append_to_note` happy / not-found / traversal-blocked paths; structured `{ok:false, error}` surface on every failure mode (no thrown errors).

### Out of scope

- Active-note editing (`edit_note(path, line_start, line_end, new_content)` under CM6 edit lock with accept/reject inline diff) — ships with [F20 tool-edit-note-with-lock](../../features-index.md).
- CM6 readonly-range edit lock, grouped `EditorTransaction`, single-undo semantics — ship with [F18 edit-lock-transactions](../../features-index.md).
- `search_vault(query, tags?)` — ships with [F33 search-vault-tool](../../features-index.md).
- The inline confirmation dialog UI itself (tool name, pretty-printed args, three buttons, amber write-tool palette, focus-trap, Esc=Deny, thread allowlist persistence) — ships with [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md); F19 only declares `requiresConfirmation: true` and relies on F17 to gate invocation.
- Plan-mode write-tool gating — ships with [F24 plan-mode-permission-gate](../../features-index.md).

## Acceptance criteria

1. At `Plugin.onload` both `create_note` and `append_to_note` are registered on the singleton [`ToolRegistry`](../tool-registry-builtin-read/feature.md) with `source: "builtin"`, a one-line `description`, and a Zod `{path: string, content: string}` schema; a Vitest assertion verifies both ids appear in `ToolRegistry.listFor(thread)` and that `lookup(id)` returns a spec with the expected shape. (FR-AGENT-04)
2. Both `ToolSpec` entries expose `requiresConfirmation: true`, so the [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) pre-invoke gate suspends the turn and emits `StreamEvent.tool_confirmation` on every invocation whose tool id is NOT present in `thread.metadata.allowedTools`; a Vitest assertion verifies no Vault call is made until the confirmation resolves `allow-once` or `allow-thread`. (FR-AGENT-04)
3. `create_note(path, content)` invokes `app.vault.create(path, content)` through the [`VaultAdapter`](../../../../architecture/architecture.md#34-adapters) — i.e. the non-active-note Vault API surface — and returns `{ok:true, data:{path, bytesWritten}}` on success; `append_to_note(path, content)` resolves the target `TFile` and calls `app.vault.append(file, content)` through the same adapter, returning `{ok:true, data:{path, bytesAppended}}`. (FR-AGENT-06)
4. The path-traversal guard rejects any `path` containing `..`, starting with `/`, or resolving outside the vault root before any Vault call, returning `{ok:false, error:"unsafe path"}`; no partial mutation is ever observed on the vault for a rejected path. (FR-AGENT-04, FR-AGENT-06)
5. On Vault failure — `create_note` on an existing path, `append_to_note` on a missing file, or any platform error thrown by `app.vault.*` — `invoke` catches the error and returns a typed `{ok:false, error:<stable-string>}` (e.g. `"file exists"`, `"not found"`); no exception escapes the tool, and the structured `tool.invoke.error` log event fires with `{toolId, thread, durationMs}` via the [F01 Logger](../plugin-bootstrap-logging/feature.md). (FR-AGENT-04, feature-specific)
6. Each successful invocation performs exactly one Vault API call (`app.vault.create` OR `app.vault.append`) — no multi-step mutation is layered on top — so the vault is either untouched or transitioned to the post-write state; a Vitest assertion counts the mocked `VaultAdapter` calls per invocation and asserts exactly one write call per `{ok:true}` result. (feature-specific)
7. Vitest unit suite covers, against a faked in-memory `VaultAdapter`: registration of both tools, `requiresConfirmation: true` on both, `create_note` happy path, `create_note` on already-existing path → `{ok:false, error:"file exists"}`, `append_to_note` happy path, `append_to_note` on missing file → `{ok:false, error:"not found"}`, traversal-guard rejection for both tools, and that no `invoke` call throws. (FR-AGENT-04, FR-AGENT-06, feature-specific)

## Dependencies

- [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) — provides the inline confirmation dialog + `StreamEvent.tool_confirmation` pre-invoke gate + thread-scoped `metadata.allowedTools` allowlist that every `requiresConfirmation: true` tool, including `create_note` and `append_to_note`, funnels through before any Vault write happens.
- [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) — provides the `ToolRegistry` + declarative `ToolSpec` shape (`{id, description, schema (Zod), requiresConfirmation, invoke, source}`), the OpenAI-compatible `tools` parameter wiring, and the serial `tool_call` → invoke → `tool_result` loop that this feature registers its two specs into; the registry + schema basis comes from F16 and this feature adds concrete write-tool entries on top.
- Drives concrete implementations of requirement IDs [FR-AGENT-04](../../context.md#fr-agent-04) and [FR-AGENT-06](../../context.md#fr-agent-06) (both declared as `covers` on F16; this slice delivers the `create_note` / `append_to_note` variants specifically).
- Downstream consumers tracked in [features-index.md](../../features-index.md): F24 (plan-mode gate layered before invocation), F22 (skill `allowedTools` filter hiding write tools per skill).

## Implementation notes

- [Architecture §3.4 Adapters — VaultAdapter](../../../../architecture/architecture.md#34-adapters) — `create_note` + `append_to_note` MUST go through `VaultAdapter`, never `app.vault.adapter` directly.
- [Architecture §4 Key Contracts — ToolSpec / ToolResult](../../../../architecture/architecture.md#4-key-contracts) — pins the `{ok, data}|{ok, error}` return shape both tools honour.
- [Architecture §5.3 Chat Turn (with tool call + confirmation)](../../../../architecture/architecture.md#53-chat-turn-with-tool-call--confirmation) — canonical path both tools travel: `tool_call` → `lookup` → `tool_confirmation` → `invoke` → `tool_result`.
- [Architecture §7 Error Handling Strategy](../../../../architecture/architecture.md#7-error-handling-strategy) — Vault failures surface as `{ok:false, error}`; no exception escapes `invoke`.
- [Architecture §8 Extension Points — New tool (built-in)](../../../../architecture/architecture.md#8-extension-points) — "Define `ToolSpec`, register in `ToolRegistry` on load" is exactly the pattern both tools follow.
- [Tech stack — Platform APIs & Interop](../../../../standards/tech-stack.md#platform-apis--interop) — Obsidian `Vault` API pins `app.vault.create` / `app.vault.append` as the non-active-note write surface.
- [Code style — Zod & Tool Schemas](../../../../standards/code-style.md#zod--tool-schemas) — one Zod schema per tool input; `z.infer` for TS types.
- [Code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer) — typed `ToolResult` `{ok, data}|{ok, error}`; `AbortSignal` through `ctx`.
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — never touch `app.vault.adapter` directly; go through `VaultAdapter`.
- [Code style — Error Handling](../../../../standards/code-style.md#error-handling) — adapters catch platform errors; tools return typed `Result` / `ToolResult`.
- [Code style — Logging](../../../../standards/code-style.md#logging) — `event: "tool.invoke.*"` structured fields with `{toolId, thread, durationMs, bytes}`; content payloads are never logged verbatim.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — `VaultAdapter` faked in-memory; provider mocked at `ProviderManager` seam.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — KISS + Single Responsibility: one Vault call per tool; no layered transaction; errors surface as values.
