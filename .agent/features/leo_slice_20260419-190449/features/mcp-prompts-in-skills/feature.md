# F54 — MCP prompts in skill picker

## Purpose

Surface the prompts discovered by [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md) through `prompts/list` as a second, separate **"From MCP"** section inside the [F22 skills-picker-active-skill](../skills-picker-active-skill/feature.md) picker per [FR-MCP-09](../../context.md#fr-mcp-09), so the user can pick an MCP prompt as the active "skill" for the current thread the same way they pick a built-in or user skill per [FR-CHAT-12](../../context.md#fr-chat-12). On selection, the chosen MCP prompt is fetched via `prompts/get` on the owning server, adapted into the `Skill` contract already consumed by [F10 agent-controller-core](../agent-controller-core/feature.md)'s `ContextAssembler`, and bound to `thread.metadata.skillId` through [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) using a namespaced id `mcp.<serverId>.<promptName>` that mirrors the `mcp.<serverId>.<toolName>` convention from [F51](../mcp-client-config-transports/feature.md) per [FR-MCP-06](../../context.md#fr-mcp-06). The take-effect-next-turn, `allowedTools`, and `defaultModel` semantics already fixed by [F22](../skills-picker-active-skill/feature.md) per [FR-SKILL-06](../../context.md#fr-skill-06) / [FR-SKILL-07](../../context.md#fr-skill-07) / [FR-SKILL-08](../../context.md#fr-skill-08) carry over unchanged; this feature adds the second source and the `prompts/get` resolution, nothing more.

## Scope

### In scope

- Extend the [F22](../skills-picker-active-skill/feature.md) `SkillPicker` dropdown with a second, visually separated section labelled **"From MCP"** rendered after the built-in / user-skill list; each row shows the prompt's human-readable `title` (falling back to `name`), one-line `description`, and the owning server's display name so collisions across servers are disambiguated per [FR-MCP-09](../../context.md#fr-mcp-09).
- `MCPPromptSkillAdapter` (pure) that maps an MCP `Prompt` descriptor returned by [F51](../mcp-client-config-transports/feature.md)'s `MCPClient.listPrompts()` into the existing `Skill` shape `{id, name, description, systemPrompt, allowedTools?, examples?, defaultModel?, source}` with `id = "mcp." + serverId + "." + promptName`, `source: "mcp"`, `mcpServerId: serverId`; `systemPrompt` remains unresolved until selection time per [FR-SKILL-01](../../context.md#fr-skill-01).
- `SkillsStore.listAll()` augmentation (or a sibling `CompositeSkillSource`) that merges the existing file-backed catalogue with the MCP-derived list on every picker read; MCP entries carry `source: "mcp"` so [F22](../skills-picker-active-skill/feature.md)'s existing grouping / sorting / active-row logic can branch on source without special-casing the picker.
- On selection of an MCP prompt row, call `MCPClient.getPrompt(serverId, promptName, signal)` (new thin seam added here on top of [F51](../mcp-client-config-transports/feature.md)'s `ServerRuntime.client`) to resolve the `messages` / text returned by `prompts/get` into a single `systemPrompt` string; the resolved body is cached per `(serverId, promptName, promptVersion?)` tuple until disconnect or `prompts/list_changed` invalidation per [FR-MCP-05](../../context.md#fr-mcp-05).
- Thread binding reuses [F22](../skills-picker-active-skill/feature.md)'s path exactly: `ConversationStore.mutate(setSkillId)` writes the namespaced `mcp.<serverId>.<promptName>` id into `thread.metadata.skillId` through [F14](../conversation-persistence-v1/feature.md), emits `thread.skill.changed`, and the `AgentRunner` re-reads the id on its next `send(msg, thread)` per [FR-SKILL-06](../../context.md#fr-skill-06).
- Resolution fallback on load: if `thread.metadata.skillId` starts with `"mcp."` but the owning server is disconnected, missing, or has removed that prompt, [F22](../skills-picker-active-skill/feature.md)'s existing missing-id fallback to `"general"` kicks in and logs `skill.mcp.resolve.miss` with `{threadId, fromId, toId}` via [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md); the thread keeps working with the default skill.
- `allowedTools` / `defaultModel` overlays: MCP prompts that expose `arguments` or metadata suggesting allowed tools or a default model are treated as **optional** on the adapted `Skill` — when absent, [F22](../skills-picker-active-skill/feature.md)'s "undefined ⇒ full registry / settings default" branches apply unchanged per [FR-SKILL-07](../../context.md#fr-skill-07) / [FR-SKILL-08](../../context.md#fr-skill-08); when present, they flow through [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md)'s `listFor(thread)` and `ProviderManager.stream` `model` option via the existing seams.
- `AbortSignal` plumbing: `getPrompt` calls ride the turn's `AbortController` from [F10](../agent-controller-core/feature.md) so [F07 chat-streaming-stop](../chat-streaming-stop/feature.md) cancels in-flight prompt-resolution alongside the LLM request.
- Structured `skill.mcp.*` log events via [F01](../plugin-bootstrap-logging/feature.md): `skill.mcp.list`, `skill.mcp.select` (`{threadId, fromId, toId}`), `skill.mcp.resolve.ok` (`{serverId, promptName, durationMs}`), `skill.mcp.resolve.err`, `skill.mcp.resolve.miss`, reusing the `skill.*` / `mcp.*` field shapes pinned by [F22](../skills-picker-active-skill/feature.md) and [F51](../mcp-client-config-transports/feature.md); prompt bodies are never logged above `debug` per [code-style — Logging](../../../../standards/code-style.md#logging).
- Vitest coverage: adapter maps MCP prompt → `Skill` shape with namespaced id, picker renders "From MCP" section with server-grouped rows, selection round-trips through [F14](../conversation-persistence-v1/feature.md), `getPrompt` body feeds `ContextAssembler` on the next turn only, disconnected-server fallback to `"general"`, structured log-event shape.

### Out of scope

- The MCP host, transports, `.leo/config.json` parsing, `prompts/list` discovery, and `ServerRuntime` lifecycle — owned by [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md) per [FR-MCP-01](../../context.md#fr-mcp-01) / [FR-MCP-02](../../context.md#fr-mcp-02) / [FR-MCP-03](../../context.md#fr-mcp-03) / [FR-MCP-04](../../context.md#fr-mcp-04) / [FR-MCP-05](../../context.md#fr-mcp-05); this feature only reads the already-discovered list and calls a new `getPrompt` helper through F51's adapter.
- The `SkillPicker` React component itself, its dropdown / listbox / keyboard semantics, `HeaderBar` badge, command-palette entry, thread-metadata binding, `ToolRegistry.listFor(thread)` filter, `defaultModel` override, and missing-id fallback infrastructure — owned by [F22 skills-picker-active-skill](../skills-picker-active-skill/feature.md) per [FR-CHAT-12](../../context.md#fr-chat-12) / [FR-SKILL-05](../../context.md#fr-skill-05) / [FR-SKILL-06](../../context.md#fr-skill-06) / [FR-SKILL-07](../../context.md#fr-skill-07) / [FR-SKILL-08](../../context.md#fr-skill-08); this feature only adds a second source and a "From MCP" section header to the existing picker.
- In-plugin skill editor UI — ships with [F39 skill-editor-ui](../../features-index.md); this feature never persists or edits MCP prompt bodies on disk.
- MCP-prompt **arguments** that take user input before resolution — MCP allows `prompts/get` to receive `arguments`. Out of scope here; parameterised prompts are deferred. See Open questions.
- Settings-tab MCP server management UI (toggle, connection status) — ships with [F55 mcp-settings-ui](../../features-index.md) per [FR-MCP-10](../../context.md#fr-mcp-10); this feature assumes enabled servers are already discovered.
- Reconnect / crash handling — owned by [F56 mcp-reconnect-shutdown](../../features-index.md) per [FR-MCP-11](../../context.md#fr-mcp-11) / [FR-MCP-12](../../context.md#fr-mcp-12); a disconnected server's prompts appear absent from the picker and the fallback-to-`"general"` branch covers the active-thread case.
- MCP tool confirmation semantics — owned by [F52 mcp-tool-confirmation](../mcp-tool-confirmation/feature.md) per [FR-MCP-07](../../context.md#fr-mcp-07); MCP **prompts** do not invoke tools by themselves, so F52's gate does not apply to selection or resolution.
- Autocompact summarization prompt behaviour when the active skill is an MCP prompt — owned by [F43 compaction-autocompact](../compaction-autocompact/feature.md); the global open question "skill mid-thread change vs autocompact summary recall" stays open and is not closed here.

## Acceptance criteria

1. With at least one [F51](../mcp-client-config-transports/feature.md) server connected and exposing prompts via `prompts/list`, the [F22](../skills-picker-active-skill/feature.md) `SkillPicker` dropdown renders a second, visually separated **"From MCP"** section after the built-in / user-skill list, each row showing the prompt's `title`/`name`, one-line `description`, and the owning server's display name; disconnected or empty servers contribute zero rows and no section header if the overall MCP list is empty. ([FR-MCP-09](../../context.md#fr-mcp-09), [FR-CHAT-12](../../context.md#fr-chat-12))
2. The `MCPPromptSkillAdapter` maps every MCP `Prompt` descriptor into a `Skill` with `id = "mcp." + serverId + "." + promptName`, `source: "mcp"`, `mcpServerId: serverId`, `name` / `description` from the descriptor, empty `allowedTools` / `defaultModel` unless the descriptor provides them; `systemPrompt` is left as a deferred marker until selection and is populated from `prompts/get` on selection. ([FR-MCP-05](../../context.md#fr-mcp-05), [FR-MCP-06](../../context.md#fr-mcp-06), [FR-SKILL-01](../../context.md#fr-skill-01))
3. Selecting a "From MCP" row writes `thread.metadata.skillId = "mcp.<serverId>.<promptName>"` through [F14](../conversation-persistence-v1/feature.md) via `ConversationStore.mutate(setSkillId)` and triggers `MCPClient.getPrompt(serverId, promptName, signal)`; the resolved `systemPrompt` is cached in-memory and applied on the **next** `AgentRunner.send` only — prior persisted turns are byte-identical per [F22](../skills-picker-active-skill/feature.md)'s next-turn rule. ([FR-SKILL-05](../../context.md#fr-skill-05), [FR-SKILL-06](../../context.md#fr-skill-06), [FR-MCP-09](../../context.md#fr-mcp-09))
4. When the bound MCP skill declares `allowedTools`, [F16](../tool-registry-builtin-read/feature.md)'s `ToolRegistry.listFor(thread)` restricts the tools array per [FR-SKILL-07](../../context.md#fr-skill-07) / [FR-AGENT-12](../../context.md#fr-agent-12); when it declares `defaultModel`, `AgentRunner` passes it to `ProviderManager.stream` per [FR-SKILL-08](../../context.md#fr-skill-08); when either is undefined the defaults from [F22](../skills-picker-active-skill/feature.md) (full registry / settings model) apply unchanged.
5. On plugin reload or server disconnect, if `thread.metadata.skillId` starts with `"mcp."` and no matching prompt is discoverable, the picker falls back to `"general"`, persists the correction through [F14](../conversation-persistence-v1/feature.md), emits `skill.mcp.resolve.miss` with `{threadId, fromId, toId}` via [F01](../plugin-bootstrap-logging/feature.md), and the thread continues working with the default skill — asserted by a Vitest scenario that disconnects the fixture server before rehydration. ([FR-SKILL-05](../../context.md#fr-skill-05), [NFR-REL-05](../../context.md#nfr-rel-05))
6. `prompts/get` calls ride the turn's `AbortController` from [F10](../agent-controller-core/feature.md): Stop per [F07](../chat-streaming-stop/feature.md) cancels any in-flight prompt resolution; resolution errors surface a single inline [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) error chip in the picker row and the thread's active skill remains whatever it was before selection (never a half-resolved MCP skill). ([FR-CHAT-05](../../context.md#fr-chat-05), [NFR-REL-05](../../context.md#nfr-rel-05))
7. Structured log events `skill.mcp.list`, `skill.mcp.select`, `skill.mcp.resolve.ok`, `skill.mcp.resolve.err`, `skill.mcp.resolve.miss` emit via [F01](../plugin-bootstrap-logging/feature.md) with fields `{threadId?, serverId, promptName, fromId?, toId?, durationMs?, bytes?}`; prompt bodies are never logged at `info` or above — covered by a log-sniff Vitest assertion. ([NFR-LOG-04](../../context.md#nfr-log-04))
8. Vitest suite covers: adapter shape, picker section rendering with server-grouped rows, selection persistence through [F14](../conversation-persistence-v1/feature.md), `prompts/get` round-trip against the [F51](../mcp-client-config-transports/feature.md) reference fixture server, next-turn-only `systemPrompt` application (turn N byte-identical, turn N+1 carries the new `systemPrompt`), disconnected-server fallback to `"general"`, `allowedTools` / `defaultModel` pass-through, Abort mid-resolution, and structured log-event shape. ([FR-MCP-09](../../context.md#fr-mcp-09), [FR-SKILL-05](../../context.md#fr-skill-05), [FR-SKILL-06](../../context.md#fr-skill-06), [FR-SKILL-07](../../context.md#fr-skill-07), [FR-SKILL-08](../../context.md#fr-skill-08), [NFR-TEST-05](../../context.md#nfr-test-05))

- [Architecture §10 Concurrency & Lifecycle Rules](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — plugin unload disconnects MCP and drops the resolved-prompt cache; reload resolves again.

## Open questions

- **`defaultModel` collision (global [open questions](../../context.md#open-questions))** — the SRS open question "Skill `defaultModel` override vs MCP-prompt Skills" asks what happens when an MCP prompt and a thread skill both set `defaultModel`. This feature treats the **active** skill as authoritative (MCP prompt when selected, otherwise file-backed skill), so there is no simultaneous collision; flagged for verifier acceptance.
- **Parameterised prompts** — MCP `prompts/get` accepts optional `arguments`. This feature resolves with zero arguments only; parameterised prompts (which would require an argument-picker UI) are deferred. Verifier to accept the scope boundary or pin a follow-up slice.
- **`systemPrompt` assembly from multi-message `prompts/get`** — MCP `prompts/get` can return a `messages[]` array mixing `role: "system" | "user" | "assistant"`. Proposing a deterministic flattener that concatenates `system` messages into `systemPrompt` and prepends non-system messages as leading turns in the next-turn prompt build; verifier to accept or pick an alternative.
- **Cache invalidation on `prompts/list_changed`** — the MCP spec defines a `notifications/prompts/list_changed` event. Proposing we invalidate the resolved-prompt cache for that server on notification and re-fetch on next use; out of scope if F51 does not yet wire the notification channel, in which case the cache lives until disconnect.
- **Visibility when the MCP list is empty** — if no server exposes prompts, proposing the "From MCP" section header is suppressed entirely so the picker is visually identical to [F22](../skills-picker-active-skill/feature.md)'s baseline; verifier to confirm versus always showing a dimmed "no MCP prompts" row.
- **Interaction with autocompact summary recall (global [open questions](../../context.md#open-questions))** — unchanged from [F22](../skills-picker-active-skill/feature.md); an MCP prompt bound mid-thread is treated by autocompact the same way any mid-thread skill change is, and the global open question stays open for [F43](../compaction-autocompact/feature.md) to resolve.
