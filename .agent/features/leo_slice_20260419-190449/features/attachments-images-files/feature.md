# F49 — Image & file attachments

## Purpose

Deliver the phase-5 image-and-file attachment tier called out in [context.md — Phase 5 scope](../../context.md#scope) ("Attachments: image paste → vision model; file drop → quote path") and catalogued in [features-index.md "Notes on coverage edges"](../../features-index.md#notes-on-coverage-edges) as an independent slice that "exposes no net-new requirement IDs beyond those already covered upstream but [is] preserved as [an] independent slice because [it] deliver[s] observable behavior tied to a distinct SRS phase deliverable" — specifically the image-paste → vision-model and file-drop → quoted-path behaviors. This slice wires the [F06 composer](../chat-composer-input/feature.md) paste + drop DOM events into an attachment-tray surface rendered in the [F07 chat-streaming-stop](../chat-streaming-stop/feature.md) turn, passes structured `image` / `document` content blocks through to a vision-capable provider (through [F02 provider-lmstudio-core](../provider-lmstudio-core/feature.md) and future [F38 cloud provider adapters](../../features-index.md)) when the active model advertises vision support, and supplies the `[image]` / `[document]` placeholder-substitution consumed by the autocompact pre-summary step already wired in [F43 compaction-autocompact](../compaction-autocompact/feature.md) per [compact.md §15 "Image/Document Stripping"](../../../../srs/compact.md#15-api-invariant-preservation). No direct FR is claimed — the observable behaviors trace to the Phase 5 deliverable in [context.md](../../context.md#scope), the pre-summary stripping contract in [compact.md §15](../../../../srs/compact.md#15-api-invariant-preservation), and the per-block token-estimation size fixed at [compact.md §4 / `IMAGE_MAX_TOKEN_SIZE = 2000`](../../../../srs/compact.md#4-token-estimation) that [F41 token-estimator-3tier](../token-estimator-3tier/feature.md) already honours.

## Scope

### In scope

- Composer attachment capture on the [F06](../chat-composer-input/feature.md) input: `paste` event inspects `ClipboardEvent.clipboardData.items[]` for any entry whose `kind === 'file'` + `type.startsWith('image/')` and calls `item.getAsFile()` to lift it into a pending-attachment; `drop` event inspects `DataTransfer.files[]` for images + non-image files and lifts each entry; `dragover` is `preventDefault()`-ed per the [HTML drag-drop API](../../../../standards/tech-stack.md#platform-apis--interop) so the `drop` fires.
- Attachment model `Attachment = { id: string; kind: 'image' | 'document'; name: string; mimeType: string; bytes: Uint8Array; size: number }` held in `ChatStore.pendingAttachments[]` per [architecture §6 State Ownership](../../../../architecture/architecture.md#6-state-ownership) — the array empties on turn submit, on explicit `×` click per attachment, or on [F07](../chat-streaming-stop/feature.md) turn cancel before the request hits the provider.
- Per-MIME size guard applied at capture time (default: 10 MB per attachment, 4 attachments per turn) using the bytes known from `File.size`; over-limit attachments are rejected with a [Notice](../../../../standards/tech-stack.md#platform-apis--interop) via [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) and never entered into `pendingAttachments`. Constants bound in the feature module (not spec-level constants).
- Quoted-path substitution for vault-internal drops: when a drop payload carries an Obsidian `text/plain` MIME with a vault-relative path (Obsidian's native file-drag payload), the composer inserts a `"[[path]]"` wikilink at the caret instead of creating a `document` attachment — matches the [Phase 5 "file drop → quote path"](../../context.md#scope) behavior and avoids duplicating vault content the indexer already owns via [F27](../vault-indexer-dirty-queue/feature.md).
- Outgoing request shape: the [F10 agent-controller-core](../agent-controller-core/feature.md) turn builder composes the user message as an array of content blocks when `pendingAttachments.length > 0` — one `{type: "text", text: composerText}` block followed by one `{type: "image", source: {type: "base64", media_type: mimeType, data: base64(bytes)}}` per image attachment and one `{type: "document", source: {type: "base64", media_type: mimeType, data: base64(bytes)}}` per non-image attachment — matching the structured content-block shape the [F43](../compaction-autocompact/feature.md) `stripImagesFromMessages` step already expects per [compact.md §15](../../../../srs/compact.md#15-api-invariant-preservation) and the OpenAI-compatible multimodal surface [F02](../provider-lmstudio-core/feature.md) already models.
- Vision-capability gate: when the active model's [F02](../provider-lmstudio-core/feature.md) capability metadata lacks a vision flag (LM Studio's `/v1/models` does not advertise vision for most local models) and any `pendingAttachments` entry has `kind === 'image'`, turn submission is blocked with a [F13](../ui-visual-states-notifications/feature.md) inline error and a [F01 Logger](../plugin-bootstrap-logging/feature.md) `attachment.image.blocked-no-vision` event; `document` attachments fall back to text-rendered filename + size blocks so they still round-trip without vision support.
- Attachment-tray render surface: a dedicated strip rendered above the [F06](../chat-composer-input/feature.md) textarea (reusing the [F04 chat-sidebar-view](../chat-sidebar-view/feature.md) ComposerInput slot) showing one thumbnail card per pending attachment with name, size, kind icon via [`setIcon`](../../../../standards/tech-stack.md#platform-apis--interop), and a dismiss `×` button; image thumbnails render via an in-memory `URL.createObjectURL(blob)` that is revoked on dismiss and on submit per [code style — React 18](../../../../standards/code-style.md#react-18) cleanup discipline.
- Placeholder-substitution contract for compaction: `stripImagesFromMessages` already ships in [F43](../compaction-autocompact/feature.md) and replaces every `image` / `document` content block with `{type: "text", text: "[image]"}` / `{type: "text", text: "[document]"}` before the summarization call — this feature supplies the live attachment blocks the stripper operates on; a Vitest integration test asserts that a turn with one image + one document summarizes into a message whose content is byte-identical to the stripped form per [compact.md §15](../../../../srs/compact.md#15-api-invariant-preservation).
- Token-estimation wiring: each `image` / `document` content block contributes `IMAGE_MAX_TOKEN_SIZE = 2000` tokens to the pre-API token count through the already-implemented [F41 token-estimator-3tier](../token-estimator-3tier/feature.md) per-block rule at [compact.md §4](../../../../srs/compact.md#4-token-estimation) — this feature does not introduce new estimator logic; it only ensures the content blocks carry the `image` / `document` `type` discriminant so F41's switch statement hits the right branch.
- Structured log events `attachment.capture.paste / attachment.capture.drop / attachment.drop.quoted-path / attachment.reject.oversize / attachment.image.blocked-no-vision / attachment.submit.count` via [F01 Logger](../plugin-bootstrap-logging/feature.md) carrying `{kind, mimeType, size, attachmentCount}` (never the byte payload) per [code style — Logging](../../../../standards/code-style.md#logging).
- Vitest unit coverage per [code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw): paste-capture lifting of a synthesized `ClipboardEvent` with an `image/png` `DataTransferItem`; drop-capture lifting of a synthesized `DragEvent` with one image + one PDF `DataTransferItem`; vault-internal drop inserting a `[[path]]` wikilink instead of creating an attachment; oversize attachment rejected with `Notice` + log event, never entering state; vision-capability gate blocking image submission with non-vision model; content-block shape byte-identical to the multimodal contract on submit; `URL.revokeObjectURL` called on every dismiss + submit; end-to-end integration with [F43](../compaction-autocompact/feature.md) asserting `stripImagesFromMessages` replaces the blocks with `[image]` / `[document]` text markers.

### Out of scope

- Image / file indexing into the RAG store — [FR-IDX-05](../../context.md#fr-idx-05) declares "PDFs / images deferred post-v1"; indexing of binary content stays out of the vault-indexer tier across all phases in this slice.
- Vision-capability *discovery* protocol beyond reading a simple boolean flag from the [F02](../provider-lmstudio-core/feature.md) model-capability metadata — actual LM Studio vision-model detection, cloud-provider model-card lookup, and user-override toggles are deferred to [F38 cloud-providers-safestorage](../cloud-providers-safestorage/feature.md) / a future settings polish.
- Image editing / cropping / annotation inside the attachment tray — attachments are passed through byte-identical from capture to provider.
- Attachment persistence across plugin reloads — `pendingAttachments` lives in `ChatStore` only and is dropped on reload; completed-turn attachments land inside the conversation JSON as structured content blocks and are persisted by [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) under the same content-block schema the provider consumed.
- Drag-and-drop of attachments *out of* the chat (e.g. save-image-to-vault) — not in scope.
- Clipboard capture of non-image mime types on paste (e.g. PDF paste from a native viewer) — `paste` flow handles images only; non-images reach the composer via `drop` only, matching the Phase 5 deliverable phrasing "image paste → vision model; file drop → quote path".

## Acceptance criteria

1. A `paste` event on the [F06](../chat-composer-input/feature.md) textarea whose `ClipboardEvent.clipboardData.items[]` contains at least one `kind === 'file'` + `type.startsWith('image/')` entry lifts each such file into `ChatStore.pendingAttachments[]` with `kind: 'image'`; the textarea's default paste is preserved for any text portion of the same clipboard payload; a Vitest test synthesizes a `ClipboardEvent` fixture and asserts both the state append and the preserved textarea text. (feature-specific, traces to [context.md Phase 5](../../context.md#scope))
2. A `drop` event on the same textarea whose `DataTransfer.files[]` contains a mix of images and non-images lifts each image as `kind: 'image'` and each non-image as `kind: 'document'`, each entry carrying `{id, name, mimeType, bytes, size}`; `dragover` is `preventDefault()`-ed so the drop fires; a Vitest test synthesizes a `DragEvent` with two files and asserts the resulting state array. (feature-specific, traces to [context.md Phase 5](../../context.md#scope))
3. A `drop` whose payload carries an Obsidian `text/plain` MIME with a vault-relative path inserts `[[path]]` at the caret in the composer textarea instead of creating an attachment; no `pendingAttachments` entry is added for the vault-internal drop; a Vitest test on the synthesized vault-drop payload asserts the textarea text and the empty attachments array. ([context.md Phase 5](../../context.md#scope) "file drop → quote path")
4. Turn submit with `pendingAttachments.length > 0` composes the user message as a content-block array: one `{type:"text", text}` block first, then one `{type:"image", source:{type:"base64", media_type, data}}` per image and one `{type:"document", source:{type:"base64", media_type, data}}` per document in capture order; a Vitest + `msw` fixture captures the outbound body and asserts block order and per-block shape byte-identical to the expected JSON. (traces to [compact.md §15](../../../../srs/compact.md#15-api-invariant-preservation))
5. When the active model's [F02](../provider-lmstudio-core/feature.md) capability metadata lacks vision support and `pendingAttachments` contains any `kind: 'image'` entry, turn submission is blocked with a [F13](../ui-visual-states-notifications/feature.md) inline error and an `attachment.image.blocked-no-vision` [F01](../plugin-bootstrap-logging/feature.md) log event; no provider call is made; Vitest asserts zero `ProviderManager.stream` invocations on the blocked branch. (feature-specific)
6. Oversize attachments (per-attachment size > 10 MB or total count > 4 per turn) are rejected at capture time with a `Notice` via [F13](../ui-visual-states-notifications/feature.md) and an `attachment.reject.oversize` [F01](../plugin-bootstrap-logging/feature.md) log event; `pendingAttachments` never holds an over-limit entry; Vitest covers both per-attachment and per-turn caps at the boundary. (feature-specific)
7. After a turn is submitted, `ChatStore.pendingAttachments` is empty and every `URL.createObjectURL` blob created for the attachment tray thumbnails has been revoked via `URL.revokeObjectURL`; dismiss-click on an attachment before submit revokes that specific blob and removes only that entry; Vitest spies on `URL.createObjectURL` / `revokeObjectURL` and asserts matched call counts. ([code style — React 18](../../../../standards/code-style.md#react-18) cleanup)
8. End-to-end with [F43](../compaction-autocompact/feature.md): a conversation containing one `image` and one `document` content block survives autocompact summarization as `[image]` and `[document]` text markers in the pre-summary request body — reusing F43's existing `stripImagesFromMessages` step unchanged. Vitest integration test asserts the outbound summarization body's content-block array is byte-identical to the expected stripped form. ([compact.md §15](../../../../srs/compact.md#15-api-invariant-preservation))
9. Token estimation for a user message containing `image` / `document` blocks credits each such block with `IMAGE_MAX_TOKEN_SIZE = 2000` tokens through the already-implemented [F41](../token-estimator-3tier/feature.md) per-block rule — this feature's content-block discriminants (`type: "image"` / `type: "document"`) are recognized by F41 without modification. A Vitest fixture with one text block + one image + one document asserts the estimator returns `len(text)/4 + 2000 + 2000` tokens pre-4/3 padding. ([compact.md §4](../../../../srs/compact.md#4-token-estimation))

## Dependencies

- [F43 compaction-autocompact](./../compaction-autocompact/feature.md) — consumes the `image` / `document` content blocks this feature emits; the pre-summary `stripImagesFromMessages` step at [compact.md §15](../../../../srs/compact.md#15-api-invariant-preservation) replaces them with `[image]` / `[document]` text markers before the summarization API call, and AC#8 asserts the round-trip works end-to-end against F43's existing implementation without modification.
- [F07 chat-streaming-stop](./../chat-streaming-stop/feature.md) — the [F06 composer](../chat-composer-input/feature.md) paste / drop capture flows ride the same turn lifecycle that F07 owns, and the "turn cancel before submit" branch in AC#7 empties `pendingAttachments` via the same `AbortController` teardown F07 installs; the attachment tray mounts beside the composer without disturbing F07's streaming surface or stop glyph.
- [F06 chat-composer-input](../chat-composer-input/feature.md) — supplies the textarea DOM surface the `paste` / `drop` / `dragover` listeners attach to and the caret where the vault-drop wikilink insertion happens; reused unchanged.
- [F02 provider-lmstudio-core](../provider-lmstudio-core/feature.md) — ingests the content-block array on submit and surfaces model-capability metadata (vision flag) the gate at AC#5 reads.
- [F10 agent-controller-core](../agent-controller-core/feature.md) — its turn builder composes `pendingAttachments` into the outgoing content-block array and clears them on submit or abort.
- [F41 token-estimator-3tier](../token-estimator-3tier/feature.md) — already credits `image` / `document` content blocks at `IMAGE_MAX_TOKEN_SIZE = 2000` per [compact.md §4](../../../../srs/compact.md#4-token-estimation); reused unchanged.
- [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) — supplies the `Notice` / inline-error surfaces used by the oversize-reject and the vision-capability gate.
- [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md) — `Logger` emits the `attachment.*` structured events.
- [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) — persists completed-turn attachments as structured content blocks in the conversation JSON alongside text messages; the on-disk schema already models content-block arrays.
- Downstream consumers tracked in [features-index.md](../../features-index.md): [F38 cloud-providers-safestorage](../../features-index.md) (cloud adapters consuming the same content-block contract with richer vision-model metadata), future MCP-resource-picker inserts in [F53](../../features-index.md) (parallel attachment surface over MCP resources).
- No direct FR coverage — this slice is classified in [features-index.md "Notes on coverage edges"](../../features-index.md#notes-on-coverage-edges) as an independent phase-5 deliverable whose observable behaviors (paste-image, file-drop, quoted-path, vision routing, placeholder substitution) are pinned by [context.md Phase 5 scope](../../context.md#scope) and [compact.md §15](../../../../srs/compact.md#15-api-invariant-preservation).

## Implementation notes

- [Architecture §3.1 UI Layer](../../../../architecture/architecture.md#31-ui-layer-react-mounted-inside-obsidian-views) — hosts the attachment tray and the composer DOM listeners; paste / drop handlers live inside the ComposerInput React subtree.
- [Architecture §3.2 Agent Layer](../../../../architecture/architecture.md#32-agent-layer) — `AgentRunner.send` composes the multimodal user message from text + `pendingAttachments` at turn boundary.
- [Architecture §4 Key Contracts — UserMessage / StreamEvent](../../../../architecture/architecture.md#4-key-contracts) — content-block arrays extend the existing `UserMessage` shape behind the same `AgentRunner` surface.
- [Architecture §5.2 Chat Turn (no tools)](../../../../architecture/architecture.md#52-chat-turn-no-tools) — the attachment-carrying turn uses the same no-tools flow plus multimodal content blocks inside the single user message.
- [Architecture §5.6 Cancellation](../../../../architecture/architecture.md#56-cancellation) — abort during submit clears `pendingAttachments` + revokes pending object URLs in the same `AbortController` teardown as the provider stream.
- [Architecture §6 State Ownership](../../../../architecture/architecture.md#6-state-ownership) — `ChatStore.pendingAttachments` is session-scoped, not persisted across reload.
- [Architecture §10 Concurrency & Lifecycle Rules](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — object-URL blobs owned by the tray scope are revoked on dismiss, submit, and unmount.
- [Tech stack — Platform APIs & Interop](../../../../standards/tech-stack.md#platform-apis--interop) — `ClipboardEvent` / `DragEvent` / `DataTransfer` / `URL.createObjectURL` / `Notice` / `setIcon` are the host-provided surfaces used here.
- [Tech stack — UI Layer](../../../../standards/tech-stack.md#ui-layer) — React 18 controlled attachment tray with blob-URL lifecycle tied to effect cleanup.
- [Code style — TypeScript](../../../../standards/code-style.md#typescript) — discriminated union on `kind: 'image' | 'document'`; `readonly` on content-block arrays; no `any`.
- [Code style — React 18](../../../../standards/code-style.md#react-18) — `URL.createObjectURL` paired with `URL.revokeObjectURL` in effect cleanup; dismiss revokes synchronously.
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — never touch `app.vault.adapter` directly; vault-drop quoted-path resolution uses Obsidian's native drop payload parsing.
- [Code style — Async & Concurrency](../../../../standards/code-style.md#async--concurrency) — `AbortSignal` threads from turn scope through submit and clears tray state on cancel.
- [Code style — Error Handling](../../../../standards/code-style.md#error-handling) — over-limit capture and vision-gate rejection surface as typed state, not thrown errors; no exception escapes the composer handler.
- [Code style — Logging](../../../../standards/code-style.md#logging) — `attachment.*` structured events carry `{kind, mimeType, size, attachmentCount}` only; byte payloads never logged.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — synthesized `ClipboardEvent` / `DragEvent` fixtures; `msw` captures outbound provider body for multimodal-shape assertions.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — Single Responsibility: this slice captures, carries, and surfaces attachments; it does not index them, edit them, or re-encode them.
- [context.md Phase 5 scope](../../context.md#scope), [compact.md §4](../../../../srs/compact.md#4-token-estimation), [compact.md §15](../../../../srs/compact.md#15-api-invariant-preservation) — authoritative external anchors; no spec content restated here.

## Open questions

- **Per-turn attachment caps (10 MB / 4 attachments)** — The SRS does not pin size or count limits. Proposal: 10 MB per attachment + 4 per turn as conservative in-module defaults, surfaced as Advanced-section settings when [F03 settings-tab-scaffold](../settings-tab-scaffold/feature.md) gains an attachments section; verifier to confirm or tighten.
- **Vision-capability flag source** — [F02](../provider-lmstudio-core/feature.md) does not currently model a `supportsVision` boolean on its model-capability metadata; LM Studio's `/v1/models` rarely advertises it. Proposal: add an opt-in `vision: boolean` field on the provider's model-capability metadata, defaulting to `false` for LM Studio until the user enables it; cloud adapters in [F38](../cloud-providers-safestorage/feature.md) will set it from model-card lookup. Verifier to confirm the seam lands in F02 vs F38.
- **Vault-drop detection heuristic** — Obsidian's native drag payload carries a `text/plain` MIME with the vault-relative path (and sometimes a wrapping `[[wikilink]]` form). Proposal: prefer the wrapped form when present, else wrap the raw path, and treat any `text/plain`-only drop whose value resolves to an existing `TFile` as a vault-drop. Verifier to confirm against current Obsidian behavior.
- **Attachment persistence shape** — [F14 conversation-persistence-v1](../conversation-persistence-v1/feature.md) stores conversation JSON; base64-encoded image bytes inflate on-disk size materially. Proposal: store the content-block array verbatim in v1 (matches the over-the-wire shape the provider and F43 both consume) and revisit after the 10k-vault perf pass in [F50](../../features-index.md). Verifier to confirm.
- **Non-image paste** — The Phase 5 deliverable phrases the split as "image paste → vision; file drop → quote path". Non-image clipboard payloads (e.g. PDF blob from a native app) fall through to the default text paste in this slice. Verifier to confirm the intentional asymmetry.
- **Document MIME allowlist** — `kind: 'document'` currently accepts any non-image MIME dropped into the composer. Proposal: allowlist `application/pdf`, `text/*`, `application/json` in v1 and reject the rest with a `Notice`; verifier to confirm or broaden.
