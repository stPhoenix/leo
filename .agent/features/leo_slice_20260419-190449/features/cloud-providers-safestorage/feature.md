# F38 — Cloud provider adapters + safeStorage keys

## Purpose

Open Leo's local-first stance to user-opted cloud inference without ever putting a plaintext secret in the vault: three concrete cloud adapters (OpenAI, Anthropic, Ollama) plus a declarative "custom" adapter slot are implemented against the `Provider` interface delivered by [F02](../provider-lmstudio-core/feature.md) per [FR-PROV-10](../../context.md#fr-prov-10), and the API keys they need are written through an Electron `safeStorage` wrapper backed by the OS keyring when available per [FR-PROV-10](../../context.md#fr-prov-10) and [NFR-DATA-01](../../context.md#nfr-data-01). When `safeStorage.isEncryptionAvailable()` returns false the store degrades to an obfuscated fallback and surfaces a persistent warning banner so the user knows their secret is only lightly protected, and plaintext in `data.json` / the vault is forbidden on every write path per [FR-PROV-10](../../context.md#fr-prov-10). No telemetry or analytics is added on top of the cloud call per [NFR-DATA-03](../../context.md#nfr-data-03); the only outbound traffic is the user-configured provider endpoint. Once a cloud provider is active, the feature also lights the $-per-message slot already wired through the token-usage footer so cost becomes visible alongside the existing input/output/total counts. It is the gate that Phase 5 relies on before [F51 mcp-client-config-transports](../../features-index.md) reuses the same `safeStorage` seam for MCP secrets per [NFR-DATA-04](../../context.md#nfr-data-04).

## Scope

### In scope

- `OpenAIProvider`, `AnthropicProvider`, `OllamaProvider` implementing the F02 `Provider` interface (chat stream + model list + cancellation signal), registered alongside `LMStudioProvider` in `ProviderManager` so selection is a settings toggle per [FR-PROV-07](../../context.md#fr-prov-07) and [FR-PROV-10](../../context.md#fr-prov-10).
- A declarative "custom" adapter taking `{baseURL, authHeader, model, embeddingModel?}` and reusing the F02 OpenAI-compatible transport so a user can point Leo at any OpenAI-compatible endpoint without shipping code.
- `SafeStorage` adapter wrapping `electron.safeStorage.encryptString` / `decryptString`, exposing `get(key) / set(key, plaintext) / delete(key) / has(key)`; cipher bytes stored through Obsidian `loadData` / `saveData` in `data.json` under an explicit `secrets` subtree that is never mirrored to the vault markdown per [FR-PROV-10](../../context.md#fr-prov-10) and [NFR-DATA-01](../../context.md#nfr-data-01).
- Keyring-unavailable degrade path: `safeStorage.isEncryptionAvailable() === false` switches to an XOR-obfuscated fallback with a persistent warning banner in the Settings tab and a `Notice` on first degraded write so the user is aware their key is not kernel-protected.
- Provider-section UI extension (mounts into the F03 Provider section): provider picker (LM Studio / OpenAI / Anthropic / Ollama / Custom), per-provider API-key password field with reveal toggle, per-provider model pickers, and a "Clear stored key" action that calls `SafeStorage.delete`.
- Cost-in-$ per message: when the active provider has a `pricePerInputToken` / `pricePerOutputToken` entry (built-in tables for the three cloud providers; user-supplied overrides in settings), the token-usage footer (already shipped by [F12](../token-usage-indicator/feature.md)) adds a `$N.NN` slot; LM Studio / Ollama-local stays silent per [FR-CHAT-11](../../context.md#fr-chat-11).
- Structured log events (`provider.cloud.selected`, `provider.cloud.request`, `safestorage.set`, `safestorage.fallback`, `safestorage.warning-shown`) through the [F01 Logger](../plugin-bootstrap-logging/feature.md), never logging key material above `debug` and redacting it there too.
- Vitest coverage via `msw` fixtures for each cloud adapter (stream parse, `listModels`, cancellation) and a `SafeStorage` suite with a mocked `electron.safeStorage` module covering the encrypted path, the unavailable/fallback path, and the round-trip of a stored key.

### Out of scope

- LM Studio local provider and the `Provider` interface itself — already shipped by [F02 provider-lmstudio-core](../provider-lmstudio-core/feature.md).
- MCP secrets (stdio `env` API keys, HTTP+SSE headers) storage through `safeStorage` — deferred to [F51 mcp-client-config-transports](../../features-index.md) and [F55 mcp-settings-ui](../../features-index.md) per [NFR-DATA-04](../../context.md#nfr-data-04); this feature delivers the reusable `SafeStorage` adapter they depend on.
- `/context` category breakdown and the per-category token grid — ships with [F47 context-command-grid](../../features-index.md).
- Tool-use / `tools` OpenAI parameter wiring on top of cloud adapters — already owned by [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md); cloud providers reuse the same `StreamEvent` shape.
- Token-usage footer itself (input / output / total counts) — already shipped by [F12 token-usage-indicator](../token-usage-indicator/feature.md); this feature only populates its `$` slot.
- Embedding providers for cloud — the embedding client stays on LM Studio / local unless the user picks a cloud provider that exposes embeddings; swap-out details beyond interface reuse are orchestration concerns for later polish.

## Acceptance criteria

1. `OpenAIProvider`, `AnthropicProvider`, `OllamaProvider`, and the declarative `CustomProvider` each implement the F02 `Provider` interface end-to-end — chat streaming (`StreamEvent.token`/`usage`/`done`), `listModels()`, and cancellation via `AbortSignal` — and are registered in `ProviderManager` so the active provider is selected by a single settings field. ([FR-PROV-07](../../context.md#fr-prov-07), [FR-PROV-10](../../context.md#fr-prov-10))
2. No code path writes an API key or other cloud secret to the vault filesystem in plaintext; every `SafeStorage.set` call is routed through `electron.safeStorage.encryptString` when available, and a Vitest assertion scans the `data.json` and vault write paths for raw key presence to guard the invariant. ([FR-PROV-10](../../context.md#fr-prov-10), [NFR-DATA-01](../../context.md#nfr-data-01))
3. When `electron.safeStorage.isEncryptionAvailable()` returns false at load, `SafeStorage` switches to an XOR-obfuscated fallback, a persistent warning banner mounts at the top of the Settings tab, and a one-shot `Notice` fires on the first degraded write; the banner remains until the keyring becomes available or the stored key is cleared. ([FR-PROV-10](../../context.md#fr-prov-10))
4. The Provider section in Settings renders per-provider API-key password fields with reveal and "Clear stored key" actions; editing a key calls `SafeStorage.set`, reloading the plugin and opening Settings restores the masked state without ever surfacing the plaintext, and deletion clears both the ciphertext and any in-memory cache. ([FR-PROV-10](../../context.md#fr-prov-10))
5. When the active provider is one of OpenAI / Anthropic / a cloud-priced Custom, the chat footer (shipped by F12) renders a `$N.NN` slot computed from the provider's per-input / per-output pricing table applied to the `StreamEvent.usage` counts; when the active provider is LM Studio or Ollama-local the $ slot stays hidden. ([FR-PROV-10](../../context.md#fr-prov-10))
6. No telemetry, analytics, or background cloud pings are introduced — the only network traffic this feature can cause is the user-initiated chat / model-list call to the configured provider endpoint, and a Vitest network-sniff over the adapter suite asserts zero unsolicited outbound requests. ([NFR-DATA-03](../../context.md#nfr-data-03))
7. All data paths continue to honour the local-first default — LM Studio remains the on-install selection, cloud providers require explicit user opt-in through the Settings picker plus an API-key entry before any cloud call is issued, and a Vitest scenario covering a fresh install asserts zero cloud fetches until both conditions are met. ([NFR-DATA-01](../../context.md#nfr-data-01))
8. Structured log events cover the cloud paths: `provider.cloud.selected`, `provider.cloud.request` (with provider id + model, never key material), `safestorage.set` / `safestorage.fallback` / `safestorage.warning-shown` (with booleans / counts only, never ciphertext or plaintext), asserted by the Vitest logger-stub matrix. ([FR-PROV-10](../../context.md#fr-prov-10), [NFR-DATA-03](../../context.md#nfr-data-03))

- Drives requirements [FR-PROV-10](../../context.md#fr-prov-10), [NFR-DATA-01](../../context.md#nfr-data-01), [NFR-DATA-03](../../context.md#nfr-data-03); lights the cloud path of [FR-CHAT-11](../../context.md#fr-chat-11).

## Implementation notes

- [Architecture §3.4 Adapters — ProviderManager / SafeStorage](../../../../architecture/architecture.md#34-adapters) — places cloud adapters and the `SafeStorage` wrapper as adapter-layer rows; this feature ships the three concrete cloud adapters plus that wrapper.
- [Architecture §6 State Ownership — MCP server configs / secrets](../../../../architecture/architecture.md#6-state-ownership) — pins secrets to `SafeStorage` (never vault plaintext); the API-key store follows the same rule for cloud-provider keys.
- [Architecture §7 Error Handling Strategy — safeStorage unavailable](../../../../architecture/architecture.md#7-error-handling-strategy) — mandates XOR-obfuscated fallback + persistent warning banner; AC 3 traces to this row.
- [Architecture §8 Extension Points — New LLM provider](../../../../architecture/architecture.md#8-extension-points) — new adapters register via the `Provider` interface; OpenAI / Anthropic / Ollama / Custom follow that seam.
- [Tech stack — Platform APIs](../../../../standards/tech-stack.md#platform-apis) — names Electron `safeStorage` (via the `electron` global) as the API for future cloud-provider API keys; the `SafeStorage` adapter is the only consumer.
- [Tech stack — Agent Layer](../../../../standards/tech-stack.md#agent-layer) — pins direct `fetch` + SSE parsing for provider calls; cloud adapters follow the same no-SDK stance chosen for LM Studio.
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — routes persistence through `loadData` / `saveData`; the `secrets` subtree in `data.json` follows that rule, never bypassing it to the adapter.
- [Code style — Error Handling](../../../../standards/code-style.md#error-handling) — typed `Result` / `ToolResult` surfacing and "never swallow silent"; the keyring-unavailable degrade logs + notices + banners rather than silently falling back.
- [Code style — Logging](../../../../standards/code-style.md#logging) — leveled logs with no key material above `debug`; dictates the `provider.cloud.*` / `safestorage.*` event shapes.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — fixes the fixture stack for the three cloud adapters and the `electron.safeStorage` mock.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — "fail fast" and "make it observable" justify surfacing the keyring-unavailable state through banner + Notice + log rather than a silent degrade.

## Open questions

- Pricing table freshness: bundled tables for OpenAI / Anthropic / Ollama drift over time; is this a user-overridable field only, a bundled-but-user-overridable seam, or a bundled-only value deferred to a future settings pass? Current lean is user-overridable with bundled defaults; verifier to confirm.
- Custom provider authHeader shape: `Authorization: Bearer <key>` covers most OpenAI-compatible endpoints, but Anthropic-style `x-api-key` + `anthropic-version` headers differ. Propose a `{headerName, headerPrefix?}` pair on the custom spec unless verification picks a fixed shape.
- Fallback cipher strength: XOR obfuscation is called out by Architecture §7 but is weak by design; should the banner copy explicitly state "not encrypted, only obfuscated" to avoid a false sense of security? Current lean is yes.
- Cloud-provider `listModels()` semantics for Anthropic: the Anthropic API does not expose a `/v1/models` endpoint matching OpenAI's shape; propose a static bundled list with an "override model id" free-text field unless verification picks a different fallback.
- Embedding-provider swap when the active chat provider is cloud: the SRS keeps embedding model configurable separately per [FR-PROV-08](../../context.md#fr-prov-08); propose keeping embeddings on LM Studio by default even when chat is cloud, flagged in release notes, pending verifier confirmation.
