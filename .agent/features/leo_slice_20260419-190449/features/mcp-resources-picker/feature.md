# F53 — MCP resource picker

## Purpose

Surface the resources discovered by [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md) through `resources/list` as a first-class, inline **resource picker** inside the chat view per [FR-MCP-08](../../context.md#fr-mcp-08). When the user opens the picker from the composer, selects one or more MCP resources, and sends the next message, Leo SHALL fetch each resource's content via `resources/read` on the owning server, inline the content as a context attachment on that single next message only, and never mutate the allowlist, skill, or provider path already shipped. No new persistence, no new agent loop — the resource content rides along the turn that the user composes after picking, scoped to the same AgentController turn contract already built in [F10 agent-controller-core](../agent-controller-core/feature.md). The inline picker lives inside the [F04 chat-sidebar-view](../chat-sidebar-view/feature.md) shell so the UI respects the six-region decomposition and the FR-UI-08 "never native modal" rule.

## Scope

### In scope

- Inline resource picker surfaced from the composer area of the [F04](../chat-sidebar-view/feature.md) `ChatView` shell (anchored above or beside `ComposerInput`), opened via a composer affordance and closable via Esc, click-outside, or explicit Cancel per [FR-UI-08](../../context.md#fr-ui-08).
- Picker reads the discovered resources from [F51](../mcp-client-config-transports/feature.md)'s `MCPClient.listResources()` read-only surface, grouped by `serverId` (server display name), each entry showing `uri`, human-readable `name`/`title` when present, and `mimeType` when available per [FR-MCP-05](../../context.md#fr-mcp-05).
- Multi-select with a lightweight listbox (arrow keys, Space/Enter toggle, Esc close) and an accumulating staging tray so a single picker session can attach several resources from the same or different servers.
- "Next-message" scoping: selections live in an in-memory `ChatView` state slot only (no disk write) and are consumed + cleared when the user submits the next turn through [F06 chat-composer-input](../chat-composer-input/feature.md) / [F10](../agent-controller-core/feature.md); cancelling, switching threads, or closing the view discards them without persistence per [FR-MCP-08](../../context.md#fr-mcp-08).
- On turn submit, the picker's staged URIs are resolved to content via `MCPClient.readResource(serverId, uri, signal)` (new thin seam added here on top of [F51](../mcp-client-config-transports/feature.md)'s `ServerRuntime.client`) and the resulting `{uri, mimeType, text|blob}` records are prepended as context blocks to the outgoing message in the order they were staged, under a stable "MCP resources" preamble so the agent can distinguish them from vault content.
- Read failures per resource surface as inline [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) error chips next to the staged row with a Retry affordance; the turn proceeds with whatever resources succeeded plus a clear in-prompt marker listing the failed URIs, never silently dropping them.
- `AbortSignal` plumbing: the picker's `readResource` calls are run inside the turn's `AbortController` from [F10](../agent-controller-core/feature.md), so Stop from [F07 chat-streaming-stop](../chat-streaming-stop/feature.md) cancels in-flight resource reads alongside the LLM request.
- Structured `mcp.resource.*` log events via [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md): `mcp.resource.picker.open`, `mcp.resource.pick`, `mcp.resource.read.ok/err`, `mcp.resource.attach` with `{serverId, uri, mimeType, bytes?, durationMs}` — never logging resource content above `debug` per [code-style — Logging](../../../../standards/code-style.md#logging).
- Vitest coverage: picker mount/unmount against the [F51](../mcp-client-config-transports/feature.md) fixture server, multi-select + stage + clear, `resources/read` happy + error + Abort paths, content-block composition onto the next turn with byte-identical ordering, and a negative assertion that no state persists across turns or thread switches.

### Out of scope

- The MCP host, transports, `.leo/config.json` parsing, `resources/list` discovery and `ServerRuntime` lifecycle — owned by [F51 mcp-client-config-transports](../mcp-client-config-transports/feature.md); this feature only reads the already-discovered list and calls `readResource` through F51's adapter.
- Tool confirmation gating — this picker attaches resource **content**, not a tool call; F52's confirmation path (owned by [F52 mcp-tool-confirmation](../mcp-tool-confirmation/feature.md)) does not apply. See Open questions.
- MCP prompts in the skill picker — ships with [F54 mcp-prompts-in-skills](../mcp-client-config-transports/feature.md) per [FR-MCP-09](../../context.md#fr-mcp-09); this feature is resource-scoped only.
- Settings-tab MCP server management UI — ships with [F55 mcp-settings-ui](../mcp-client-config-transports/feature.md) per [FR-MCP-10](../../context.md#fr-mcp-10); this feature assumes enabled servers are already discovered.
- Reconnect / crash handling — owned by [F56 mcp-reconnect-shutdown](../mcp-client-config-transports/feature.md) per [FR-MCP-11](../../context.md#fr-mcp-11) / [FR-MCP-12](../../context.md#fr-mcp-12); a disconnected server's resources appear empty and this feature surfaces that via the picker's empty-state.
- Cross-turn persistence / sticky resources — the SRS scopes this to "the next message"; durable per-thread resource pinning is explicitly out of scope.
- Vault attachment surface (paste-image / file-drop) — owned by [F49 attachments-images-files](../attachments-images-files/feature.md); this feature composes alongside F49 when both are present but does not depend on it.
- Content transformation, chunking, or embedding of resource payloads — the content is inlined verbatim (truncated only by existing compaction layers [F42](../compaction-microcompact/feature.md) / [F43](../compaction-autocompact/feature.md)) into the outgoing turn's context.

## Acceptance criteria

1. With at least one [F51](../mcp-client-config-transports/feature.md) server connected and exposing resources via `resources/list`, the composer renders a visible "MCP resources" affordance that, when activated (mouse or keyboard), mounts an inline picker inside the [F04](../chat-sidebar-view/feature.md) shell (never an Obsidian native `Modal`) — asserted by a Vitest test that opens the picker and inspects that no `Modal` subclass was constructed. ([FR-MCP-08](../../context.md#fr-mcp-08), [FR-UI-08](../../context.md#fr-ui-08))
2. The picker lists every resource from `MCPClient.listResources()` grouped by `serverId` with the server's human name, the resource `uri`, optional `name`/`title`, and `mimeType`; empty / disconnected servers render a dimmed "no resources" row so the user sees the known surface. ([FR-MCP-05](../../context.md#fr-mcp-05), [FR-MCP-08](../../context.md#fr-mcp-08))
3. The picker supports multi-select via mouse click and keyboard (Space/Enter to toggle, arrow keys to navigate, Home/End to jump, Esc to close without committing); Tab moves focus out; the staged selection is visible as a small chip row anchored to the composer and persists across picker reopen within the same unsent turn. ([FR-UI-08](../../context.md#fr-ui-08), [NFR-USE-05](../../context.md#nfr-use-05), [NFR-USE-06](../../context.md#nfr-use-06))
4. On the next message submit, every staged resource is resolved via `MCPClient.readResource(serverId, uri, signal)` inside the turn's `AbortController`; the content blocks are prepended to the outgoing message in staging order under a stable "MCP resources" preamble, before any user text or [F08 editor-bridge-focused-context](../editor-bridge-focused-context/feature.md) context — asserted via Vitest byte-identity snapshot of the composed content blocks. ([FR-MCP-08](../../context.md#fr-mcp-08), [FR-AGENT-01](../../context.md#fr-agent-01))
5. A failing `resources/read` (transport error, server crash, timeout) is surfaced inline on the staged chip via [F13](../ui-visual-states-notifications/feature.md)'s error palette with a Retry action, the turn still proceeds with the remaining successful reads, and the outgoing message includes a synthetic marker listing the failed URIs so the agent is never silently missing context. ([NFR-REL-05](../../context.md#nfr-rel-05), [FR-UI-06](../../context.md#fr-ui-06))
6. Pressing Stop during streaming per [F07](../chat-streaming-stop/feature.md) cancels any still in-flight `resources/read` calls; no resource content lands in any post-cancel message. Closing the picker, switching threads, unloading the plugin, or cancelling a turn discards all staged URIs with zero persistence — asserted by a negative Vitest test scanning both in-memory state and on-disk thread JSON. ([FR-CHAT-05](../../context.md#fr-chat-05), [FR-MCP-08](../../context.md#fr-mcp-08))
7. Structured log events `mcp.resource.picker.open`, `mcp.resource.pick`, `mcp.resource.read.ok`, `mcp.resource.read.err`, `mcp.resource.attach` emit via [F01](../plugin-bootstrap-logging/feature.md) with `{serverId, uri, mimeType, bytes?, durationMs, attachedCount?}` fields; resource payloads are never logged at `info` or above — covered by a log-sniff Vitest assertion. ([NFR-LOG-04](../../context.md#nfr-log-04))
8. Vitest coverage per [NFR-TEST-05](../../context.md#nfr-test-05) uses the [F51](../mcp-client-config-transports/feature.md) reference fixture server to drive: discovery → open picker → multi-select → submit → byte-identical content-block composition → clear; plus an SSE path via `msw`, a `resources/read` failure path with Retry, and an Abort mid-read scenario. ([NFR-TEST-05](../../context.md#nfr-test-05), [NFR-TEST-01](../../context.md#nfr-test-01))

- [F13 ui-visual-states-notifications](../ui-visual-states-notifications/feature.md) — provides the `error` and `tool-running`-adjacent palettes used on staged-chip failure states and retry affordance.
- [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md) — provides the `Logger` for the `mcp.resource.*` structured events.
- Drives requirement [FR-MCP-08](../../context.md#fr-mcp-08); contributes to [NFR-TEST-05](../../context.md#nfr-test-05), [NFR-LOG-04](../../context.md#nfr-log-04), [NFR-REL-05](../../context.md#nfr-rel-05).
- Downstream consumers tracked in [features-index.md](../../features-index.md): none; F55 may surface an empty-state hint "no resources yet" when a configured server has none.

## Implementation notes

- [Architecture §1 Local-first + Registry pattern](../../../../architecture/architecture.md#1-architectural-principles) — MCP resources are opt-in network egress already gated by F51's user-configured `mcpServers`; this feature does not open any new network seam.
- [Architecture §3.2 Agent Layer — MCPClient](../../../../architecture/architecture.md#32-agent-layer) — the resource-read helper lives beside `listResources()`; domain code never imports SDK types.
- [Architecture §4 Key Contracts — ToolSpec / StreamEvent](../../../../architecture/architecture.md#4-key-contracts) — this feature does NOT register a tool; it attaches content blocks on the user message instead, so `ToolSpec` / tool confirmation contracts are untouched.
- [Architecture §5.1 Plugin Startup](../../../../architecture/architecture.md#51-plugin-startup) — F51 registers MCP tools / resources as servers resolve; the picker observes that list lazily and does not add startup work.
- [Architecture §5.5 MCP Tool Call](../../../../architecture/architecture.md#55-mcp-tool-call) — the resource-read path is structurally similar but skips the confirmation branch since no tool invocation happens.
- [Architecture §6 State Ownership](../../../../architecture/architecture.md#6-state-ownership) — staged picks live in `ChatView`-scoped React state only; no entry is added to `ConversationStore` or `data.json` per the "next-message only" contract.
- [Architecture §7 Error Handling Strategy](../../../../architecture/architecture.md#7-error-handling-strategy) — `readResource` returns a typed `{ok:true, data} | {ok:false, error}` shape so the picker can render Retry without exceptions escaping.
- [Architecture §10 Concurrency & Lifecycle Rules](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — on `ChatView` close or plugin unload the in-flight resource reads abort via the shared controller; no new lifecycle hook is introduced.
- [Tech stack — Agent / Tool / Skill / MCP Wiring](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) — uses `@modelcontextprotocol/sdk` client's resource primitives already loaded by F51; picker itself is plain React 18 over Obsidian's `ItemView` children.
- [Tech stack — UI Layer](../../../../standards/tech-stack.md#ui-layer) — the picker and staged chip row are React 18 components styled with Obsidian CSS variables; no external icon fonts per [FR-UI-11](../../context.md#fr-ui-11).
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — resource content is read via the SDK client, never the Node filesystem; no `app.vault.adapter` usage added.
- [Code style — Async & Concurrency](../../../../standards/code-style.md#async--concurrency) — resource reads fan out via `Promise.allSettled` under the turn's `AbortSignal`; partial success is the normal case.
- [Code style — Error Handling](../../../../standards/code-style.md#error-handling) — picker surface translates SDK / transport errors into typed results; no throws escape the submit path.
- [Code style — Logging](../../../../standards/code-style.md#logging) — `mcp.resource.*` events follow the `mcp.*` field shape fixed by F51; payloads and URIs remain below `debug` for non-OK branches to avoid leaking server URLs at `info`.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — the F51 stdio fixture server is extended with a couple of canned resources; SSE path covered via `msw`.
- [Best practices — Make It Observable](../../../../standards/best-practices.md#core-principles) — every user-visible resource fetch emits a structured event pair (start-or-read.ok / read.err) so the picker UX and the log line stay in lockstep.

## Open questions

- Confirmation policy for resource reads — [FR-MCP-07](../../context.md#fr-mcp-07) mandates confirmation for MCP **tool calls**; FR-MCP-08 is silent on resources. Proposing no per-resource confirmation here since the user explicitly picks each URI, but a vault-wide setting to require confirmation on first resource from a server could be added by F55. Verifier to accept or specify.
- Binary / large-payload resources — MCP `resources/read` can return `blob` (base64 bytes). Proposing we attach images as the existing image content-block shape used by [F49](../attachments-images-files/feature.md) when the MIME type is recognised; other binaries are rejected with a user-visible message and a `mcp.resource.read.err` log, rather than silently dropped. Verifier to confirm.
- Preamble wording and marker format — the "MCP resources" preamble that prefixes the content blocks affects prompt determinism. Proposing a stable sentinel such as `<mcp-resources>\n<resource uri="…" server="…" mimeType="…">…</resource>\n</mcp-resources>` so [F42 compaction-microcompact](../compaction-microcompact/feature.md) / [F43 compaction-autocompact](../compaction-autocompact/feature.md) can detect and preserve it. Verifier to accept or pin an alternative shape.
- Interaction with [F22 skills-picker-active-skill](../skills-picker-active-skill/feature.md) `allowedTools` — MCP resources are not tools, so `allowedTools` does not gate them; a skill that wants to restrict MCP resource usage has no knob today. Out of scope here; flagging for a possible skill-level `allowedResources` in a later slice.
