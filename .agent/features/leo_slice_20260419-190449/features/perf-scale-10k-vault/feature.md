# F50 — 10k+ vault perf polish

## Purpose

Close the Phase 5 "10k+ vault perf optimization" bullet from [srs.md §6 Roadmap](../../../../srs/srs.md) with a profiling-and-tuning pass that proves — and, where necessary, restores — the three scale-tied NFR budgets already claimed upstream at a realistic 10 000-markdown-note vault: [NFR-PERF-02](../../context.md#nfr-perf-02) (indexing off main thread / editor stays responsive), [NFR-PERF-03](../../context.md#nfr-perf-03) (RAG queries ≤ 200 ms), and [NFR-PERF-04](../../context.md#nfr-perf-04) (initial index resumable across Obsidian restarts). The slice ships no new end-user surface and no new observable command — its deliverables are a reproducible 10 000-note perf-bench harness (fixture generator + Vitest bench target + CI gate), a one-time profiling report committed to the repo, and the minimum set of in-place tunings (bench-budget adjustments, batch sizes, heap thresholds, `requestIdleCallback` yield cadence, `VectorStore.getAll()` access pattern, graph-cache warm-up order) needed to keep [F27 vault-indexer-dirty-queue](../vault-indexer-dirty-queue/feature.md), [F31 rag-cosine-search](../rag-cosine-search/feature.md), [F34 graph-cache-symmetric](../graph-cache-symmetric/feature.md), and [F35 graphrag-boosts](../graphrag-boosts/feature.md) inside their respective budgets at scale. Any deeper structural swap — `hnswlib-wasm` per [tech stack — Retrieval Layer](../../../../standards/tech-stack.md#retrieval-layer) "Naive in-memory cosine … → `hnswlib-wasm` (phase 5 if slow)" and [architecture §8 Extension Points — Vector store swap](../../../../architecture/architecture.md#8-extension-points), worker-thread indexer offload, or a graph-cache representation change — is explicitly **conditional** on the profiling report triggering it; by default this feature ships fixtures + gates + tuning only.

## Scope

### In scope

- **10 k-note fixture generator** per [code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw): a deterministic `tests/perf/fixtures/make10kVault.ts` pure factory producing 10 000 synthetic markdown notes (heading tree, frontmatter tags, inline `[[wikilinks]]` at realistic fan-out) plus matching `fake-indexeddb`-seeded `vectors` rows (1024-dim `Float32`-backed `number[]`) and `GraphCache` adjacency; a seeded RNG guarantees run-to-run reproducibility per [best practices — Core Principles](../../../../standards/best-practices.md#core-principles). Consumed by every bench below.
- **RAG hot-path bench** at `tests/perf/ragEngine.bench.ts` asserting [F31 rag-cosine-search](../rag-cosine-search/feature.md) + [F35 graphrag-boosts](../graphrag-boosts/feature.md) stay ≤ 200 ms p50 and ≤ 400 ms p95 over 50 queries per [NFR-PERF-03](../../context.md#nfr-perf-03); extends the existing [F31](../rag-cosine-search/feature.md) single-run budget to p50/p95 distributions and covers the boost-enabled path omitted there. Emits `perf.rag.p50` / `perf.rag.p95` events via [F01 Logger](../plugin-bootstrap-logging/feature.md) per [NFR-LOG-04](../../context.md#nfr-log-04).
- **Indexer-responsiveness bench** at `tests/perf/vaultIndexer.bench.ts` validating [F27 vault-indexer-dirty-queue](../vault-indexer-dirty-queue/feature.md) + [F28 chunking-metadata](../chunking-metadata/feature.md) + [F29 embeddings-indexeddb-store](../embeddings-indexeddb-store/feature.md) stay off the hot path at scale per [NFR-PERF-02](../../context.md#nfr-perf-02): the `requestIdleCallback`-chunked loop yields at every file boundary, no single `processDueWork` tick blocks longer than a configurable `INDEX_YIELD_BUDGET_MS = 16 as const` (≈1 frame), and a `performance.mark` before/after the bench proves the fake main-thread is never starved > 16 ms across the 10 000-note drain.
- **Resumability bench** at `tests/perf/resumability.bench.ts` per [NFR-PERF-04](../../context.md#nfr-perf-04): seed 10 000 paths into the `<vault>/.leo/index/queue.json` mirror via [F27](../vault-indexer-dirty-queue/feature.md), simulate `Plugin.onunload` mid-drain at random tick boundaries (100×), assert every simulated restart reloads the unfinished paths and drains the remainder with zero duplicate embedding calls and zero lost entries — `seen = seenBeforeShutdown ∪ seenAfterRestart` equals the seeded set exactly.
- **Graph-cache warm-up bench** at `tests/perf/graphCache.bench.ts` per [NFR-PERF-02](../../context.md#nfr-perf-02): the initial symmetric adjacency build from [F34 graph-cache-symmetric](../graph-cache-symmetric/feature.md) against the 10 000-note fixture completes within a configurable `GRAPH_WARMUP_BUDGET_MS = 500 as const` on the `metadataCache.on('resolved')` hook and never blocks editor ticks > 16 ms.
- **Profiling report** committed to `tests/perf/REPORT.md` capturing the baseline p50 / p95 for every bench above, the top-5 flame-graph hotspots (Node `--cpu-prof` + `node --inspect-brk` harness), and the decision log for each tuning candidate (accepted / deferred / rejected). One-time artefact; regenerated when benches regress.
- **In-place tunings** (no new public API): any mix of `DEFAULT_TOP_K` heap allocation avoidance in [F31](../rag-cosine-search/feature.md)'s min-heap, batched `VectorStore.getAll()` streaming if the one-shot read dominates, `requestIdleCallback` yield cadence adjustment in [F27](../vault-indexer-dirty-queue/feature.md), embedding batch size in [F29](../embeddings-indexeddb-store/feature.md), graph-cache warm-up ordering relative to the first `VaultIndexer.init()` tick — each change guarded by a bench before/after delta ≥ 10 % (accept) or ≥ 5 % (reject unless paired with another win).
- **CI gate** via a new `pnpm bench` npm script per [tech stack — Runtime & Build](../../../../standards/tech-stack.md#runtime--build) driving all four benches under Vitest's bench runner; a `perf` GitHub Actions job (or equivalent CI job) runs it on every PR and fails when any budget breaches its `2×` upper bound (CI-only relaxed multiplier per [F31 open question](../rag-cosine-search/feature.md)); nominal budgets remain the SRS figures.
- **Structured log events** per [NFR-LOG-04](../../context.md#nfr-log-04) via the [F01 Logger](../plugin-bootstrap-logging/feature.md): `perf.fixture.seeded` (`{notes, vectors, edges, ms}`), `perf.rag.p50` / `perf.rag.p95` (`{ms, runs}`), `perf.indexer.yield-violation` (`{tickMs}` when > 16 ms), `perf.resumability.restart` (`{before, after, duplicates}`), `perf.graph.warmup.ms`; no note content, no query text above `debug` per [NFR-LOG-01](../../context.md#nfr-log-01) and [code style — Logging](../../../../standards/code-style.md#logging).
- **Vitest bench-runner integration** per [code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw): benches use `vitest bench` (not `test`) so they live separately from the unit suite, report deterministic p50/p95, and are gated off the default `pnpm test` run to keep local TDD fast; only `pnpm bench` / CI executes them.

### Out of scope

- `hnswlib-wasm` (or any ANN) swap behind the `VectorStore` interface — reserved behind the same [architecture §8 Extension Points — Vector store swap](../../../../architecture/architecture.md#8-extension-points) seam and activated **only if** the RAG bench fails the 200 ms budget per [tech stack — Retrieval Layer](../../../../standards/tech-stack.md#retrieval-layer) "phase 5 if slow" trigger and [tech stack — perf note 151](../../../../standards/tech-stack.md) "measure at 5 k notes. If p95 RAG > 200 ms, swap to HNSW." When triggered, it ships as its own follow-up feature, not inside F50.
- Worker-thread offloading of the indexer or embedding client — deferred alongside the HNSW swap; the bench-driven decision record lives in `tests/perf/REPORT.md` but no runtime code lands here unless a budget fails.
- Any change to the [F31 rag-cosine-search](../rag-cosine-search/feature.md) public `RAGEngine.query` contract, the [F35 graphrag-boosts](../graphrag-boosts/feature.md) `Scorer.applyBoosts` signature, the [F34 graph-cache-symmetric](../graph-cache-symmetric/feature.md) `GraphCache` surface, or the [F27 vault-indexer-dirty-queue](../vault-indexer-dirty-queue/feature.md) `VaultIndexer` API — tunings are internal-only per [best practices — Core Principles](../../../../standards/best-practices.md#core-principles) (open–closed).
- End-user UX for perf reporting — no `/perf` command, no settings row, no status-bar widget; the report is a dev-facing markdown file and the gate is CI-side only.
- Scale beyond 10 000 notes (100 k / 1 M vault) — explicitly named in [F31 in-scope](../rag-cosine-search/feature.md) out-of-scope ("1 M-row scale"). F50 owns only the 10 k envelope.
- Reindex command polish — the adjacent Phase 5 bullet in [srs.md §6 Roadmap](../../../../srs/srs.md) ships as its own feature / row, not here.
- Network-, provider-, and chat-turn-latency measurement (streaming throughput, `/v1/embeddings` wall-clock) — NFR-PERF-05 / NFR-PERF-06 / NFR-PERF-07 are owned by their respective features; F50 only covers the three scale-tied NFR-PERF budgets listed above.

## Acceptance criteria

1. `tests/perf/fixtures/make10kVault.ts` produces a deterministic 10 000-note markdown fixture plus matching `fake-indexeddb` `vectors` rows (1024-dim) and `GraphCache` adjacency; two invocations with the same seed yield byte-identical outputs and a Vitest fixture asserts the `(notes, vectors, edges)` counts are all exactly 10 000 / ≥ 10 000 / ≥ 10 000. ([NFR-PERF-03](../../context.md#nfr-perf-03), [NFR-PERF-02](../../context.md#nfr-perf-02))
2. `tests/perf/ragEngine.bench.ts` runs 50 [F31](../rag-cosine-search/feature.md) `RAGEngine.query` calls (25 with `activeNotePath` + `activeNoteTags` to exercise the [F35](../graphrag-boosts/feature.md) boost path, 25 without), computes p50 and p95 via `performance.now()`, asserts p50 ≤ 200 ms and p95 ≤ 400 ms, and emits `perf.rag.p50` + `perf.rag.p95` events through [F01 Logger](../plugin-bootstrap-logging/feature.md). ([NFR-PERF-03](../../context.md#nfr-perf-03))
3. `tests/perf/vaultIndexer.bench.ts` drains the full 10 000-note dirty queue through [F27](../vault-indexer-dirty-queue/feature.md) `processDueWork` and asserts no single `requestIdleCallback` tick runs > `INDEX_YIELD_BUDGET_MS = 16 as const`; a tick breach emits `perf.indexer.yield-violation` with the offending `tickMs` and fails the bench. ([NFR-PERF-02](../../context.md#nfr-perf-02))
4. `tests/perf/resumability.bench.ts` simulates 100 random mid-drain `Plugin.onunload` + restart cycles on the 10 000-note fixture through [F27](../vault-indexer-dirty-queue/feature.md)'s `<vault>/.leo/index/queue.json` persistence and asserts every run ends with `seenBeforeShutdown ∪ seenAfterRestart` equal to the seeded set exactly, zero duplicate embedding calls, zero lost entries. ([NFR-PERF-04](../../context.md#nfr-perf-04))
5. `tests/perf/graphCache.bench.ts` drives the [F34](../graph-cache-symmetric/feature.md) symmetric adjacency warm-up on the 10 000-note fixture and asserts total warm-up wall clock ≤ `GRAPH_WARMUP_BUDGET_MS = 500 as const` with no main-thread tick > 16 ms. ([NFR-PERF-02](../../context.md#nfr-perf-02))
6. A `pnpm bench` npm script runs all four benches under `vitest bench`; CI runs it on every PR and fails when any bench exceeds its SRS budget × 2 (CI-only relaxed upper bound); nominal budgets remain the raw SRS figures from AC 2–5. ([NFR-PERF-02](../../context.md#nfr-perf-02), [NFR-PERF-03](../../context.md#nfr-perf-03), [NFR-PERF-04](../../context.md#nfr-perf-04))
7. `tests/perf/REPORT.md` is committed with baseline p50 / p95 for every bench, top-5 hotspot flame-graph entries, and a decision row per tuning candidate (accepted / deferred / rejected, with delta %); when any bench regresses the report is regenerated and re-committed. ([NFR-PERF-02](../../context.md#nfr-perf-02), [NFR-PERF-03](../../context.md#nfr-perf-03), [NFR-PERF-04](../../context.md#nfr-perf-04))
8. No public API of [F27](../vault-indexer-dirty-queue/feature.md) / [F31](../rag-cosine-search/feature.md) / [F34](../graph-cache-symmetric/feature.md) / [F35](../graphrag-boosts/feature.md) changes; a Vitest type-compat snapshot asserts the exported surfaces are byte-identical before and after F50 lands. ([NFR-PERF-02](../../context.md#nfr-perf-02), [NFR-PERF-03](../../context.md#nfr-perf-03))
9. When any AC 2–5 bench fails its budget, the report explicitly records the HNSW / worker-thread swap decision per [architecture §8 Extension Points — Vector store swap](../../../../architecture/architecture.md#8-extension-points) and [tech stack — Retrieval Layer](../../../../standards/tech-stack.md#retrieval-layer) "phase 5 if slow"; the swap itself is deferred to a follow-up feature and does not land in F50. ([NFR-PERF-03](../../context.md#nfr-perf-03))

## Dependencies

- [F31 rag-cosine-search](../rag-cosine-search/feature.md) — supplies the `RAGEngine.query` hot path, `Scorer.cosine`, `VectorStore.getAll()` one-shot scan, and the existing 10 k-row single-run bench that F50 extends to p50 / p95 distributions and composes with the [F35](../graphrag-boosts/feature.md) boost path. F50 is the downstream "perf polish swapping in HNSW behind the same `RAGEngine` contract" row already reserved in [F31 `## Dependencies`](../rag-cosine-search/feature.md).
- [F35 graphrag-boosts](../graphrag-boosts/feature.md) — supplies the boost-enabled hot path (`GraphTraversal.neighbors1h/neighbors2h`, `Scorer.applyBoosts`) that the RAG bench must exercise alongside raw cosine; the [F35 `## Dependencies`](../graphrag-boosts/feature.md) explicitly names F50 as the "keeps boost pass under the ≤ 200 ms budget at scale" downstream consumer.
- [F27 vault-indexer-dirty-queue](../vault-indexer-dirty-queue/feature.md) — owns `DirtyQueue`, the `requestIdleCallback`-chunked drain loop (the [NFR-PERF-02](../../context.md#nfr-perf-02) off-main-thread contract), and the `<vault>/.leo/index/queue.json` persistence mirror (the [NFR-PERF-04](../../context.md#nfr-perf-04) resumability contract); F50 benches both invariants without touching the API.
- [F34 graph-cache-symmetric](../graph-cache-symmetric/feature.md) — supplies the `GraphCache` warm-up path measured by the graph-cache bench; F34's symmetric adjacency build is the scale-sensitive work F50 budgets against [NFR-PERF-02](../../context.md#nfr-perf-02).
- [F28 chunking-metadata](../chunking-metadata/feature.md), [F29 embeddings-indexeddb-store](../embeddings-indexeddb-store/feature.md) — the indexer bench exercises chunking + embedding + IndexedDB write through them; F50 touches neither API.
- [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md) — provides the `Logger` surface for `perf.*` structured events per [NFR-LOG-04](../../context.md#nfr-log-04).
- Drives no net-new FR (per [features-index.md "Notes on coverage edges"](../../features-index.md) — "F50 … perf polish exposes no net-new requirement IDs"); upholds [NFR-PERF-02](../../context.md#nfr-perf-02), [NFR-PERF-03](../../context.md#nfr-perf-03), [NFR-PERF-04](../../context.md#nfr-perf-04) at scale; references [srs.md §6 Roadmap Phase 5](../../../../srs/srs.md) "10k+ vault perf optimization".

## Implementation notes

- [Architecture §8 Extension Points — Vector store swap](../../../../architecture/architecture.md#8-extension-points) fixes the `VectorStore` interface as the swap seam for any HNSW follow-up; F50's benches are written against the interface, not the in-memory implementation.
- [Architecture §3.4 Adapters — VectorStore](../../../../architecture/architecture.md#34-adapters) and [§3.3 Domain / Core — Scorer](../../../../architecture/architecture.md#33-domain--core-pure) pin the measurement boundaries: benches wrap adapter calls and time pure scoring separately.
- [Architecture §5.2 Chat Turn (no tools)](../../../../architecture/architecture.md#52-chat-turn-no-tools) places RAG on the hot path — the ≤ 200 ms envelope is enforced end-to-end, not per component.
- [Architecture §10 Concurrency & Lifecycle Rules](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) governs `AbortController` propagation through every bench so mid-run teardown exits cleanly.
- [Tech stack — Retrieval Layer](../../../../standards/tech-stack.md#retrieval-layer) pins the "linear scan handles 10 k × 1024-dim fine" claim F50 validates and the "phase 5 if slow" trigger for the HNSW follow-up.
- [Tech stack — Runtime & Build](../../../../standards/tech-stack.md#runtime--build) governs the `pnpm bench` script and Vitest bench runner choice.
- [Code style — TypeScript](../../../../standards/code-style.md#typescript) — `INDEX_YIELD_BUDGET_MS = 16 as const`, `GRAPH_WARMUP_BUDGET_MS = 500 as const`, `RAG_P50_BUDGET_MS = 200 as const`, `RAG_P95_BUDGET_MS = 400 as const` literals.
- [Code style — Async & Concurrency](../../../../standards/code-style.md#async--concurrency) — `AbortSignal` threaded through every bench and yield cadence respected.
- [Code style — Logging](../../../../standards/code-style.md#logging) — `perf.*` events ship counts and milliseconds only; note content, query text, and paths stay at `debug`.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — `vitest bench` (not `test`) for all perf harnesses; `fake-indexeddb` backs the vector store fixture; benches are excluded from the default `pnpm test` run.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — single-responsibility (measure + tune only), open-closed (no API changes), observability (`perf.*` events on every bench).
- [srs.md §4.1 Performance](../../../../srs/srs.md) — authoritative budgets for [NFR-PERF-02](../../context.md#nfr-perf-02) / [NFR-PERF-03](../../context.md#nfr-perf-03) / [NFR-PERF-04](../../context.md#nfr-perf-04); F50 does not restate them, only links.
- [srs.md §6 Roadmap Phase 5](../../../../srs/srs.md) — "10k+ vault perf optimization" bullet; F50 is the slice that cashes it in.

## Open questions

- **CI bench stability multiplier** — inherits the [F31 rag-cosine-search open question](../rag-cosine-search/feature.md) on `performance.now()` noise on shared runners. AC 6 proposes `2×` budget on CI only. Verifier to confirm the multiplier and whether the `perf` job should run on a dedicated self-hosted runner instead of GitHub-hosted for stability per [NFR-PERF-03](../../context.md#nfr-perf-03).
- **HNSW swap trigger threshold** — [tech stack perf note](../../../../standards/tech-stack.md) says "measure at 5 k notes. If p95 RAG > 200 ms, swap." F50 measures at 10 k (per [NFR-PERF-03](../../context.md#nfr-perf-03)). AC 2 sets p95 ≤ 400 ms at 10 k as the "stay-in-linear-scan" ceiling. Verifier to reconcile the 5 k / 10 k measurement points and confirm the 400 ms p95 gate is right, or tighten to 200 ms p95 (matching the SRS p50 budget) and accept earlier HNSW activation.
- **`Plugin.onunload` mid-drain simulation realism** — AC 4's 100-cycle random-interrupt loop is an abstraction over real Obsidian shutdowns. Verifier to decide whether an in-app Playwright/E2E smoke on real Obsidian is also required (owned by [F57](../../features-index.md) per [features-index.md NFR-TEST mapping](../../features-index.md)), or whether the Vitest simulation is sufficient alone for [NFR-PERF-04](../../context.md#nfr-perf-04).
- **`perf.*` event cardinality** — proposed events fire per-bench-run; in CI the volume is bounded, but if a dev runs `pnpm bench --watch` locally the log channel could flood at `info`. Verifier to confirm default level (proposal: `info` for p50/p95 summary, `debug` for per-sample events).
- **Report regeneration cadence** — AC 7 regenerates `tests/perf/REPORT.md` "when any bench regresses"; is that every PR that touches F27/F31/F34/F35 internals, or only on explicit tuning PRs? Proposing "on every bench-budget-breach regression-fix PR"; verifier to confirm.
- **Overlap with F57 (test harness / coverage)** — per [features-index.md NFR-TEST mapping](../../features-index.md), NFR-TEST-03/04 map to F57. Is the `pnpm bench` CI job owned by F50 (perf-specific) or F57 (general test harness)? Proposing F50 owns the bench target and CI job definition, F57 owns the unit test infrastructure; verifier to confirm the split.
