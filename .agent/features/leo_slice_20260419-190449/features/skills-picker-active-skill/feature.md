# F22 — Skill picker & active-skill application

## Purpose

Turn the passive [F21 `SkillsStore`](../skills-loader-builtin/feature.md) catalogue into a live per-thread binding: render a `SkillPicker` inside [F04](../chat-sidebar-view/feature.md)'s `HeaderBar` (dropdown + "Leo: Select skill" command palette entry) so the user can choose the active skill for the current thread and see its name on the header badge per [FR-CHAT-12](../../context.md#fr-chat-12), persist `thread.metadata.skillId` through [F14](../conversation-persistence-v1/feature.md) so the default "General" is applied per [FR-SKILL-05](../../context.md#fr-skill-05), and treat mid-thread switches as take-effect-next-turn — the new `systemPrompt` is consumed by the `AgentRunner`'s next prompt build while prior turns stay verbatim on disk per [FR-SKILL-06](../../context.md#fr-skill-06). When the bound `Skill` declares `allowedTools`, [F16](../tool-registry-builtin-read/feature.md)'s `ToolRegistry.listFor(thread)` filters its catalogue to that allowlist before the OpenAI-compatible `tools` array is handed to the provider per [FR-SKILL-07](../../context.md#fr-skill-07) / [FR-AGENT-12](../../context.md#fr-agent-12); when it declares `defaultModel`, that id overrides the globally-configured chat model for every provider call originating on this thread per [FR-SKILL-08](../../context.md#fr-skill-08). This slice wires the UI, the thread-metadata binding, and the two registry/provider hooks — nothing more.

## Scope

### In scope

- `SkillPicker` React component mounted in [F04](../chat-sidebar-view/feature.md)'s `HeaderBar` region: Obsidian-themed dropdown (`role="listbox"` with `role="option"` rows) listing `SkillsStore.list()` sorted by name, each row showing `skill.name` + one-line `description`; selected skill's `name` rendered as a `HeaderBar` badge (truncated with `title` on overflow, collapsed below 280px per [NFR-USE-09](../../context.md#nfr-use-09)) per [FR-CHAT-12](../../context.md#fr-chat-12).
- "Leo: Select skill" command palette entry opening the same picker (palette path + dropdown path converge on one handler) per [FR-UI-04](../../context.md#fr-ui-04); keyboard reachable per [NFR-USE-05](../../context.md#nfr-use-05), visible Obsidian focus ring on the trigger.
- Per-thread active-skill state: `ConversationStore.thread.metadata.skillId: string` (default `"general"`) persisted via [F14](../conversation-persistence-v1/feature.md); on thread load the stored id is resolved through `SkillsStore.get(id)` with fallback to `"general"` if the id is missing / its file was deleted per [FR-SKILL-05](../../context.md#fr-skill-05).
- Mid-thread switch semantics: picking a new skill calls `ConversationStore.mutate(setSkillId)` and emits a `thread.skill.changed` event; the `AgentRunner` re-reads `thread.metadata.skillId` on its **next** `send(msg, thread)` so the new `systemPrompt` feeds the next prompt build only — previously persisted assistant / user turns are untouched per [FR-SKILL-06](../../context.md#fr-skill-06).
- `ToolRegistry.listFor(thread)` filter implementation: reads the thread's bound `Skill`, and if `skill.allowedTools` is defined returns only specs whose `id` is in that list; if undefined returns the full registry (default skill = all tools) per [FR-SKILL-07](../../context.md#fr-skill-07) / [FR-AGENT-12](../../context.md#fr-agent-12). The pass-through hook [F16](../tool-registry-builtin-read/feature.md) already exposes is now backed by this filter.
- `defaultModel` override: when the bound `Skill.defaultModel` is defined, `AgentRunner` passes it to `ProviderManager.stream(prompt, {signal, tools, model})` for this thread's next turn, overriding the global chat model from settings; absent `defaultModel` falls back to the settings default per [FR-SKILL-08](../../context.md#fr-skill-08).
- Structured log events `skill.picker.open`, `skill.select` (`{threadId, fromId, toId}`), `skill.filter.applied` (`{threadId, allowedToolsCount}`), `skill.model.override` (`{threadId, model}`) via the F01 `Logger` per [NFR-LOG-04](../../context.md#nfr-log-04); Vitest coverage for picker mount/select, metadata persistence, next-turn-only effect, `allowedTools` filter, `defaultModel` pass-through, and fallback on missing id.

### Out of scope

- MCP prompts surfaced in the picker as a separate "From MCP" section — ships with [F54 mcp-prompts-in-skills](../../features-index.md) (and resolves the `defaultModel` collision open question).
- In-plugin skill editor UI (create / edit / delete / duplicate from settings or dedicated view) — ships with [F39 skill-editor-ui](../../features-index.md); this feature only selects from the catalogue.
- Loading + parsing + bundled built-in skills + FS watch + Zod validation — owned by [F21 skills-loader-builtin](../skills-loader-builtin/feature.md); this feature consumes `SkillsStore` read-only.

## Acceptance criteria

1. The `SkillPicker` mounts in [F04](../chat-sidebar-view/feature.md)'s `HeaderBar` region as an Obsidian-themed dropdown listing every entry from `SkillsStore.list()` sorted by `name`, each row showing `name` + `description`; a "Leo: Select skill" command palette entry opens the same picker. ([FR-CHAT-12](../../context.md#fr-chat-12))
2. The currently-selected skill's `name` is rendered as a `HeaderBar` badge (`role="status"` friendly, `aria-label="Active skill: <name>"`); on a fresh thread the default `skillId = "general"` resolves through `SkillsStore.get("general")` and the "General" name is shown. ([FR-CHAT-12](../../context.md#fr-chat-12), [FR-SKILL-05](../../context.md#fr-skill-05))
3. Selecting a new skill writes `thread.metadata.skillId` through `ConversationStore.mutate(setSkillId)` (persisted by [F14](../conversation-persistence-v1/feature.md)); reloading the plugin restores the same bound skill on the thread. ([FR-SKILL-05](../../context.md#fr-skill-05))
4. Switching skill mid-thread does **not** modify any prior persisted message; the new `skill.systemPrompt` is applied on the **next** `AgentRunner.send(msg, thread)` only, verified by asserting the assembled prompt on turn N+1 contains the new `systemPrompt` and turn N's stored content is byte-identical. ([FR-SKILL-06](../../context.md#fr-skill-06))
5. When the bound `Skill.allowedTools` is defined, `ToolRegistry.listFor(thread)` returns exactly the specs whose `id` is in that list and the OpenAI-compatible `tools` array passed to `ProviderManager.stream` is correspondingly filtered; when `allowedTools` is undefined the full registry is returned (default skill = all tools). ([FR-SKILL-07](../../context.md#fr-skill-07), [FR-AGENT-12](../../context.md#fr-agent-12))
6. When the bound `Skill.defaultModel` is defined, `AgentRunner` passes it as the `model` option on the next `ProviderManager.stream` call, overriding the settings default; when undefined the settings default is used. ([FR-SKILL-08](../../context.md#fr-skill-08))
7. If `thread.metadata.skillId` resolves to no `Skill` (user deleted the file), the picker falls back to `"general"`, persists the correction, and emits `skill.select` with `fromId: <missing>` + `toId: "general"` via the F01 `Logger`. ([FR-SKILL-05](../../context.md#fr-skill-05))
8. Vitest suite covers: picker mount + dropdown list shape, command-palette entry wiring, selection persists `skillId` through [F14](../conversation-persistence-v1/feature.md), next-turn-only prompt effect, `allowedTools` filter on `listFor(thread)` + `tools` payload, `defaultModel` pass-through to `ProviderManager.stream`, fallback on missing skill id, structured log-event shape. ([FR-CHAT-12](../../context.md#fr-chat-12), [FR-SKILL-05](../../context.md#fr-skill-05), [FR-SKILL-06](../../context.md#fr-skill-06), [FR-SKILL-07](../../context.md#fr-skill-07), [FR-SKILL-08](../../context.md#fr-skill-08), [FR-AGENT-12](../../context.md#fr-agent-12))

## Open questions

- **Skill mid-thread change vs autocompact summary recall (global [open questions](../../context.md#open-questions))** — not resolved here; when autocompact ships (F43) it must decide whether the summarization system prompt reflects the skill at summary time or the currently-bound skill. This feature ships the "next-turn-only" rule for chat turns; autocompact ordering is deferred.
- **`defaultModel` override vs MCP-prompt skills (global [open questions](../../context.md#open-questions))** — MCP prompts enter the catalogue via [F54](../../features-index.md). This feature treats `defaultModel` as a flat per-skill override; collision rules (MCP prompt + thread skill both set `defaultModel`) resolve at F54 when the second source appears.
- **Reindex-on-model-switch interaction with skill `defaultModel` ([F27](../../features-index.md))** — if `defaultModel` names a different embedding model than the global setting, does switching skill trigger a reindex prompt? Flagged for F27; this feature only routes the chat `model`, not the embedding model (separately configured per [FR-PROV-08](../../context.md#fr-prov-08)).
