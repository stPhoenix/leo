# F24 — Plan mode controller & permission gate

## Purpose

Deliver the plan-mode controller that layers on top of [F23 plan-files-todos-store](../plan-files-todos-store/feature.md) and [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) to make plan-mode a first-class, permission-enforced mode — not a prompt hint. The controller exposes `EnterPlanMode` / `ExitPlanMode` tools on the [F16 ToolRegistry](../tool-registry-builtin-read/feature.md) and forbids them in subagent contexts per [FR-PLAN-04](../../context.md#fr-plan-04); flips a permission-system flag (`mode: "normal" | "plan"`) that blocks every write-capable tool while `mode === 'plan'` — only `Read`, `Grep`, `Glob`, `WebFetch`, and the `plan-file-write` path through `PlanStore.writePlan` are permitted — enforced in the permission layer pre-invoke, never by prompt per [FR-PLAN-05](../../context.md#fr-plan-05) and [NFR-REL-07](../../context.md#nfr-rel-07); queues a mode-transition attachment wrapped in `<system-reminder>` tags for injection on the next turn on each `EnterPlanMode` / `ExitPlanMode` event with the rapid-toggle "opposing-flag clearing" rule so a quick enter→exit (or exit→enter) cancels a still-pending opposite reminder and nothing ships per [FR-PLAN-08](../../context.md#fr-plan-08); and injects the rate-limited stale-todo reminder at turn boundaries (only when `TodoStore` has items, last reminder was more than N messages ago, and the model did non-trivial work without calling `TodoWrite`) wrapped in `<system-reminder>` tags and never surfaced to the user per [FR-PLAN-03](../../context.md#fr-plan-03) and plan.md §3.8. This slice is the permission spine plan approval and session resume stack on.

## Scope

### In scope

- `PlanModeController` module (agent layer per [architecture §3.2](../../../../architecture/architecture.md#32-agent-layer)) owning the in-memory `mode: "normal" | "plan"` flag per thread, transition handlers, the pending-transition-attachment queue, and the stale-todo tracking state; lives beside [F23](../plan-files-todos-store/feature.md)'s `PlanStore` / `TodoStore` and the [F17 ConfirmationController](../tool-confirmation-flow/feature.md) on the same agent-layer seam.
- `EnterPlanMode` and `ExitPlanMode` `ToolSpec`s registered with the [F16 ToolRegistry](../tool-registry-builtin-read/feature.md) per [FR-PLAN-04](../../context.md#fr-plan-04); both tools check `ctx.thread.agentId == null` (main-agent only) and fail with a typed `PlanModeForbiddenInSubagent` error when invoked from a subagent context, logged as `plan.mode.subagent-reject` via the [F01 Logger](../plugin-bootstrap-logging/feature.md); `ExitPlanMode` takes `{plan: string}` input and writes it through `PlanStore.writePlan(plan)` before transitioning back to `normal` (approval dialog itself is [F25](../../features-index.md)'s job — this slice only performs the write and flag flip).
- Permission-system gate in [F10 AgentRunner](../agent-controller-core/feature.md)'s serial turn loop, sitting before [F17 ConfirmationController](../tool-confirmation-flow/feature.md): when the controller's flag reads `mode === 'plan'`, `ToolRegistry.lookup(id)` is filtered through a plan-mode allowlist (`{Read, Grep, Glob, WebFetch, EnterPlanMode, ExitPlanMode}` plus the `plan-file-write` path — the module-level `PlanStore.writePlan` API exposed by [F23](../plan-files-todos-store/feature.md)); any other tool id resolves to a typed `PlanModeBlocked` tool-error (`"blocked by plan mode: <toolId>"`) fed back into the graph per [architecture §7](../../../../architecture/architecture.md#7-error-handling-strategy), with no prompt-level plea or `requiresConfirmation` prompt shown — enforcement is permission-system, not prompt, per [NFR-REL-07](../../context.md#nfr-rel-07) and [FR-PLAN-05](../../context.md#fr-plan-05).
- Subagent prohibition: in addition to the tool-level guard above, the same `agentId == null` check applies at transition time; any attempt to set `mode = 'plan'` (or call `EnterPlanMode` / `ExitPlanMode`) from a subagent-flagged `ToolCtx` is rejected with `PlanModeForbiddenInSubagent` and `plan.mode.subagent-reject` logged per [FR-PLAN-04](../../context.md#fr-plan-04); flagged as vacuous today since Leo has no subagent runtime but wired so the guard is already in place when [context.md open questions](../../context.md#open-questions) "Plan mode in subagent contexts" is resolved.
- Mode-transition attachments queue: each transition pushes a `PendingReminder` of kind `plan-enter` | `plan-exit` wrapped in `<system-reminder>…</system-reminder>` tags into a per-thread FIFO; on the next `AgentRunner.send` the head of the queue is prepended to the outgoing message stream and the queue is drained per [FR-PLAN-08](../../context.md#fr-plan-08); rapid-toggle opposing-flag clearing: when a new transition is queued and the queue tail is its opposite (`plan-enter` vs `plan-exit`), both are dropped so zero reminders ship for the round-trip, matching plan.md §6 and [context.md FR-PLAN-08 "opposing-flag rule"](../../context.md#fr-plan-08).
- Stale-todo reminder rate-limiter: `PlanModeController.maybeInjectStaleTodoReminder(thread, lastAction)` runs at each turn boundary inside [F10](../agent-controller-core/feature.md); it emits a `<system-reminder>`-wrapped reminder only when all of: (a) `TodoStore.get(thread.agentId ?? thread.id)` is non-empty, (b) `messagesSinceLastReminder >= N` (N from settings, default 10), (c) the most recent assistant turn performed "non-trivial work" (at least one `tool_call` event) without a `TodoWrite` invocation per plan.md §3.8; the reminder is never surfaced to the user (model-only message stream) per [FR-PLAN-03](../../context.md#fr-plan-03); N and the last-reminder message-count anchor are held in-memory keyed by `thread.id` and cleared on `Plugin.onunload`.
- Structured log events through the [F01 Logger](../plugin-bootstrap-logging/feature.md) per [NFR-LOG-04](../../context.md#nfr-log-04): `plan.mode.enter`, `plan.mode.exit`, `plan.mode.subagent-reject`, `plan.mode.tool-blocked` (`{toolId, thread}` — no args payload logged), `plan.attachment.queued`, `plan.attachment.flushed`, `plan.attachment.cleared-opposing`, `plan.stale-todo.reminder`, `plan.stale-todo.suppressed` (`{reason: "empty"|"rate-limit"|"todowrite-called"}`); reminder body and plan content never logged above `debug` per [code-style — Logging](../../../../standards/code-style.md#logging).
- Vitest coverage per [NFR-TEST-01](../../context.md#nfr-test-01) and [NFR-TEST-07](../../context.md#nfr-test-07) of: permission gate blocks every non-allowlisted tool in plan mode (not prompt-only, asserted by invoking a fake `create_note` through the registry path and checking it returns `PlanModeBlocked` without the [F17](../tool-confirmation-flow/feature.md) dialog firing); allowlist passthrough for `Read` / `Grep` / `Glob` / `WebFetch` / `plan-file-write`; `EnterPlanMode` / `ExitPlanMode` refused in subagent context with the typed error + log event; `ExitPlanMode` writes `{plan}` through `PlanStore.writePlan` before flipping the flag; mode-transition attachment queued on enter/exit and flushed into the next turn wrapped in `<system-reminder>` tags; opposing-flag clearing cancels both reminders on rapid toggle; stale-todo reminder fires only when all three conditions hold, is suppressed otherwise with the correct `reason`, is rate-limited per plan.md §3.8, and is never routed to the user-visible message stream.

### Out of scope

- Plan approval dialog with Approve / Edit / Reject actions, inline modal rendering, `planWasEdited` rewrite-on-edit, and the result-message variants for approved / edited / empty plans per [FR-PLAN-07](../../context.md#fr-plan-07) / [FR-UI-09](../../context.md#fr-ui-09) → ships with [F25 plan-approval-dialog](../../features-index.md); this feature only flips the flag and writes the raw `{plan}` payload through `PlanStore.writePlan` inside `ExitPlanMode`.
- Plan / todo session resume (todo rehydration from transcript; plan-content recovery fallback chain `file_snapshot` → `ExitPlanMode` tool_use → user attachments) per [FR-PLAN-09](../../context.md#fr-plan-09) → ships with [F26 plan-session-resume](../../features-index.md); the controller's `mode` flag starts at `normal` on every plugin load and is not persisted.
- The `TodoWrite` tool itself (schema, registration, in-memory store, verbatim plan.md §3.3 prompt) per [FR-PLAN-01](../../context.md#fr-plan-01) / [FR-PLAN-02](../../context.md#fr-plan-02) → shipped by [F23 plan-files-todos-store](../plan-files-todos-store/feature.md); this feature only reads `TodoStore` to drive the stale-todo reminder.

## Acceptance criteria

1. `PlanModeController` exposes a per-thread `mode: "normal" | "plan"` flag defaulting to `"normal"`; `EnterPlanMode` flips it to `"plan"` and `ExitPlanMode` flips it back to `"normal"` after writing its `{plan: string}` input through [`PlanStore.writePlan`](../plan-files-todos-store/feature.md); log events `plan.mode.enter` and `plan.mode.exit` fire with `{threadId}` via the [F01 Logger](../plugin-bootstrap-logging/feature.md). ([FR-PLAN-04](../../context.md#fr-plan-04))
2. The permission-system gate in [F10 AgentRunner](../agent-controller-core/feature.md)'s turn loop blocks every tool invocation while `mode === 'plan'` EXCEPT the allowlist `{Read, Grep, Glob, WebFetch, EnterPlanMode, ExitPlanMode, plan-file-write}`; a blocked call returns a typed `PlanModeBlocked` tool-error `"blocked by plan mode: <toolId>"` fed back into the graph per [architecture §7](../../../../architecture/architecture.md#7-error-handling-strategy) and emits `plan.mode.tool-blocked`; a Vitest assertion confirms the [F17 ConfirmationController](../tool-confirmation-flow/feature.md) dialog is NEVER invoked along this path (enforcement is permission-system, not prompt-only). ([FR-PLAN-05](../../context.md#fr-plan-05), [NFR-REL-07](../../context.md#nfr-rel-07))
3. Both `EnterPlanMode` and `ExitPlanMode` fail with a typed `PlanModeForbiddenInSubagent` tool-error and emit `plan.mode.subagent-reject` when invoked from a `ToolCtx` whose `thread.agentId != null`; no state transition, no attachment queued, and no write through `PlanStore.writePlan` occurs on that path. ([FR-PLAN-04](../../context.md#fr-plan-04))
4. Every transition enqueues a `PendingReminder` of kind `plan-enter` or `plan-exit` (body wrapped byte-for-byte in `<system-reminder>…</system-reminder>` tags from plan.md §6) into a per-thread FIFO; on the next [F10 AgentRunner](../agent-controller-core/feature.md)`.send`, the head is prepended to the outgoing message stream and the queue drained; `plan.attachment.queued` and `plan.attachment.flushed` fire with `{threadId, kind}`. ([FR-PLAN-08](../../context.md#fr-plan-08))
5. Opposing-flag clearing: when a transition is enqueued and the queue tail holds its opposite kind, both entries are dropped (net zero reminders ship for a rapid enter→exit or exit→enter round-trip); `plan.attachment.cleared-opposing` logs `{threadId, droppedKinds}` and the next-turn message stream contains no `<system-reminder>` from this queue. ([FR-PLAN-08](../../context.md#fr-plan-08))
6. `PlanModeController.maybeInjectStaleTodoReminder(thread)` at turn boundaries injects a `<system-reminder>`-wrapped reminder into the next outgoing message stream (never the user-visible transcript) iff ALL of: (a) `TodoStore.get(thread.agentId ?? thread.id)` is non-empty, (b) messages-since-last-reminder ≥ `settings.staleTodoReminderThreshold` (default 10), (c) the most recent assistant turn produced ≥ 1 `tool_call` event without a `TodoWrite` invocation; otherwise a `plan.stale-todo.suppressed` event fires with `reason` in `{"empty", "rate-limit", "todowrite-called"}`. ([FR-PLAN-03](../../context.md#fr-plan-03))
7. Vitest suite per [NFR-TEST-01](../../context.md#nfr-test-01) covers: permission gate blocks every non-allowlisted tool with `PlanModeBlocked` and zero [F17](../tool-confirmation-flow/feature.md) dialog invocations; allowlist `{Read, Grep, Glob, WebFetch, plan-file-write}` pass-through; `EnterPlanMode` / `ExitPlanMode` refusal in subagent context with typed error + log; `ExitPlanMode` writes through `PlanStore.writePlan` before flipping the flag; attachment queued + flushed with `<system-reminder>` wrapping; opposing-flag clearing on rapid toggle; stale-todo reminder fires on all-three conditions, is rate-limited, suppressed with correct `reason`, and is never routed to the user-visible message stream. ([FR-PLAN-03](../../context.md#fr-plan-03), [FR-PLAN-04](../../context.md#fr-plan-04), [FR-PLAN-05](../../context.md#fr-plan-05), [FR-PLAN-08](../../context.md#fr-plan-08), [NFR-REL-07](../../context.md#nfr-rel-07))

## Dependencies

- [F23 plan-files-todos-store](../plan-files-todos-store/feature.md) — supplies `PlanStore.writePlan` (called by `ExitPlanMode` before flag flip), `PlanStore.currentSlug()` (read when emitting `plan.mode.exit` log payload), and `TodoStore.get(key)` (read by the stale-todo rate-limiter); this feature layers the mode flag and permission gate on top without adding persistence per [context.md FR-PLAN-06](../../context.md#fr-plan-06) / [context.md FR-PLAN-01](../../context.md#fr-plan-01).
- [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) — the permission gate sits BEFORE `ConfirmationController` in the turn loop per [architecture §5.3](../../../../architecture/architecture.md#53-chat-turn-with-tool-call--confirmation): blocked tools short-circuit to `PlanModeBlocked` tool-error without the Allow / Allow-for-thread / Deny dialog firing; the gate ordering is the load-bearing invariant for [NFR-REL-07](../../context.md#nfr-rel-07) ("not prompt-only" enforcement).
- [F10 agent-controller-core](../agent-controller-core/feature.md) — hosts the serial turn loop where the gate, the transition-attachment flush, and the stale-todo rate-limiter hook in per [architecture §5.2 / §5.3](../../../../architecture/architecture.md#52-chat-turn-no-tools).
- [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) — provides the `ToolRegistry.register` surface for `EnterPlanMode` / `ExitPlanMode` and the `ToolSpec` / `ToolResult` contract the gate returns errors through per [architecture §4](../../../../architecture/architecture.md#4-key-contracts).
- [F01 plugin-bootstrap-logging](../plugin-bootstrap-logging/feature.md) — receives the `plan.mode.*` / `plan.attachment.*` / `plan.stale-todo.*` structured log events per [NFR-LOG-04](../../context.md#nfr-log-04).
- Drives requirements [FR-PLAN-03](../../context.md#fr-plan-03), [FR-PLAN-04](../../context.md#fr-plan-04), [FR-PLAN-05](../../context.md#fr-plan-05), [FR-PLAN-08](../../context.md#fr-plan-08), [NFR-REL-07](../../context.md#nfr-rel-07); related glossary entries [context.md Glossary](../../context.md#glossary) for Plan mode / Subagent / System reminder.
- Downstream consumers tracked in [features-index.md](../../features-index.md): F25 (approval dialog renders the plan written by `ExitPlanMode` and calls back into `PlanStore.writePlan` on Edit), F26 (session resume replays `EnterPlanMode` / `ExitPlanMode` tool_use entries to reconstruct `mode` and rehydrate `TodoStore`).

## Implementation notes

- [Architecture §1 Architectural Principles](../../../../architecture/architecture.md#1-architectural-principles) — permission enforcement lives in the agent layer, never in prompts; plan-mode flag is first-class state.
- [Architecture §3.2 Agent Layer](../../../../architecture/architecture.md#32-agent-layer) — places `PlanModeController` alongside `AgentRunner` / `ToolRegistry`; per-thread flag held by the controller, consumed by the turn loop.
- [Architecture §4 Key Contracts](../../../../architecture/architecture.md#4-key-contracts) — pins the `ToolSpec` / `ToolResult` / `ToolCtx` types `EnterPlanMode` and `ExitPlanMode` implement; `ctx.thread.agentId` drives the subagent check.
- [Architecture §5.3 Chat Turn (with tool call + confirmation)](../../../../architecture/architecture.md#53-chat-turn-with-tool-call--confirmation) — canonical sequence: permission gate (this feature) → [F17 confirmation](../tool-confirmation-flow/feature.md) → `ToolRegistry.invoke`; blocked calls short-circuit to tool-error before confirmation.
- [Architecture §6 State Ownership](../../../../architecture/architecture.md#6-state-ownership) — `mode`, pending-attachment queue, stale-todo counters in-memory on `PlanModeController`; not persisted (resume is [F26](../../features-index.md)).
- [Architecture §7 Error Handling Strategy](../../../../architecture/architecture.md#7-error-handling-strategy) — typed `PlanModeBlocked` / `PlanModeForbiddenInSubagent` surface as tool-errors and structured log events, never silent.
- [Architecture §10 Concurrency & Lifecycle Rules](../../../../architecture/architecture.md#10-concurrency--lifecycle-rules) — per-thread state cleared on `Plugin.onunload`; no timers held open; attachments drained on next turn only.
- [Tech stack — Agent / Tool / Skill / MCP Wiring](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) — pins `@langchain/core/tools` `tool()` + Zod for `EnterPlanMode` / `ExitPlanMode` registration shape.
- [Tech stack — Dependencies — Production](../../../../standards/tech-stack.md#dependencies--production) — `zod` for the `ExitPlanMode` `{plan: string}` schema.
- [Code style — TypeScript](../../../../standards/code-style.md#typescript) — strict mode, named exports, `as const` mode union, no `enum`.
- [Code style — Zod & Tool Schemas](../../../../standards/code-style.md#zod--tool-schemas) — one Zod schema per external shape; `.describe(...)` on `ExitPlanMode.plan` field (LLM reads it).
- [Code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer) — permission gate runs before tool invoke; typed tool-error surface; `AbortSignal` threaded through `invoke(ctx)`.
- [Code style — Error Handling](../../../../standards/code-style.md#error-handling) — Fail Fast on subagent invocation; typed errors, no swallow.
- [Code style — Logging](../../../../standards/code-style.md#logging) — structured `plan.mode.*` / `plan.attachment.*` / `plan.stale-todo.*` events; reminder body and plan content never logged above `debug`.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — Vitest unit suite with [F10](../agent-controller-core/feature.md)'s turn loop stubbed; [F17 ConfirmationController](../tool-confirmation-flow/feature.md) mocked to assert zero-dialog invocations along the blocked path.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — Single Responsibility: controller owns flag + attachments + stale-todo rate-limit; gate is a pure predicate; tools are thin adapters over `PlanStore` / controller.
- [Best practices — Testing & Quality Gates](../../../../standards/best-practices.md#testing--quality-gates) — happy plus edge coverage (block-every-write / allowlist-passthrough / subagent-reject / rapid-toggle clearing / rate-limit boundary) seeded deterministically; no clocks, no sleeps, message-count rate-limit only.

## Open questions

- **`plan-file-write` tool name vs whitelisted path — global [open questions](../../context.md#open-questions) "Plan-file writes allowed in plan mode"**: [F23](../plan-files-todos-store/feature.md) left this deferred to F24; this feature picks the "module-level `PlanStore.writePlan` API, not a registered `ToolSpec`" reading — the allowlist names the `plan-file-write` path abstractly and the gate recognises it as the `PlanStore.writePlan` call site, not a tool id. Verifier to confirm the SRS does not require a distinct `WritePlan` `ToolSpec` surface.
- **Stale-todo reminder threshold N — plan.md §3.8 names "more than N messages ago" without a numeric value**: this feature exposes `settings.staleTodoReminderThreshold` (default 10) on [F03 settings-tab-scaffold](../settings-tab-scaffold/feature.md)'s Plan/Todos section; verifier to confirm 10 matches plan.md's intent or whether N should be derived from another signal (e.g. token count).
- **"Non-trivial work" predicate — plan.md §3.8 does not define**: this feature reads it as "≥ 1 `tool_call` StreamEvent in the most recent assistant turn"; verifier to confirm the predicate matches plan.md §3.8 rather than a token-count or wall-clock threshold.
- **Subagent-flag source on `ToolCtx` — [context.md open questions](../../context.md#open-questions) "Plan mode in subagent contexts"**: Leo has no subagent runtime yet, so `ctx.thread.agentId` is always `null` in Phase 2; this feature wires the guard anyway so the check is in place when subagents land. Flagged for F24-follow-up once subagent scope is decided.
