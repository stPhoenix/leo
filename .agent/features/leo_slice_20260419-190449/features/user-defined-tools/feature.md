# F40 — User-defined tools

## Purpose

Open the [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) `ToolRegistry` seam to user-authored tool declarations loaded from `.leo/tools/<name>.json` — the phase-5 extension point called out in [architecture §8](../../../../architecture/architecture.md#8-extension-points) and the [tech-stack — Agent / Tool / Skill / MCP Wiring](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) row — so a vault owner can register new tools without editing plugin source. Each declaration becomes a `ToolSpec` with `source: "user"` per [architecture §4](../../../../architecture/architecture.md#4-key-contracts), a Zod-validated argument schema, a `requiresConfirmation` default of `true` (write/destructive semantics per [FR-AGENT-04](../../context.md#fr-agent-04) and reinforced by the same guard policy as the plans-directory path-traversal rule in [NFR-REL-08](../../context.md#nfr-rel-08)), and a bounded `invoke` implementation — either a declarative `vault-op` built from the existing [`VaultAdapter`](../../../../architecture/architecture.md#34-adapters) non-active-note surface per [FR-AGENT-06](../../context.md#fr-agent-06), or an optional sandboxed JavaScript snippet evaluated via `Function` constructor per the [tech-stack row](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) and [tech-stack — Open Decisions](../../../../standards/tech-stack.md#open-decisions). No FR ID names user-defined tools explicitly; traceability flows through [FR-AGENT-04](../../context.md#fr-agent-04)'s ToolRegistry mandate, [FR-AGENT-06](../../context.md#fr-agent-06)'s Vault-API rule, and [FR-AGENT-10](../../context.md#fr-agent-10) / [FR-AGENT-11](../../context.md#fr-agent-11) for the confirmation + thread-scoped allowlist that user tools reuse.

## Scope

### In scope

- `UserToolsLoader` module under `src/tools/user/` per [architecture §9](../../../../architecture/architecture.md#9-project-file-layout-proposed) that, at `Plugin.onload` (after built-in registration per [architecture §5.1](../../../../architecture/architecture.md#51-plugin-startup)), scans `.leo/tools/*.json` via the [`VaultAdapter`](../../../../architecture/architecture.md#34-adapters), parses each file with a Zod declaration schema, and registers the result on the singleton [F16 `ToolRegistry`](../tool-registry-builtin-read/feature.md) with `source: "user"` per [architecture §4](../../../../architecture/architecture.md#4-key-contracts).
- Declaration schema (Zod, per [code style — Zod & Tool Schemas](../../../../standards/code-style.md#zod--tool-schemas)): `{id: string, description: string, parameters: JSONSchemaShape, requiresConfirmation?: boolean, impl: {kind: "vault-op", op: "read" | "create" | "append", pathArg: string, contentArg?: string} | {kind: "js", source: string}}`. The `parameters` JSON-Schema shape is compiled into a Zod validator at load time and fed to the provider's OpenAI-compatible `tools` array per [FR-PROV-04](../../context.md#fr-prov-04) through the same F16 path used by built-ins.
- `requiresConfirmation` defaults to `true` when omitted per [FR-AGENT-04](../../context.md#fr-agent-04)'s write/destructive default, so every user tool routes through the [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) inline dialog and honours the thread-scoped `metadata.allowedTools` allowlist from [FR-AGENT-11](../../context.md#fr-agent-11); declarations may opt into `false` only when the declared `impl.kind` is `"vault-op"` with `op: "read"` (a read-only surface) — any `"js"` impl is forced to `true` regardless of the declaration.
- `vault-op` impls resolve their argument path via the declared `pathArg` key, apply the same path-traversal guard as [F16](../tool-registry-builtin-read/feature.md)'s `read_note` (reject `..`, leading `/`, anything resolving outside the vault root — mirroring [NFR-REL-08](../../context.md#nfr-rel-08)'s plans-directory guard), and call the [`VaultAdapter`](../../../../architecture/architecture.md#34-adapters) surface (`read` / `create` / `append`), never `app.vault.adapter`, per [code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns); results follow the typed `{ok:true, data}|{ok:false, error}` [`ToolResult`](../../../../standards/code-style.md#langgraph--agent-layer) shape with no thrown errors escaping `invoke`.
- `js` impls are compiled once via `Function(ctx, args) → Promise<ToolResult>` per the [tech-stack row](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring). The sandbox boundary is a narrow context object exposing only `{vault: VaultAdapter, logger: Logger, signal: AbortSignal}` from [architecture §4 `ToolCtx`](../../../../architecture/architecture.md#4-key-contracts) — no `require`, no `app`, no `window`, no `fetch` (same-origin privilege too broad for a declarative tool, deferred); `invoke` wraps every call in try/catch per [code style — Error Handling](../../../../standards/code-style.md#error-handling), coerces any thrown error into `{ok:false, error}`, and honours `signal.aborted` per [code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer).
- Load-time validation surfaces a `Notice` + structured `tool.user.load.error` log event (via the [F01 Logger](../plugin-bootstrap-logging/feature.md) per [code style — Logging](../../../../standards/code-style.md#logging)) on schema-parse failure, id collisions with built-ins, or unreadable files; the offending declaration is skipped and the rest of the loader continues per [code style — Error Handling](../../../../standards/code-style.md#error-handling).
- Interop with skills: user-tool ids flow through the same [FR-AGENT-12](../../context.md#fr-agent-12) / [FR-SKILL-07](../../context.md#fr-skill-07) `allowedTools` filter as built-ins (F22 layer); no new filtering path is introduced here.
- Vitest unit coverage per [code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw): directory-scan + per-file Zod-parse, id collision rejection, `requiresConfirmation` default flip for omitted field, vault-op happy / not-found / traversal-blocked paths against a faked `VaultAdapter`, `js` impl happy path + thrown-error coercion + `AbortSignal.aborted` propagation, and a negative test asserting `"js"` declarations cannot opt into `requiresConfirmation: false`.

### Out of scope

- The in-plugin UI for creating / editing / deleting user tool files → no row in [features-index.md](../../features-index.md) covers a user-tool editor view; file edits happen manually (mirrors the Phase-2 skills story deferred to [F39 skill-editor-ui](../../features-index.md)).
- Hot reload on `.leo/tools/*.json` FS events → not a requirement in [srs.md](../../context.md); reload happens on `Plugin` reload, same as user skills in [F21 skills-loader-builtin](../../features-index.md) prior to F39's FS watcher.
- MCP-sourced tools (`source: "mcp"`, namespace `mcp.<server>.<tool>`) → ship with [F51 mcp-client-config-transports](../../features-index.md) and later per [FR-MCP-*](../../context.md#fr-mcp-01).
- Stronger JS isolation (spawned Node worker, Web Worker, V8 isolate) → tracked as an [open tech-stack decision](../../../../standards/tech-stack.md#open-decisions); this slice keeps the `Function`-constructor sandbox documented in [tech-stack — Agent / Tool / Skill / MCP Wiring](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) and punts the harder boundary.
- Arbitrary network / `fetch` access inside `js` impls → out of scope for v1; any future network surface goes through a dedicated adapter with its own permission gate (see Open questions).
- Cross-skill discoverability UX (surfacing which user tools a skill has opted into) → deferred to [F22](../../features-index.md) / [F39](../../features-index.md); this slice only delivers registration.

## Acceptance criteria

1. At `Plugin.onload`, after built-in tools are registered per [architecture §5.1](../../../../architecture/architecture.md#51-plugin-startup), the `UserToolsLoader` scans `.leo/tools/*.json` via the [`VaultAdapter`](../../../../architecture/architecture.md#34-adapters); each file is Zod-parsed against the declaration schema and registered on the singleton [F16 `ToolRegistry`](../tool-registry-builtin-read/feature.md) with `source: "user"`; `ToolRegistry.listFor(thread)` returns the union of built-in + user specs. (FR-AGENT-04)
2. The declaration schema parses `{id, description, parameters, requiresConfirmation?, impl}` where `impl` is either `{kind:"vault-op", op, pathArg, contentArg?}` or `{kind:"js", source}`; invalid declarations (missing fields, unknown `impl.kind`, bad JSON) are skipped with a `Notice` + `tool.user.load.error` log event and do not abort loader iteration. (FR-AGENT-04)
3. An id collision with any built-in tool id, or with a previously-loaded user tool id, causes the colliding declaration to be skipped with a `Notice` + `tool.user.load.error` log event; the registry retains the first (built-in) registration. (FR-AGENT-04)
4. When `requiresConfirmation` is omitted, the registered `ToolSpec` carries `requiresConfirmation: true`; declarations with `impl.kind: "js"` are forced to `requiresConfirmation: true` regardless of the declared value; only `impl.kind: "vault-op"` with `op: "read"` may opt into `false`. Every `requiresConfirmation: true` user tool routes through the [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) pre-invoke gate per [FR-AGENT-10](../../context.md#fr-agent-10) / [FR-AGENT-11](../../context.md#fr-agent-11). (FR-AGENT-04, FR-AGENT-10)
5. `vault-op` invocation applies the path-traversal guard (reject `..`, leading `/`, anything resolving outside vault root) before any Vault call and returns `{ok:false, error:"unsafe path"}` on rejection; success calls `VaultAdapter.read` / `create` / `append` (never `app.vault.adapter`) and returns `{ok:true, data}`; failure returns `{ok:false, error:<stable-string>}`; no exception escapes `invoke`. (FR-AGENT-06)
6. `js` invocation compiles the declared `source` once via `Function(ctx, args)`, feeds a narrow context exposing only `{vault: VaultAdapter, logger: Logger, signal: AbortSignal}`, wraps every call in try/catch so any thrown error coerces to `{ok:false, error}`, and honours `signal.aborted` by returning `{ok:false, error:"aborted"}`; no `require` / `app` / `window` / `fetch` is exposed on the context. (FR-AGENT-04)
7. Structured log events `tool.user.load.ok`, `tool.user.load.error`, and the existing `tool.invoke.start / ok / error` carry `{toolId, source:"user", thread, durationMs}` via the [F01 Logger](../plugin-bootstrap-logging/feature.md); payload contents are never logged verbatim per [code style — Logging](../../../../standards/code-style.md#logging). (FR-AGENT-04)
8. Vitest unit suite covers: directory-scan with mixed valid / invalid files, id collision with a built-in is rejected, `requiresConfirmation` default flip for omitted field, `js` declarations cannot opt into `false`, vault-op happy / not-found / traversal-blocked paths against a faked `VaultAdapter`, `js` impl happy path + thrown-error coercion + abort propagation, and that no `invoke` call throws. (FR-AGENT-04, FR-AGENT-06)

## Dependencies

- [F16 tool-registry-builtin-read](../tool-registry-builtin-read/feature.md) — provides the `ToolRegistry` + declarative `ToolSpec` shape, OpenAI-compatible `tools` parameter wiring, and the serial `tool_call` → invoke → `tool_result` loop that this feature registers `source: "user"` specs into.
- [F17 tool-confirmation-flow](../tool-confirmation-flow/feature.md) — provides the inline confirmation dialog + thread-scoped `metadata.allowedTools` allowlist gating every `requiresConfirmation: true` user tool invocation.
- [F22 skills-picker-active-skill](../../features-index.md) — applies the `allowedTools` filter per [FR-AGENT-12](../../context.md#fr-agent-12) / [FR-SKILL-07](../../context.md#fr-skill-07) to user tool ids identically to built-ins.
- Drives no new FR ID; traces to [FR-AGENT-04](../../context.md#fr-agent-04) (ToolRegistry mandate), [FR-AGENT-06](../../context.md#fr-agent-06) (Vault API for non-active-note tools), [FR-AGENT-10](../../context.md#fr-agent-10) / [FR-AGENT-11](../../context.md#fr-agent-11) (confirmation + per-thread allowlist), and [NFR-REL-08](../../context.md#nfr-rel-08) (path-traversal guard policy this feature mirrors).
- Phase-5 deliverable per the [`Scope` bullet](../../context.md#scope) "Phase 5 — Polish, Skill Editor, User Tools, More Providers"; [Out of scope](../../context.md#out-of-scope) explicitly confirms "User-defined custom tools before phase 5" is not shipped earlier.

- [Architecture §4 Key Contracts — ToolSpec](../../../../architecture/architecture.md#4-key-contracts) — pins `source: "user"` and the `{id, description, schema, requiresConfirmation, invoke}` fields user declarations must produce.

- [Architecture §7 Error Handling Strategy](../../../../architecture/architecture.md#7-error-handling-strategy) — tool throws caught and converted to tool-error; no exception escapes `invoke`.
- [Architecture §8 Extension Points — User tool (phase 5)](../../../../architecture/architecture.md#8-extension-points) — "Declarative JSON in `.leo/tools/<name>.json` loaded by `ToolRegistry`" is the pattern this feature implements.
- [Architecture §9 Project File Layout](../../../../architecture/architecture.md#9-project-file-layout-proposed) — places this loader under `src/tools/user/` and reserves `.leo/tools/` for declarations.
- [Tech stack — Agent / Tool / Skill / MCP Wiring](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) — pins the "config-driven declarations in `.leo/tools/*.json` + optional sandboxed JS via `Function` constructor (documented-risk)" shape.
- [Tech stack — Open Decisions](../../../../standards/tech-stack.md#open-decisions) — documents the `Function`-constructor vs Node-worker sandbox decision deferred to phase 5; this feature picks `Function` for the first cut.
- [Code style — Zod & Tool Schemas](../../../../standards/code-style.md#zod--tool-schemas) — one Zod schema per tool input; `.describe(...)` on fields so the LLM reads them.
- [Code style — LangGraph / Agent Layer](../../../../standards/code-style.md#langgraph--agent-layer) — typed `{ok, data}|{ok, error}` result; `AbortSignal` threaded through the sandbox `ctx`.
- [Code style — Obsidian Plugin Patterns](../../../../standards/code-style.md#obsidian-plugin-patterns) — never touch `app.vault.adapter` directly; the `.leo/tools/` scan and every `vault-op` go through `VaultAdapter`.
- [Code style — Error Handling](../../../../standards/code-style.md#error-handling) — Fail fast at Zod boundary; adapters surface `ToolResult`; `js` throws coerced, not swallowed.
- [Code style — Logging](../../../../standards/code-style.md#logging) — structured `event: "tool.user.*"` with `{toolId, source, thread, durationMs}`; no payload content beyond debug level.
- [Code style — Testing (Vitest + msw)](../../../../standards/code-style.md#testing-vitest--msw) — `VaultAdapter` faked in-memory; `js` impl evaluated against the narrow sandbox ctx.
- [Best practices — Core Principles](../../../../standards/best-practices.md#core-principles) — KISS + Single Responsibility: one loader, one schema, two impl kinds; no speculative sandbox upgrades.

## Open questions

- **Sandbox boundary for `js` impls** — The `Function` constructor shares the renderer realm, so the narrow `ctx` object is a discipline boundary, not a security boundary. Is that acceptable for v1 given the user owns both the vault and the `.leo/tools/` files, or must we move to a Web Worker / Node child process before shipping? Tracked in [tech-stack — Open Decisions](../../../../standards/tech-stack.md#open-decisions).
- **Declaration schema format** — The [tech-stack row](../../../../standards/tech-stack.md#agent--tool--skill--mcp-wiring) says "`.leo/tools/*.json`" but the SRS does not pin JSON vs YAML vs markdown-with-frontmatter the way skills do; proposing JSON-only to keep the Zod parse trivial, but worth confirming versus the open skill-format question ([skill-file-format open question](../../context.md#open-questions)).
- **Cross-skill discoverability** — Should user tool ids show up in the F22 / F39 skill editor's `allowedTools` picker alongside built-ins (same picker, tagged by `source`), or as a separate "Custom tools" group? No FR answers this; leaving to UI design.
- **Network / `fetch` access inside `js` impls** — Currently blocked by the narrow `ctx`; the moment someone wants to call a local HTTP service from a user tool we need a permission model (similar to [FR-AGENT-10](../../context.md#fr-agent-10) confirmation, but per-domain). Out of scope here, flagged for phase 5.
- **File-system reload semantics** — Should changes to `.leo/tools/*.json` at runtime hot-reload like user skills plan to in [F39](../../features-index.md), or stay reload-on-`Plugin`-reload only? The SRS is silent; the conservative default is the latter, matching [F21 skills-loader-builtin](../../features-index.md) before F39.
- **Declared `parameters` JSON-Schema dialect** — The OpenAI tools API accepts a JSON-Schema subset; user declarations need a matching subset. Proposing to compile JSON-Schema → Zod at load and reject unsupported keywords with a clear error, but the exact subset is unspecified.
